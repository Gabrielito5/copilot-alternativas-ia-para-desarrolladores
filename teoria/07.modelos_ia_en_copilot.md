# Tema 7. MODELOS IA EN COPILOT: ELECCIÓN, LÍMITES Y POLÍTICA DE USO

Un Senior debe saber que no todos los problemas requieren la misma potencia de cálculo. En este módulo aprenderemos a elegir el "motor" adecuado (GPT-4o, Claude Sonnet, modelos de razonamiento de OpenAI o Google) según la complejidad de la tarea en nuestro backend Spring.

---

## 1. Qué significa "modelos soportados" en Copilot y por qué importa

GitHub Copilot ya no es un modelo único: es una plataforma multi-modelo. En el modo Chat y Agent de VS Code puedes elegir el LLM subyacente. En IntelliJ el soporte está evolucionando, pero la lógica de selección aplica igualmente.

### El modelo no es solo "la IA": es una combinación de capacidades

Cada modelo tiene características que lo hacen más o menos adecuado para distintos tipos de trabajo:

| Capacidad | Qué implica en la práctica |
|---|---|
| **Ventana de contexto** | Cuántos archivos del proyecto puede "leer" a la vez. Modelos con contextos de 128k o 200k tokens pueden procesar muchas más clases simultáneamente. |
| **Velocidad de respuesta** | A mayor capacidad de razonamiento, mayor latencia. Un modelo de razonamiento puede tardar 30-60 segundos en responder. |
| **Seguimiento de instrucciones** | Qué tan fielmente el modelo respeta las restricciones que le das ("no uses Lombok", "mantén las firmas"). |
| **Conocimiento de Spring** | Los modelos más recientes conocen Spring Boot 3.x; los más antiguos pueden sugerir patrones de Spring 2.x o configuraciones deprecadas. |

### Por qué el modelo elegido cambia el resultado

El mismo prompt puede producir respuestas muy distintas según el modelo:

```
Prompt: "Diseña la arquitectura de un sistema de procesamiento de pedidos 
con alta disponibilidad para 10.000 pedidos/hora"
```

| Modelo | Tipo de respuesta típica |
|---|---|
| **Modelo rápido (Fast)** | Sugiere un monolito Spring Boot con una base de datos y "escala horizontalmente". Genérico y poco fundamentado. |
| **Modelo balanceado** | Propone separar lectura/escritura (CQRS), sugiere Kafka para desacoplar el procesamiento, menciona circuit breakers. Fundamentado pero no exhaustivo. |
| **Modelo de razonamiento** | Analiza los trade-offs de CQRS vs Event Sourcing para ese volumen, propone una arquitectura concreta con justificación de cada decisión, detecta los cuellos de botella y cómo medirlos. |

> 💡 **El criterio del Senior:** Un modelo rápido es un junior eficiente. Un modelo de razonamiento es un arquitecto lento. Usar el segundo para tareas del primero es un error de coste; usar el primero para tareas del segundo es un error de calidad.

## 2. Cuándo usar modelos rápidos vs. modelos más profundos (código vs. razonamiento)

La selección correcta del modelo no es "siempre el más potente". Es elegir el modelo cuyo nivel de razonamiento sea proporcional a la complejidad de la decisión que hay que tomar.

### Modelos Rápidos (Fast / Standard)

**Cuándo usarlos:** La tarea es estructuralmente predecible. El modelo no necesita "pensar"; necesita ejecutar un patrón conocido con precisión.

**Tareas ideales para modelos rápidos:**
- Generar DTOs, Records, Mappers
- Escribir Javadoc a partir del código existente
- Autocompletado inline de métodos boilerplate
- Tests unitarios simples (happy path de un servicio con mocks)
- Convertir un `for` clásico en un `stream()`

```java
// Tarea para modelo rápido: generar el mapper
// El patrón es predecible, el modelo solo necesita conocer los campos

public record CreateOrderRequest(
    String customerId,
    List<OrderItemRequest> items,
    String shippingAddress
) {}

// Prompt: "Genera el método estático from() para convertir 
// CreateOrderRequest a un Order del dominio"
// Resultado esperado: código directo, sin necesidad de razonamiento profundo
public static Order from(CreateOrderRequest request) {
    return new Order(
        CustomerId.of(request.customerId()),
        request.items().stream().map(OrderItem::from).toList(),
        Address.of(request.shippingAddress())
    );
}
```

### Modelos Profundos (Reasoning / Advanced)

**Cuándo usarlos:** La tarea requiere entender un sistema, razonar sobre consecuencias o tomar decisiones con múltiples trade-offs. El coste de una respuesta incorrecta es alto.

**Tareas que requieren razonamiento profundo:**
- Diseñar la arquitectura de un nuevo módulo
- Debuggear un deadlock o una condición de carrera
- Evaluar si una migración de Spring Security 5 → 6 introduce vulnerabilidades
- Diseñar una estrategia de consistencia eventual en un sistema distribuido
- Analizar qué impacto tiene cambiar una anotación `@Transactional` en un flujo complejo

```java
// Tarea para modelo de razonamiento: detectar el problema de concurrencia

@Service
public class InventoryService {
    private final Map<String, Integer> stockCache = new HashMap<>();
    
    public void reserveStock(String productId, int quantity) {
        Integer current = stockCache.get(productId);
        if (current == null || current < quantity) {
            throw new InsufficientStockException(productId);
        }
        stockCache.put(productId, current - quantity);
    }
}

// Prompt: "Analiza este servicio. ¿Tiene problemas de concurrencia? 
// ¿Por qué? ¿Cuál es la solución correcta para un sistema con 
// 100 requests/segundo al mismo productId?"
// Un modelo rápido dirá "usa synchronized". 
// Un modelo de razonamiento explicará el problema de las operaciones 
// no atómicas, comparará synchronized vs ConcurrentHashMap vs 
// optimistic locking en BD, y recomendará la opción según el SLA.
```

### La matriz de decisión práctica

```
                        COMPLEJIDAD DE LA DECISIÓN
                    Baja              Media             Alta
                 ┌──────────────┬──────────────┬──────────────┐
    IMPACTO   Alto│  Balanceado  │  Razonamiento│  Razonamiento│
    DEL       Medio│    Rápido   │  Balanceado  │  Razonamiento│
    ERROR     Bajo│    Rápido   │    Rápido    │  Balanceado  │
                 └──────────────┴──────────────┴──────────────┘
```

> ⚠️ **El error más caro:** Usar un modelo rápido para una decisión de arquitectura. No porque la respuesta sea obviamente mala (lo cual sería fácil de detectar), sino porque suena plausible y la aceptas sin cuestionarla.

## 3. Estrategia de coste/valor: cuándo merece "premium request"

En una suscripción de equipo, los modelos de razonamiento tienen un límite de peticiones premium por mes. Usarlos mal no solo es ineficiente: puede dejarte sin acceso al final del sprint cuando más lo necesitas.

### El coste real no es dinero, es tiempo de espera

Para un Senior, el "coste" de un modelo avanzado no es la factura (en la mayoría de suscripciones corporativas esto está cubierto), sino la latencia. Un modelo de razonamiento puede tardar 45 segundos en responder. En el flujo de trabajo diario, eso importa.

| Criterio | Usa modelo rápido | Usa modelo de razonamiento |
|---|---|---|
| **¿Cuántas veces harás esta tarea hoy?** | Muchas veces (boilerplate, tests) | Una o dos veces (diseño, debugging complejo) |
| **¿Cuánto tiempo te costaría hacerlo tú?** | < 10 minutos | > 1 hora |
| **¿El error tiene consecuencias en producción?** | No (lo detectas en tests) | Sí (decisión de arquitectura, seguridad) |
| **¿Necesitas justificación de la decisión?** | No | Sí (ADR, PR, revisión con el equipo) |

### Política de uso para equipos

Establecer una política explícita evita que el equipo "queme" las peticiones premium en tareas que no lo justifican:

```markdown
# Política de uso de modelos IA — Equipo Backend

## Modelo estándar (sin límite)
- Autocompletado inline
- Generación de boilerplate (DTOs, mappers, tests unitarios simples)
- Consultas puntuales ("¿Cuál es la firma de este método de Spring Data?")

## Modelo avanzado (gestionar con criterio)
- Diseño de nuevos módulos o servicios
- Debugging de problemas de concurrencia, transacciones o memoria
- Revisión de arquitectura antes de un PR importante
- Migraciones de versión (Spring Boot 2 → 3, Spring Security 5 → 6)
- Análisis de vulnerabilidades de seguridad

## No usar IA (decisión humana obligatoria)
- Definición de requisitos de negocio
- Decisiones de modelo de datos que afecten contratos externos
- Política de retención de datos y cumplimiento regulatorio
```

## 4. Sesgos habituales del modelo en Java/Spring

La IA tiene "prejuicios" basados en su entrenamiento. El corpus de código Java en internet es mayoritariamente antiguo: el código Java 8 con Lombok y `@Autowired` es mucho más abundante que el código Java 21 con Records y arquitectura hexagonal. Esto se traduce en sesgos predecibles.

### Sesgo de Verbosidad: el estilo "Java pre-8"

El modelo genera código correcto pero más largo de lo necesario. No es un error funcional, pero introduce ruido y hace el código más difícil de leer.

```java
// ❌ Lo que el modelo suele generar (estilo Java 6-8)
public List<String> getActiveUserEmails(List<User> users) {
    List<String> emails = new ArrayList<>();
    for (User user : users) {
        if (user.isActive()) {
            emails.add(user.getEmail());
        }
    }
    return emails;
}

// ✅ Lo que un Senior escribe en Java 17+
public List<String> getActiveUserEmails(List<User> users) {
    return users.stream()
        .filter(User::isActive)
        .map(User::getEmail)
        .toList();
}
```

**Cómo corregirlo:** Añade a tus prompts "usa Java 21, prefiere streams y records sobre código imperativo y clases mutables".

### Sesgo de Popularidad: la librería más famosa, no la más adecuada

El modelo propone lo más visto en internet, no lo más apropiado para tu contexto.

```java
// ❌ El modelo propone RestTemplate porque hay millones de ejemplos en internet
@Service
public class PaymentClient {
    private final RestTemplate restTemplate;
    
    public PaymentResponse charge(ChargeRequest request) {
        return restTemplate.postForObject(
            "/payments", request, PaymentResponse.class);
    }
}

// ⚠️ RestTemplate está en modo mantenimiento desde Spring 5.
// Para código nuevo en Spring Boot 3.x, WebClient o RestClient son la opción.

// ✅ Lo que deberías pedirle explícitamente
// Prompt: "Usa RestClient (Spring 6.1+), no RestTemplate ni WebClient"
@Service
public class PaymentClient {
    private final RestClient restClient;
    
    public PaymentResponse charge(ChargeRequest request) {
        return restClient.post()
            .uri("/payments")
            .body(request)
            .retrieve()
            .body(PaymentResponse.class);
    }
}
```

**Otros sesgos de popularidad frecuentes en Spring:**

| El modelo propone | Lo que deberías evaluar | Razón |
|---|---|---|
| `@Autowired` por campo | Inyección por constructor | Testabilidad, inmutabilidad |
| `RestTemplate` | `RestClient` / `WebClient` | RestTemplate está en modo mantenimiento |
| `@Data` de Lombok | Java Records | Para DTOs inmutables, Records son idiomáticos en Java 17+ |
| `Optional.get()` | `.orElseThrow()` / `.map()` | `get()` puede lanzar `NoSuchElementException` |
| `@SpringBootTest` para todo | `@WebMvcTest` / `@DataJpaTest` | Los slices son más rápidos y acotados |

### Sesgo de Configuración: mezcla de generaciones de Spring

El modelo puede mezclar estilos de configuración de distintas versiones de Spring porque ha visto todos en su entrenamiento.

```java
// ❌ El modelo mezcla configuración de Spring Security 5 con Spring Boot 3
@Configuration
@EnableWebSecurity
public class SecurityConfig extends WebSecurityConfigurerAdapter { // ← Deprecado en Spring 6
    
    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http.authorizeRequests() // ← API deprecada
            .antMatchers("/api/public/**").permitAll()
            .anyRequest().authenticated();
    }
}

// ✅ Spring Security 6 / Spring Boot 3.x
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .anyRequest().authenticated()
            )
            .build();
    }
}
```

> ⚠️ Este sesgo es especialmente peligroso porque el código deprecado compila sin errores. Solo falla en runtime o introduce vulnerabilidades silenciosas. **Regla:** Cuando el modelo genera configuración de Spring Security, verifica siempre la versión de la API contra la documentación oficial de la versión que usas.

### Sesgo de Alucinación de API: métodos que no existen

El modelo genera firmas que "suenan a Spring Data" pero no existen. El código compila solo si los métodos derivados de nombres son parseables por Spring Data, pero el comportamiento puede ser inesperado.

```java
// ❌ El modelo inventa una firma plausible que no existe
userRepository.findByStatusAndCreatedAtBetweenAndOrderByLastName(
    Status.ACTIVE, startDate, endDate
);
// Error: Spring Data no puede parsear "AndOrderBy" como parte del nombre del método.

// ❌ El modelo confunde el orden de los parámetros
userRepository.findByCreatedAtBetween(endDate, startDate); // El orden correcto es (start, end)

// ✅ Cuando dudes, usa @Query explícita: el error sale en arranque, no en runtime
@Query("SELECT u FROM User u WHERE u.status = :status " +
       "AND u.createdAt BETWEEN :start AND :end ORDER BY u.lastName")
List<User> findActiveUsersCreatedBetween(
    @Param("status") Status status,
    @Param("start") LocalDateTime start,
    @Param("end") LocalDateTime end
);
```

## 5. Criterios de validación: La pirámide de confianza

Antes de que el código generado entre en el repositorio, debe pasar estos filtros jerárquicos. Son jerárquicos porque no tiene sentido comprobar el nivel superior si el inferior ya ha fallado.

```
                        ┌─────────────────────────────┐
                        │     4. CUMPLE SEGURIDAD      │  ← El más difícil de automatizar
                        ├─────────────────────────────┤
                        │    3. CUMPLE ARQUITECTURA    │  ← El más difícil de testar
                    ┌───┴─────────────────────────────┴───┐
                    │         2. PASA LOS TESTS            │
                ┌───┴─────────────────────────────────────┴───┐
                │            1. COMPILA                        │  ← El mínimo indispensable
                └─────────────────────────────────────────────┘
```

### Nivel 1: Compilación

El código es sintácticamente correcto en el JDK objetivo y no tiene importaciones rotas.

**Automatización:** Tu IDE lo hace por ti. Si el código del agente no compila, es un rechazo inmediato.

**Trampas comunes:**
- El modelo genera código Java 17 en un proyecto Java 11 (records, sealed classes)
- El modelo usa una API de Spring Boot 3.x en un proyecto que aún está en 2.7
- Importaciones de librerías que no están en el `pom.xml`

### Nivel 2: Pasa los tests

El código funciona según el comportamiento esperado definido en los tests.

```bash
# El comando mínimo antes de cualquier commit con código generado por IA
mvn test

# Si hay tests de integración con Testcontainers, ejecutarlos también:
mvn verify
```

> ⚠️ **La trampa del "verde engañoso":** Los tests pueden pasar porque el agente los modificó para que pasaran, no porque el código sea correcto. Revisa siempre los diffs de los archivos de test además de los de producción.

```java
// ❌ El agente "arregla" el test en lugar de arreglar el código
// Antes (el test correcto):
@Test
void createOrder_shouldValidateMinimumAmount() {
    assertThrows(ValidationException.class, 
        () -> service.createOrder(new CreateOrderCommand(customerId, BigDecimal.ZERO)));
}

// Después (el agente lo "arregla" así para que pase):
@Test
void createOrder_shouldValidateMinimumAmount() {
    // TODO: revisar este test
    assertDoesNotThrow(
        () -> service.createOrder(new CreateOrderCommand(customerId, BigDecimal.ZERO)));
}
```

### Nivel 3: Cumple Arquitectura

El código respeta los límites de capas, los patrones de inyección y el estilo del equipo. **Este nivel no se puede automatizar completamente**: requiere revisión humana.

**Checklist de revisión arquitectónica:**

```
□ El dominio no importa nada de la capa de infraestructura
  → Busca imports de javax.persistence / jakarta.persistence en domain/
  
□ Los servicios no dependen de implementaciones concretas, sino de interfaces
  → Busca `new ConcreteRepository()` o `@Autowired ConcreteRepositoryImpl`
  
□ No hay lógica de negocio en los controladores REST
  → Los @RestController solo deben mapear request/response y delegar al servicio
  
□ Las transacciones están en la capa de servicio, no en el repositorio
  → Busca @Transactional en clases @Repository

□ No hay estado mutable en beans Singleton (@Service, @Component)
  → Busca campos no-final sin @Scope("prototype")
```

### Nivel 4: Cumple Seguridad

El código no introduce vulnerabilidades ni expone información sensible.

**Las vulnerabilidades más frecuentes introducidas por la IA:**

```java
// ❌ 1. SQL Injection por concatenación de strings
@Query("SELECT u FROM User u WHERE u.email = '" + email + "'") // ← NUNCA
// ✅ Siempre con parámetros nombrados:
@Query("SELECT u FROM User u WHERE u.email = :email")
User findByEmail(@Param("email") String email);
```

```java
// ❌ 2. Exposición de información sensible en respuestas de error
@ExceptionHandler(Exception.class)
public ResponseEntity<String> handleException(Exception e) {
    return ResponseEntity.status(500).body(e.getMessage()); // Stack trace al cliente
}
// ✅ Log interno, mensaje genérico al cliente:
@ExceptionHandler(Exception.class)
public ResponseEntity<ErrorResponse> handleException(Exception e) {
    log.error("Unexpected error", e); // El detalle va al log, no al cliente
    return ResponseEntity.status(500)
        .body(new ErrorResponse("An unexpected error occurred"));
}
```

```java
// ❌ 3. Datos sensibles en logs (el agente lo hace "por utilidad")
log.info("User {} logged in with password {}", username, password);
// ✅ Nunca loguear credenciales, tokens ni datos de tarjeta:
log.info("User {} logged in successfully", username);
```

```java
// ❌ 4. Endpoints sin control de acceso (el agente omite @PreAuthorize)
@GetMapping("/admin/users")
public List<UserResponse> getAllUsers() { ... }

// ✅ Con autorización explícita:
@GetMapping("/admin/users")
@PreAuthorize("hasRole('ADMIN')")
public List<UserResponse> getAllUsers() { ... }
```

## 6. Cómo comunicar las restricciones de versión al modelo

El sesgo de versión del modelo es predecible y se puede contrarrestar. La clave es ser explícito en cada prompt sobre el stack que usas.

### Prompt base de contexto de proyecto

En lugar de repetir las restricciones en cada prompt, crea una plantilla de contexto que pegues al inicio de conversaciones importantes:

```
# Contexto del proyecto (incluir al inicio de conversaciones de diseño)

Stack:
- Java 21 (usa records, sealed classes, pattern matching cuando aplique)
- Spring Boot 3.4.x
- Spring Security 6.x (usa SecurityFilterChain, no WebSecurityConfigurerAdapter)
- Spring Data JPA con Hibernate 6.x
- Maven 3.9

Estilo de código:
- Inyección por constructor, nunca @Autowired por campo
- DTOs como Java Records (no Lombok @Data para DTOs)
- Optional solo como tipo de retorno, nunca como parámetro
- Arquitectura hexagonal: domain/ no importa infrastructure/

Restricciones:
- No generes código compatible con versiones anteriores de Spring Boot 3.x
- No uses @Value sobre campos, usa @ConfigurationProperties
- No uses RestTemplate (usa RestClient)
```

## 🧪 Ejercicio: "Duelo de Modelos"

> **Contexto:** Tu equipo tiene un bug en producción. El sistema tiene un `OrderService` con el siguiente problema: en momentos de alta carga, algunos pedidos se crean con stock duplicado (dos pedidos consumen el mismo stock).

**Fase 1 — El problema (5 min):**

```java
@Service
public class OrderService {
    private final InventoryRepository inventoryRepository;
    private final OrderRepository orderRepository;

    @Transactional
    public Order createOrder(CreateOrderCommand command) {
        Inventory inventory = inventoryRepository
            .findByProductId(command.productId())
            .orElseThrow();
        
        if (inventory.getAvailableStock() < command.quantity()) {
            throw new InsufficientStockException();
        }
        
        inventory.setAvailableStock(inventory.getAvailableStock() - command.quantity());
        inventoryRepository.save(inventory);
        
        return orderRepository.save(Order.from(command));
    }
}
```

**Fase 2 — El duelo (15 min):**

1. Abre el Chat del IDE y selecciona el **modelo más rápido disponible**.
2. Envía este prompt:
   ```
   El siguiente servicio tiene un bug de concurrencia en producción: 
   dos pedidos simultáneos consumen el mismo stock. 
   ¿Cuál es el problema y cómo lo resuelves?
   [pega el código]
   ```
3. Copia y guarda la respuesta.
4. Cambia al **modelo de razonamiento más potente disponible**.
5. Envía exactamente el mismo prompt.
6. Compara las dos respuestas.

**Fase 3 — Análisis (10 min):**

Evalúa cada respuesta según estos criterios:

| Criterio | Modelo Rápido | Modelo Razonamiento |
|---|---|---|
| ¿Identifica la condición de carrera (race condition)? | | |
| ¿Explica por qué `@Transactional` solo no es suficiente? | | |
| ¿Menciona el Optimistic Locking con `@Version`? | | |
| ¿Menciona el Pessimistic Locking con `SELECT FOR UPDATE`? | | |
| ¿Compara los trade-offs de latencia de cada solución? | | |
| ¿Menciona el impacto en el comportamiento bajo alta concurrencia? | | |

**Respuesta que deberías esperar del modelo de razonamiento:**

El problema es una **race condition**: dos threads pueden leer el mismo stock disponible antes de que cualquiera de los dos lo decremente. `@Transactional` garantiza atomicidad de la escritura, pero no previene que dos transacciones lean el mismo valor antes de escribir.

```java
// Solución 1: Optimistic Locking (mejor para baja contención)
@Entity
public class Inventory {
    @Version
    private Long version; // Hibernate lanza OptimisticLockException si hay conflicto
    // ...
}
// Ventaja: no bloquea lecturas. Desventaja: requiere reintentos en alta contención.

// Solución 2: Pessimistic Locking (mejor para alta contención)
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT i FROM Inventory i WHERE i.productId = :productId")
Optional<Inventory> findByProductIdForUpdate(@Param("productId") String productId);
// Ventaja: garantiza exclusividad. Desventaja: degrada throughput bajo alta carga.

// Solución 3: Update atómico en BD (más eficiente)
@Modifying
@Query("UPDATE Inventory i SET i.availableStock = i.availableStock - :quantity " +
       "WHERE i.productId = :productId AND i.availableStock >= :quantity")
int decrementStock(@Param("productId") String productId, @Param("quantity") int quantity);
// Si devuelve 0, el stock era insuficiente. Sin locks, máximo throughput.
```

**Reflexión:** ¿El modelo rápido llegó al mismo nivel de análisis? ¿Propuso solo una solución o evaluó trade-offs? Esta diferencia es exactamente el criterio para elegir modelo.

## 💡 Tip de Senior: "The Model Switcher"

En el Chat de Copilot (VS Code o IntelliJ con soporte), el selector de modelo está en la parte superior del panel. La estrategia de un Senior no es "elegir el mejor una vez"; es **cambiar de modelo según la fase de la conversación**:

```
Fase de exploración   → Modelo balanceado  (genera ideas rápido, no importa si hay ruido)
Fase de diseño        → Modelo razonamiento (las decisiones importan)
Fase de implementación→ Modelo rápido      (genera código repetitivo a bajo coste)
Fase de revisión      → Modelo razonamiento (detectar problemas sutiles)
```

No es necesario empezar una conversación nueva para cambiar de modelo en muchos clientes. Puedes continuar la misma conversación con un modelo más potente cuando la complejidad lo requiera.
