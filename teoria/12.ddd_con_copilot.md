# Tema 12. DDD CON COPILOT: MODELADO ASISTIDO, DECISIÓN HUMANA

Implementar DDD requiere un nivel de precisión lingüística y técnica que choca con la tendencia de la IA a generar soluciones genéricas. En este tema aprenderemos a usar el agente para construir modelos de dominio ricos, manteniendo siempre el control sobre las decisiones estratégicas. La IA no decide cuáles son los agregados de tu negocio — eso lo decides tú con el experto de dominio. La IA implementa lo que has decidido.

Stack de referencia:

- Java 21+ (records para Value Objects, sealed classes para jerarquías de dominio)
- Spring Boot 3.4+
- JUnit 5 sin Spring para tests de dominio

---

## 1. Explorar modelos y Lenguaje Ubicuo sin inventar el negocio

El mayor error al usar la IA en DDD es pedirle que "diseñe el modelo de dominio". La IA inventará entidades genéricas con nombres técnicos (`Manager`, `Processor`, `Helper`) que no reflejan el lenguaje del negocio. El agente es útil como espejo para refinar ideas, no como arquitecto de dominio.

### Brainstorming de dominio

El rol correcto de la IA en esta fase es de "pato de goma": escucha la descripción del negocio y devuelve preguntas que fuerzan a clarificar conceptos ambiguos.

**Prompt al Agente:**
> "Voy a describir un proceso de negocio. Tu rol no es proponer soluciones técnicas: es hacerme preguntas que me ayuden a clarificar los conceptos del dominio. Cuando uses un término ambiguo, señálalo y pregunta qué significa exactamente para el negocio. Aquí la descripción: en un marketplace B2B, un comprador realiza un pedido de productos digitales. El sistema verifica el stock, procesa el cobro y notifica al comprador."

El resultado esperado son preguntas como:
- ¿Un "pedido" puede modificarse después de crearse o es inmutable desde el primer momento?
- ¿"Verificar el stock" bloquea el stock inmediatamente o solo comprueba disponibilidad?
- ¿"Procesar el cobro" y "notificar al comprador" son pasos del mismo proceso o pueden ocurrir de forma independiente?

Cada respuesta a estas preguntas define invariantes de dominio que el agente implementará después.

### El Glosario del Lenguaje Ubicuo

Antes de escribir código, el equipo debe tener un glosario de términos acordado con los expertos de negocio. Este glosario se convierte en el vocabulario que el agente debe respetar.

**Prompt al Agente:**
> "Dado este glosario de dominio, genera una lista de nombres de clases, métodos y estados que respeten el lenguaje ubicuo. Señala cualquier término técnico genérico (`Manager`, `Handler`, `Data`, `Info`) y propón el equivalente en el lenguaje del negocio.\n\nGlosario:\n- Pedido: intención de compra formal de un comprador.\n- Línea de pedido: cada producto individual dentro de un pedido con su cantidad y precio.\n- Confirmar: acción de cerrar el pedido para que no se puedan añadir más líneas.\n- Cancelar: anular un pedido que no ha sido enviado aún."

Ejemplo de glosario técnico resultante:

| Término de negocio | Clase / método Java | Prohibido |
| :--- | :--- | :--- |
| Pedido | `Order` | `OrderData`, `OrderInfo`, `OrderDTO` en dominio |
| Línea de pedido | `OrderLine` (record) | `OrderItem`, `LineEntry` |
| Confirmar | `order.confirm()` | `order.setStatus("CONFIRMED")` |
| Cancelar | `order.cancel()` | `order.setStatus("CANCELLED")` |
| Comprador | `CustomerId` (Value Object) | `String customerId` |

### Refinado de definiciones con el agente

Una vez definido el glosario, el agente puede ayudar a identificar qué estados y transiciones necesita cada concepto.

**Prompt al Agente:**
> "Dado el requisito de negocio 'Un pedido puede cancelarse salvo que ya haya sido enviado', qué estados debería tener la entidad `Order` para modelar esto correctamente, y qué método de dominio implementa la regla? No generes código todavía — solo el modelo conceptual con estados, transiciones permitidas y cuáles lanzan excepción."

Ejemplo de respuesta esperada:

```
Estados: DRAFT → CONFIRMED → SHIPPED → DELIVERED
                           → CANCELLED (desde DRAFT o CONFIRMED)
                           
cancel(): permitido desde DRAFT y CONFIRMED. Lanza OrderAlreadyShippedException desde SHIPPED.
```

Solo cuando este modelo conceptual está validado con el negocio se pide al agente que lo implemente.

### Checklist de exploración de dominio

- El glosario está acordado con el experto de negocio antes de escribir código.
- Ninguna clase de dominio usa términos técnicos genéricos (`Manager`, `Helper`, `Service` como sufijo en entidades).
- Los estados y transiciones están explicitados antes de implementar.
- El agente no ha inventado conceptos que no existen en el negocio.

## 2. Entidades, Agregados, Invariantes y Value Objects

Esta es la sección más técnica del tema. El agente tiende a generar POJOs con getters y setters. El Senior debe instruirlo para que genere objetos con comportamiento.

### Value Objects por defecto

Un Value Object encapsula un concepto del dominio con su validación. La regla es: si un campo tiene restricciones de negocio (un precio no puede ser negativo, un email debe tener @, una cantidad debe ser positiva), ese campo no es un `String` ni un `double` — es un Value Object.

**Prompt al Agente:**
> "Crea el Value Object `Money` como `record` de Java 21 con campos `amount` (BigDecimal) y `currency` (String). Incluye en el constructor compacto: validar que `amount` no sea nulo ni negativo, validar que `currency` tenga exactamente 3 caracteres (ISO 4217). Lanza `InvalidMoneyException` con mensaje descriptivo. Añade métodos `add(Money other)` que verifique que ambas monedas son la misma, y `multiply(int factor)`. Sin anotaciones de framework."

```java
package com.shopflow.orders.domain.model;

import java.math.BigDecimal;
import java.util.Objects;

public record Money(BigDecimal amount, String currency) {

    public Money {
        Objects.requireNonNull(amount, "Amount is required");
        Objects.requireNonNull(currency, "Currency is required");
        if (amount.compareTo(BigDecimal.ZERO) < 0) {
            throw new InvalidMoneyException("Amount cannot be negative: " + amount);
        }
        if (currency.length() != 3) {
            throw new InvalidMoneyException("Currency must be a 3-letter ISO 4217 code: " + currency);
        }
    }

    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new InvalidMoneyException(
                "Cannot add amounts in different currencies: " + this.currency + " vs " + other.currency);
        }
        return new Money(this.amount.add(other.amount), this.currency);
    }

    public Money multiply(int factor) {
        if (factor < 0) {
            throw new InvalidMoneyException("Factor cannot be negative: " + factor);
        }
        return new Money(this.amount.multiply(BigDecimal.valueOf(factor)), this.currency);
    }

    public static Money of(String amount, String currency) {
        return new Money(new BigDecimal(amount), currency);
    }

    public static Money zero(String currency) {
        return new Money(BigDecimal.ZERO, currency);
    }
}
```

Más Value Objects habituales en un dominio de pedidos:

```java
// Identificador tipado — evita confundir OrderId con CustomerId
public record OrderId(UUID value) {
    public OrderId {
        Objects.requireNonNull(value, "OrderId value is required");
    }
    public static OrderId generate() { return new OrderId(UUID.randomUUID()); }
    public static OrderId of(String value) { return new OrderId(UUID.fromString(value)); }
}

// Cantidad con mínimo de negocio
public record Quantity(int value) {
    public Quantity {
        if (value <= 0) throw new InvalidQuantityException("Quantity must be positive: " + value);
    }
}

// Email con validación básica de formato
public record Email(String value) {
    public Email {
        Objects.requireNonNull(value, "Email is required");
        if (!value.contains("@") || value.length() < 5) {
            throw new InvalidEmailException("Invalid email format: " + value);
        }
        value = value.toLowerCase().strip();
    }
}
```

### Protección de Invariantes

Las invariantes son las reglas que el dominio nunca puede violar. Viven en la entidad, no en el servicio.

**Prompt al Agente:**
> "Implementa el método `confirm()` en la entidad `Order`. Reglas de negocio:\n1. Solo se puede confirmar un pedido en estado DRAFT.\n2. Un pedido sin líneas no puede confirmarse.\n3. El total del pedido debe ser mayor que cero.\nPara cada regla violada, lanza una excepción de dominio específica (no RuntimeException genérico). El cambio de estado debe ocurrir solo si todas las validaciones pasan."

```java
public void confirm() {
    if (status != OrderStatus.DRAFT) {
        throw new OrderAlreadyConfirmedException(id);
    }
    if (lines.isEmpty()) {
        throw new EmptyOrderException(id);
    }
    if (calculateTotal().amount().compareTo(BigDecimal.ZERO) <= 0) {
        throw new ZeroValueOrderException(id);
    }
    this.status = OrderStatus.CONFIRMED;
}
```

Las excepciones de dominio son parte del modelo — deben estar en `domain/model/exceptions/`:

```java
// Excepción con contexto de negocio, no mensaje genérico
public class OrderAlreadyConfirmedException extends DomainException {
    public OrderAlreadyConfirmedException(OrderId orderId) {
        super("Order " + orderId.value() + " is already confirmed and cannot be modified");
    }
}

public class EmptyOrderException extends DomainException {
    public EmptyOrderException(OrderId orderId) {
        super("Order " + orderId.value() + " has no lines and cannot be confirmed");
    }
}

// Base sellada para que el ControllerAdvice pueda mapearlas todas
public abstract sealed class DomainException extends RuntimeException
    permits OrderAlreadyConfirmedException, EmptyOrderException,
            ZeroValueOrderException, InvalidMoneyException,
            InvalidQuantityException, InvalidEmailException {

    protected DomainException(String message) {
        super(message);
    }
}
```

La `sealed class` garantiza que el `@ControllerAdvice` en infraestructura puede tratar todas las excepciones de dominio con un `switch` exhaustivo.

### Definición de Agregados y Aggregate Root

Un Agregado es un grupo de entidades que se gestiona como una unidad. Solo la Aggregate Root (raíz) es accesible desde fuera del agregado.

**Prompt al Agente:**
> "Tengo estas entidades: `Order`, `OrderLine`, `Customer`. Ayúdame a decidir cuál es el Aggregate Root y por qué. Reglas del negocio relevantes: 1) Las líneas de pedido no existen sin un pedido. 2) Un cliente puede existir independientemente de sus pedidos. 3) Las líneas se añaden y eliminan siempre a través del pedido, nunca directamente. Responde con el razonamiento antes del diseño."

Reglas para que el agente respete los límites del agregado:

```java
// ✅ CORRECTO — OrderLine solo se crea a través de Order
public class Order {
    private final List<OrderLine> lines = new ArrayList<>();

    public void addLine(String productName, Quantity quantity, Money unitPrice) {
        if (status != OrderStatus.DRAFT) {
            throw new OrderAlreadyConfirmedException(id);
        }
        lines.add(new OrderLine(productName, quantity, unitPrice));
    }

    // La lista interna nunca se expone mutable
    public List<OrderLine> getLines() {
        return Collections.unmodifiableList(lines);
    }
}

// ❌ INCORRECTO — el repositorio no devuelve OrderLine directamente
// El servicio no llama a orderLineRepository.save(line) — no existe
```

### Checklist de entidades, agregados y VOs

- Ningún campo de dominio usa primitivos donde debería haber un Value Object con validación.
- Todas las invariantes están en métodos de la entidad, no en el servicio de aplicación.
- Las excepciones de dominio tienen mensajes descriptivos con el identificador de la entidad afectada.
- Las colecciones internas del agregado se devuelven con `Collections.unmodifiableList()`.
- La Aggregate Root es el único punto de acceso a las entidades hijas.

## 3. Repositorios, Factories y Domain Services

### Factories para creación compleja

Cuando crear una entidad requiere múltiples pasos o dependencias externas (generar un ID, calcular una fecha de expiración), el constructor público no es suficiente. El factory method encapsula esta lógica.

**Prompt al Agente:**
> "En la entidad `Order`, crea dos factory methods estáticos: `create(CustomerId customerId)` para nuevos pedidos y `reconstitute(OrderId id, CustomerId customerId, List<OrderLine> lines, OrderStatus status, LocalDateTime createdAt)` para reconstruir desde persistencia. El primero genera el UUID y fija el estado como DRAFT con la fecha actual. El segundo no aplica validaciones de creación porque los datos ya vienen de la BD. Constructor privado para forzar el uso de los factory methods."

```java
public class Order {

    // Constructor privado — nadie crea Order directamente
    private Order(OrderId id, CustomerId customerId, List<OrderLine> lines,
                  OrderStatus status, LocalDateTime createdAt) {
        this.id = id;
        this.customerId = customerId;
        this.lines = new ArrayList<>(lines);
        this.status = status;
        this.createdAt = createdAt;
    }

    // Para pedidos nuevos — aplica validaciones de creación
    public static Order create(CustomerId customerId) {
        Objects.requireNonNull(customerId, "CustomerId is required");
        return new Order(
            OrderId.generate(),
            customerId,
            new ArrayList<>(),
            OrderStatus.DRAFT,
            LocalDateTime.now()
        );
    }

    // Para reconstrucción desde BD — no aplica validaciones de creación
    public static Order reconstitute(OrderId id, CustomerId customerId,
                                      List<OrderLine> lines, OrderStatus status,
                                      LocalDateTime createdAt) {
        return new Order(id, customerId, lines, status, createdAt);
    }
}
```

### Domain Services vs. Application Services

La distinción que el agente más frecuentemente confunde:

| Tipo | Cuándo usarlo | Ejemplo |
| :--- | :--- | :--- |
| **Domain Service** | Lógica de negocio que involucra más de una entidad o agregado y no pertenece naturalmente a ninguna | `PricingService.calculate(Order, CustomerTier)` |
| **Application Service** | Orquestación del flujo: cargar entidades → ejecutar lógica → persistir → publicar evento | `CreateOrderService.execute(CreateOrderCommand)` |

**Prompt al Agente:**
> "Tengo una lógica de cálculo de precio final que necesita el pedido y el nivel de cliente (GOLD, SILVER, BASIC). ¿Esto es un Domain Service o un Application Service? Justifica la respuesta y muéstrame la firma del método antes de implementar."

Ejemplo de Domain Service correcto:

```java
package com.shopflow.orders.domain.service;

import com.shopflow.orders.domain.model.Order;
import com.shopflow.orders.domain.model.CustomerTier;
import com.shopflow.orders.domain.model.Money;

// Domain Service — lógica de negocio pura, sin framework
public class OrderPricingService {

    public Money calculateFinalPrice(Order order, CustomerTier customerTier) {
        Money subtotal = order.calculateTotal();
        return switch (customerTier) {
            case GOLD   -> subtotal.multiply(85).divide(100);   // 15% descuento
            case SILVER -> subtotal.multiply(92).divide(100);   // 8% descuento
            case BASIC  -> subtotal;
        };
    }
}
```

El Domain Service no tiene `@Service` de Spring — se registra como `@Bean` en `infrastructure/config/` o se instancia directamente donde se necesita.

### Repositorios orientados a colección

El puerto de salida del dominio no debe tener métodos con sabor a SQL. Debe hablar el lenguaje del dominio.

**Prompt al Agente:**
> "Crea el puerto de salida `OrderRepository` en `domain/port/out/`. Debe declarar solo las operaciones que el dominio necesita: guardar un pedido, buscar por ID, buscar pedidos confirmados de un cliente, y eliminar borradores expirados. Sin métodos JDBC, sin nombres tipo `findAllByStatusAndCustomerIdOrderByCreatedAtDesc`. Usa lenguaje de negocio."

```java
package com.shopflow.orders.domain.port.out;

import com.shopflow.orders.domain.model.Order;
import com.shopflow.orders.domain.model.CustomerId;
import com.shopflow.orders.domain.model.OrderId;

import java.util.List;
import java.util.Optional;

public interface OrderRepository {
    Order save(Order order);
    Optional<Order> findById(OrderId id);
    List<Order> findConfirmedOrdersByCustomer(CustomerId customerId);
    void removeExpiredDrafts();  // nombre de negocio, no removeByStatusAndCreatedAtBefore
}
```

### Checklist de repositorios, factories y domain services

- Los factory methods `create()` y `reconstitute()` están diferenciados.
- El constructor de la entidad es privado o package-private.
- Los Domain Services son Java puro sin `@Service` directamente (se registran como `@Bean`).
- Los métodos del repositorio tienen nombres de dominio, no nombres de query SQL.
- La distinción entre Domain Service y Application Service está clara en el código.

## 4. Evolución del modelo: Refactor, Migraciones y Compatibilidad

El modelo de dominio evoluciona. Lo que empezó como un `String price` puede necesitar convertirse en un `Money`. Lo que estaba en un servicio puede necesitar moverse a la entidad. El agente puede gestionar este proceso incrementalmente.

### Extracción de un Value Object de una entidad existente

Este es uno de los refactors más frecuentes: el código lleva meses usando `BigDecimal price` y ahora necesitamos `Money price` con validaciones.

**Prompt al Agente:**
> "Refactoriza la entidad `Order` para reemplazar el campo `BigDecimal totalAmount` por el Value Object `Money`. Pasos requeridos:\n1. Crea `Money` si no existe.\n2. Actualiza el campo y todas las referencias internas en `Order`.\n3. Actualiza el mapper `OrderJpaMapper.toDomain()` para reconstruir `Money` desde los campos `amount` y `currency` de la entidad JPA.\n4. Actualiza el mapper `toJpaEntity()` para persistir los dos campos por separado.\n5. Ejecuta `mvn test` y corrige si hay fallos.\nNo cambies el DDL de la base de datos todavía — haz el cambio solo en Java."

La clave de este prompt es la instrucción de ir paso a paso y ejecutar tests entre pasos. Sin esto, el agente hace el cambio completo de golpe y los errores de compilación se multiplican.

### Migración de lógica desde el servicio a la entidad

El modelo anémico clásico: toda la lógica de negocio vive en `OrderService` y `Order` es un POJO.

**Prompt al Agente:**
> "En `OrderService.createOrder()` hay esta lógica: `if (command.lines().isEmpty()) throw new RuntimeException(\"Order must have lines\")`. Esto es una invariante del dominio, no del servicio.\n\nMuévela al método `Order.confirm()` como precondición. Después actualiza `CreateOrderService` para que no repita la validación. Genera el test unitario para `Order.confirm()` que cubra el caso de lista vacía. No uses Spring en el test."

Antes de mover lógica, el agente debe buscar si ya existe un test que lo cubra:

```java
// Test previo al refactor — verifica el comportamiento que debe preservarse
@Test
@DisplayName("No se puede confirmar un pedido sin líneas")
void shouldNotConfirmOrderWithoutLines() {
    Order order = Order.create(new CustomerId(UUID.randomUUID()));
    assertThatThrownBy(order::confirm)
        .isInstanceOf(EmptyOrderException.class)
        .hasMessageContaining(order.getId().value().toString());
}
```

Si este test pasa antes del refactor (con la validación en el servicio, el método `confirm()` no lanza nada) y pasa después (con la validación en la entidad), el refactor es seguro.

### Manejo de Breaking Changes en el modelo

Cuando un cambio en el agregado rompe la API pública (el contrato del puerto de entrada), el agente debe señalarlo antes de aplicarlo.

**Prompt al Agente:**
> "Quiero cambiar el tipo del campo `customerId` en `Order` de `UUID` a `CustomerId` (Value Object). Analiza el impacto antes de hacer ningún cambio:\n1. Qué clases compilan actualmente con `UUID customerId`.\n2. Qué ports (interfaces) exponen `UUID` como parámetro.\n3. Si el cambio es breaking para los consumidores del puerto de entrada.\n4. Propón el orden de cambios para minimizar el tiempo con código roto."

Ejemplo de plan esperado del agente:

```
Impacto:
- Order.java: campo y constructor (1 cambio)
- CreateOrderCommand.java: parámetro del caso de uso (breaking change en el puerto)
- OrderMapper.java: conversión desde JPA (adaptador)
- CreateOrderRequest.java: DTO web (sin impacto directo, ya es UUID en JSON)

Plan de migración (sin romper compilación entre pasos):
1. Crear CustomerId record.
2. Añadir overload a create() que acepte CustomerId (compatibilidad temporal).
3. Actualizar Order internamente a CustomerId.
4. Actualizar CreateOrderCommand a CustomerId.
5. Actualizar adaptadores y mappers.
6. Eliminar overload de compatibilidad.
```

### Checklist de evolución del modelo

- Refactors de extracción de VO se hacen en pasos con `mvn test` entre cada uno.
- Tests de dominio escritos antes de mover lógica desde el servicio.
- El agente produce un análisis de impacto antes de aplicar cualquier breaking change.
- El mapper (`reconstitute`) se actualiza en el mismo commit que la entidad.

## 5. "Guardrails" para que Copilot no degrade el dominio a CRUD

La IA tiene una tendencia estructural hacia el modelo anémico porque es estadísticamente más frecuente en el código con el que fue entrenada. El Senior debe poner límites explícitos en cada prompt.

### No Setters

**Prompt al Agente:**
> "Genera la entidad `Order` sin ningún método setter público. Toda modificación de estado debe ocurrir a través de métodos con nombre semántico que reflejen el lenguaje ubicuo: `confirm()`, `cancel()`, `addLine()`, `removeLine()`. Si necesitas modificar el estado internamente para el mapper, usa el método `reconstitute()` de factory con todos los campos."

Ejemplo de anti-patrón a rechazar:

```java
// ❌ MALO — setter genérico que el agente genera por defecto
public void setStatus(OrderStatus status) {
    this.status = status;
}

// ❌ MALO — no hay validación, cualquiera puede poner cualquier estado
order.setStatus(OrderStatus.SHIPPED);

// ✅ CORRECTO — método con intención de negocio y validación
public void ship(TrackingNumber trackingNumber) {
    if (status != OrderStatus.CONFIRMED) {
        throw new OrderNotConfirmedException(id);
    }
    Objects.requireNonNull(trackingNumber, "Tracking number is required to ship");
    this.status = OrderStatus.SHIPPED;
    this.trackingNumber = trackingNumber;
}
```

### Validación en Constructor

Los Value Objects inválidos nunca deben existir en el sistema. La validación ocurre en la creación, no en el uso.

**Prompt al Agente:**
> "Para el record `OrderLine`, añade validación en el constructor compacto que garantice: productName no nulo ni en blanco (mínimo 3 caracteres), quantity positiva, unitPrice positivo. Usa excepciones de dominio específicas para cada regla, no `IllegalArgumentException`. El objeto solo puede crearse si todas las validaciones pasan."

```java
public record OrderLine(String productName, Quantity quantity, Money unitPrice) {

    // Constructor compacto — se ejecuta en cada instanciación del record
    public OrderLine {
        Objects.requireNonNull(productName, "Product name is required");
        if (productName.isBlank() || productName.length() < 3) {
            throw new InvalidProductNameException(productName);
        }
        Objects.requireNonNull(quantity, "Quantity is required");
        Objects.requireNonNull(unitPrice, "Unit price is required");
    }

    public Money subtotal() {
        return unitPrice.multiply(quantity.value());
    }
}
```

### Encapsulamiento estricto

**Prompt al Agente:**
> "Revisa todos los getters de la entidad `Order` y de su agregado. Si alguno devuelve una colección mutable interna, cámbialo para devolver una copia inmutable con `Collections.unmodifiableList()`. Si devuelve tipos de infraestructura (entidades JPA, tipos de Spring), señálalo como violación arquitectónica."

```java
// ❌ MALO — expone la lista interna mutable
public List<OrderLine> getLines() {
    return lines;  // cualquiera puede hacer lines.add() desde fuera
}

// ✅ CORRECTO — vista inmutable de la colección interna
public List<OrderLine> getLines() {
    return Collections.unmodifiableList(lines);
}
```

### El prompt de "audit de modelo" como guardrail final

Úsalo tras cualquier sesión de generación de código de dominio:

**Prompt al Agente:**
> "Audita las clases en `domain/model/` y responde en tabla:\n1. ¿Alguna clase tiene setter público? → Nombre del método y alternativa semántica.\n2. ¿Algún campo usa primitivos donde debería haber un Value Object?\n3. ¿Alguna colección interna se devuelve sin inmutabilidad?\n4. ¿Alguna excepción lanzada es `RuntimeException` o `IllegalArgumentException` genérico?\n5. ¿Algún constructor público permite crear objetos inválidos?"

### Checklist de guardrails anti-CRUD

- Ninguna clase de `domain/model/` tiene setter público.
- Todas las colecciones internas se devuelven con `Collections.unmodifiableList()`.
- Ningún campo de negocio usa primitivos donde debería haber un VO.
- Las excepciones de dominio son específicas y con mensaje que incluye el ID de la entidad afectada.
- El prompt de auditoría se ejecuta al final de cada sesión de generación de dominio.

## 6. Plugin del Curso: Asistente de Modelado de Dominio

Los cinco puntos anteriores definen el estándar: dominio rico con invariantes en las entidades, Value Objects para todo lo que no es primitivo, factories con validación, y ningún setter público. El problema es que la IA genera consistentemente lo contrario: modelos anémicos con lógica en el servicio y primitivos en lugar de tipos de dominio. Este punto convierte ese estándar en un detector automático y en una herramienta de enriquecimiento guiado.

El subagente actúa como el ojo crítico que detecta el "CRUD disfrazado de DDD". La skill orquesta la transformación del modelo anémico al rico siguiendo el flujo del punto 4, con un checkpoint humano antes de cada fase.

### Subagente: `spring-domain-analyst`

Analiza el modelo de dominio de un módulo y produce un diagnóstico de riqueza: qué entidades son anémicas, qué conceptos deberían ser Value Objects, qué invariantes están desplazadas al servicio de aplicación, y qué colecciones están expuestas sin protección.

Crea el fichero `.claude/agents/spring-domain-analyst.md`:

```markdown
---
name: spring-domain-analyst
description: Domain model analyst for Spring Boot projects following DDD principles. Use when reviewing entities and aggregates to detect anemic models, primitive obsession, invariants displaced to services, and missing Value Objects. Analyzes domain/model/ and domain/service/ classes together to assess where business logic lives. Does NOT rewrite code — produces a domain richness report with a score per entity and specific recommendations.
model: claude-sonnet-4-6
---

Eres un especialista en Domain-Driven Design aplicado a proyectos Java con Spring Boot 3.4+ y Java 21+. Tu función es diagnosticar la riqueza del modelo de dominio y detectar dónde se aplica el anti-patrón del modelo anémico. No reescribes código — produces un informe de diagnóstico.

## Qué analizas

Para cada entidad o agregado recibido, analiza en paralelo la entidad de dominio Y el servicio de aplicación que la usa. La comparación revela dónde vive realmente la lógica de negocio.

### 1. Modelo anémico (lógica en el servicio, no en la entidad)

Detecta en el servicio de aplicación:
- Condicionales que verifican el estado de una entidad antes de modificarla (`if (order.getStatus() == DRAFT) { order.setStatus(CONFIRMED); }`). Debería ser `order.confirm()`.
- Cálculos que usan solo datos de una entidad (`order.getLines().stream().mapToDouble(...).sum()`). Debería ser `order.calculateTotal()`.
- Validaciones de reglas de negocio que usan exclusivamente atributos de una entidad. Deberían ser invariantes dentro de la entidad.

Para cada caso: fragmento del servicio afectado, método que debería existir en la entidad, nombre semántico en Lenguaje Ubicuo.

### 2. Obsesión por primitivos

Detecta en las entidades y sus constructores:
- `double` o `Double` para precios, cantidades monetarias → sugerir `Money`.
- `String` para identificadores de dominio (`customerId`, `productId`) → sugerir `record CustomerId(UUID value)`.
- `String` para emails, teléfonos, códigos postales → sugerir Value Object con validación.
- `int` para cantidades con reglas de negocio → sugerir record con validación en constructor compacto.

### 3. Invariantes ausentes o mal ubicadas

- Constructor público sin validaciones de las reglas de creación.
- Factory method que lanza `RuntimeException` o `IllegalArgumentException` genérico en lugar de excepción de dominio específica.
- Métodos de transición de estado (`cancel()`, `ship()`) que no verifican si el estado actual permite la transición.

### 4. Colecciones internas expuestas

- Getter que devuelve directamente la `List` interna mutable.
- Ausencia de métodos de negocio para modificar la colección (`addLine()`) y presencia de `setLines(list)`.

### 5. Setters disfrazados

- `updateStatus(String)` en lugar de métodos por transición específica.
- `setDiscount(BigDecimal)` en lugar de `applyDiscount(DiscountCode)`.
- Métodos que reciben el estado final sin validar la transición.

## Formato de respuesta

1. **Resumen ejecutivo**: entidades analizadas, puntuación media de riqueza (0–10).
2. **Diagnóstico por entidad**:
   - Puntuación de riqueza (0–10).
   - Tabla de hallazgos por categoría.
   - Top 3 mejoras prioritarias con nombre concreto del método o tipo a crear.
3. **Mapa de migración sugerido**: orden recomendado para enriquecer el modelo.

## Puntuación de riqueza

- **8–10**: modelo rico. La lógica vive en las entidades.
- **5–7**: modelo parcialmente rico. Algunas invariantes en entidades, resto en servicios.
- **2–4**: modelo anémico. POJOs con getters/setters, toda la lógica en servicios.
- **0–1**: modelo CRUD. Sin distinción entre dominio e infraestructura.
```

### Skill: `/enrich-domain`

Orquesta la transformación de un modelo anémico hacia un modelo DDD rico en cuatro fases: extracción de Value Objects, traslado de invariantes a la entidad, adelgazado del servicio y verificación final. Checkpoint humano antes de cada fase.

Crea el fichero `.claude/commands/enrich-domain.md`:

```markdown
---
description: Transforma un modelo de dominio anémico en un modelo DDD rico. Extrae Value Objects, mueve invariantes a la entidad y adelgaza el servicio. Usa spring-domain-analyst para el diagnóstico y ejecuta los tests tras cada fase.
---

Actúa como guía de enriquecimiento de dominio DDD en un proyecto Spring Boot con Java 21+.

Si el usuario no ha indicado qué entidad trabajar, pregunta:
- ¿Qué entidad o agregado quieres enriquecer? (nombre de la clase)
- ¿Cuál es el servicio de aplicación asociado? (ruta del fichero)

## Fase 1 — Diagnóstico con `spring-domain-analyst`

Recopila la clase de dominio y el servicio de aplicación asociado.

Invoca el subagente `spring-domain-analyst` pasándole ambos ficheros.

Muestra el informe completo al usuario, destacando las tres mejoras prioritarias.

Espera confirmación:
**"¿Procedo con el enriquecimiento? (sí/no)"**

No avances sin "sí" explícito.

## Fase 2 — Extracción de Value Objects

Para cada primitivo identificado que deba convertirse en Value Object:

1. Crea el `record` en `domain/model/` con constructor compacto con validación y excepción de dominio específica.
2. Actualiza la entidad para usar el nuevo tipo.
3. Actualiza `create()` y `reconstitute()`.
4. Actualiza el servicio de aplicación donde construye el primitivo.

Tras cada Value Object extraído (no al final de todos):
    ```bash
        mvn -q -DskipTests compile -pl <módulo>
    ```
Corrige errores de compilación antes de continuar.

Ejecuta `mvn test -pl <módulo>` cuando todos los VOs estén extraídos.

Muestra el diff y espera confirmación antes de la Fase 3:
**"¿Continúo con el traslado de invariantes? (sí/no)"**

## Fase 3 — Traslado de invariantes a la entidad

Para cada lógica de negocio del servicio que pertenece a la entidad:

1. Crea el método semántico en la entidad con validación de estado previo, lógica trasladada y excepción de dominio específica. Sin acceso a repositorios ni servicios externos.
2. Reemplaza en el servicio la lógica inline por la llamada al método de la entidad.
3. Si había tests que verificaban la lógica en el servicio, muévelos a un test unitario de la entidad sin Spring.

Ejecuta `mvn test -pl <módulo>` tras cada método trasladado.

Muestra el diff y espera confirmación antes de la Fase 4:
**"¿Continúo con el adelgazado del servicio? (sí/no)"**

## Fase 4 — Adelgazado del servicio y verificación

El servicio debe quedar como orquestador puro: cargar → ejecutar método de dominio → persistir → devolver. Verifica:
- Sin condicionales de estado (`if/switch` sobre `getStatus()`).
- Sin cálculos que usen solo datos de la entidad.
- Sin excepciones de negocio lanzadas directamente en el servicio.

Invoca `spring-domain-analyst` de nuevo sobre los ficheros modificados. Compara la puntuación de riqueza antes → después.

Ejecuta `mvn test -pl <módulo>` y reporta resultado.

## Informe final

- Puntuación de riqueza: antes → después.
- Value Objects creados con su validación.
- Métodos de dominio añadidos.
- Líneas de lógica eliminadas del servicio.
- Tests actualizados o creados.
```

#### 💡 Tip de Senior: "El Glosario como Contexto Permanente"

El mayor problema del agente en DDD no es técnico sino lingüístico: sin contexto del lenguaje ubicuo genera nombres genéricos. La solución es incluir el glosario del dominio en el `CLAUDE.md` del repositorio o en `.github/copilot-instructions.md` para que esté disponible en cada sesión sin tener que repetirlo en cada prompt.

Ejemplo de sección de glosario en `shopflow/CLAUDE.md`:

```markdown
## Lenguaje Ubicuo — ShopFlow

Usa estos términos en nombres de clases, métodos y variables:

| Término correcto | Prohibido |
|---|---|
| `Order` | `OrderData`, `OrderInfo`, `OrderDTO` en dominio |
| `order.confirm()` | `order.setStatus("CONFIRMED")` |
| `order.cancel()` | `order.setStatus("CANCELLED")` |
| `CustomerId` | `String customerId`, `UUID customerId` |
| `Money` | `Double price`, `BigDecimal amount` sin encapsular |
| `OrderLine` | `OrderItem`, `LineEntry`, `Item` |
```

Con esto en el contexto, el agente respeta el lenguaje ubicuo en cada generación de código sin necesidad de recordárselo en cada prompt.
