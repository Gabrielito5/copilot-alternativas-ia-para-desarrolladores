# Tema 4. GITHUB COPILOT EN INTELLIJ: COMPLETIONS PRO, NO “AUTOCOMPLETAR”

En este tema abandonamos la idea de que Copilot es un simple autocompletado de texto. Aprenderemos a gestionar el **contexto** y las **señales** para que la IA proponga código que respete la arquitectura de nuestra aplicación Spring.

---

## 1. Cómo funciona el autocompletado contextual (señales y contexto)

No es magia, es una ventana de contexto optimizada. El IDE construye un **"prompt invisible"** y lo envía al modelo. Entender qué ve la IA nos permite manipular sus sugerencias.

```
┌──────────────────────────────────────────────────────┐
│               SEÑALES QUE ENVÍA EL IDE                │
│                                                        │
│  📄 Archivo actual        → Código ANTES del cursor    │
│  📄 Archivo actual        → Código DESPUÉS del cursor  │
│  📂 Archivos abiertos     → "Neighboring Files"        │
│  📝 Nombre del archivo    → UserService.java           │
│  💬 Comentarios cercanos  → JavaDoc, TODOs             │
│  🏷️ Tipos y anotaciones   → @Service, Optional<>       │
└──────────────────────────────────────────────────────┘
```

#### Fill-in-the-Middle (FIM): el mecanismo real del autocompletado

Desde 2022, Copilot no envía solo el código anterior al cursor: usa el modelo en modo **Fill-in-the-Middle**. El prompt incluye tanto el prefijo (código antes) como el sufijo (código después), y el modelo genera lo que encaja entre ambos:

```
<|fim_prefix|>  ← código ANTES del cursor
<|fim_suffix|>  ← código DESPUÉS del cursor
<|fim_middle|>  ← aquí genera la sugerencia
```

Esto significa que el código que escribes *después* del cursor también influye en la sugerencia. Si ya tienes la línea de cierre de un método, Copilot genera un cuerpo coherente con ella.

### Señales Clave en Spring Boot

| Señal | Impacto en la Sugerencia |
|---|---|
| **Tipos Java** (Records, Optional, etc.) | Ancla el autocompletado a usar tipos modernos |
| **Anotaciones Spring** (@Service, @Transactional) | Guía el patrón de inyección y transaccionalidad |
| **Nombre del archivo** (UserRepository.java) | La IA infiere que debe generar métodos de persistencia |
| **Archivos abiertos** (DTO + Mapper) | Favorece la coherencia entre clases relacionadas |

> ⚠️ `pom.xml` y `build.gradle` los usa Copilot en el **modo agente/workspace** para analizar módulos y dependencias, pero **no son señales directas del autocompletado inline**. No cuentes con que Copilot "detecte" tu versión de Spring Boot mientras escribes código; eso pertenece a otro modo de operación.

### Neighboring Files: El Truco Más Infravalorado

Copilot analiza los archivos abiertos en el editor, los **rankea por similitud textual** con el código que estás escribiendo, y solo incluye los más relevantes respetando el presupuesto de tokens. No todos los archivos abiertos tienen el mismo peso: ganan los que comparten nombres de variables, patrones de naming o estructura similar.

Si estás escribiendo un **Mapper** y tienes abierto el **DTO** y la **Entidad de Dominio**, el autocompletado:
- Conoce los campos exactos de ambas clases.
- Genera los mapeos adecuados sin que tú los dictes.
- Respeta los tipos (ej: convierte `LocalDateTime` correctamente).

> 💡 **Tip de Senior:** Antes de empezar a escribir un componente, abre manualmente los 2-3 archivos relacionados. Es como "dar contexto" sin escribir un prompt.

## 2. Escribir “prompt implícito” con código: nombres, comentarios, ejemplos mínimos

La mejor forma de "hablar" con el autocompletado no es mediante el chat, sino mediante **el código que ya estás escribiendo**.

### Naming como Contrato

```java
// ❌ MAL — Nombre vago, sugerencia genérica
public List<User> getUsers() {
    // La IA sugiere: return userRepository.findAll();
}

// ✅ BIEN — Nombre preciso, sugerencia precisa
public List<User> findAllUsersWithActiveSubscriptions() {
    // La IA sugiere: return userRepository.findBySubscriptionStatusAndExpirationDateAfter(
    //     SubscriptionStatus.ACTIVE, LocalDate.now());
}
```

### Comentarios Técnicos como Pre-Prompt

```java
/**
 * Busca usuarios cuya suscripción expire en los próximos N días.
 * Usa paginación para evitar cargar toda la tabla en memoria.
 * @param days número de días de anticipación
 * @return página de usuarios con suscripción próxima a expirar
 */
public Page<User> findUsersWithExpiringSubscription(int days, Pageable pageable) {
    // La IA ahora tiene contexto completo: paginación, días, tipo de retorno.
}
```

### Few-Shot Prompting en Código

Escribe **el primer caso manualmente** para que la IA complete los demás siguiendo el patrón:

```java
// Tú escribes el primero:
public record UserResponse(String name, String email) {
    public static UserResponse from(User user) {
        return new UserResponse(user.getName(), user.getEmail());
    }
}

// Al crear OrderResponse, la IA sugiere automáticamente el mismo patrón:
public record OrderResponse(String id, BigDecimal total, String status) {
    public static OrderResponse from(Order order) {
        return new OrderResponse(order.getId(), order.getTotal(), order.getStatus().name());
    }
}
```

### Declaración de Interfaces Primero

Definir la interfaz **antes** de la implementación fuerza a la IA a respetar el contrato:

```java
// 1. Defines el puerto (interfaz)
public interface OrderRepository {
    Order save(Order order);
    Optional<Order> findById(OrderId id);
    List<Order> findByCustomerId(CustomerId customerId);
}

// 2. Al implementar, la IA respeta EXACTAMENTE las firmas
@Repository
public class JpaOrderRepository implements OrderRepository {
    // Las sugerencias respetan los tipos de dominio (OrderId, CustomerId)
    // en lugar de inventar firmas genéricas con String/Long
}
```

## 3. Patrones de uso: “scaffold → constrain → complete → verify”

Un flujo estructurado para generar código de calidad, paso a paso:

### 1. Scaffold (Andamiaje)
Defines la estructura de la clase: nombre, campos, firmas de métodos.

```java
public class OrderService {
    private final OrderRepository orderRepository;
    private final EventPublisher eventPublisher;

    // Constructor injection (la IA lo completa)

    public Order createOrder(CreateOrderCommand command) {
        // TODO: implementar
    }
}
```

### 2. Constrain (Restricción)
Añades anotaciones y comentarios que limitan lo que la IA puede generar:

```java
@Service
@Transactional(readOnly = true)
public class OrderService {
    // ...

    @Transactional  // La IA sabe que este método modifica datos
    public Order createOrder(@Valid CreateOrderCommand command) {
        // Validar que el cliente existe
        // Crear la orden con estado PENDING
        // Publicar evento OrderCreated
        // NO calcular descuentos aquí (eso es responsabilidad del dominio)
    }
}
```

### 3. Complete (Completar)
Dejas que la IA rellene los huecos basados en tus restricciones.

### 4. Verify (Verificar)
El paso **crítico**: revisión humana + ejecución de tests.

```
┌─────────────────────────────────────────────┐
│  Scaffold → Constrain → Complete → Verify   │
│     ↑                               │       │
│     └──── Si falla, refina ←────────┘       │
└─────────────────────────────────────────────┘
```

## 4. Cómo guiar sugerencias con estilo, convenciones y arquitectura del equipo

### `.github/copilot-instructions.md`

Este archivo estandariza el estilo del equipo. La IA lo lee automáticamente:

```markdown
## Reglas de Código
- Usa siempre Java Records para DTOs en lugar de Lombok @Data
- Inyección de dependencias SOLO por constructor, nunca @Autowired en campos
- Los Value Objects del dominio deben validar en el constructor
- Naming: UserResponse, OrderRequest (no UserDTO, OrderDTO)
- Usa Optional solo como tipo de retorno, nunca como parámetro

## Arquitectura
- Sigue Arquitectura Hexagonal estricta
- El paquete domain/ NO puede importar nada de infrastructure/
- Usa @ControllerAdvice para manejo global de excepciones
```

### Bibliotecas Propias

Para que la IA sugiera utilidades internas en lugar de librerías genéricas:
- Mantén los archivos de tus utilidades internas **abiertos en tabs**.
- Incluye en las instrucciones: *"Usa la clase `MoneyUtils` de nuestro paquete `shared` para conversiones de divisa"*.

## 5. Errores típicos (código plausible pero incorrecto) y cómo detectarlos rápido

La IA es experta en "parecer" correcta mientras rompe el sistema. Los errores más peligrosos no son los que fallan en local: son los que pasan los tests y explotan en producción.

| Sugerencia Sospechosa | Riesgo Real | Detección Rápida |
|---|---|---|
| `@Transactional` en método privado | El proxy de Spring no intercepta | Buscar `private` + `@Transactional` |
| Query JPA con múltiples `JOIN FETCH` | Producto cartesiano en memoria | Buscar más de 2 `JOIN FETCH` |
| `Optional.get()` sin chequeo | `NoSuchElementException` en prod | Buscar `.get()` sin `.isPresent()` o `.orElse()` |
| Inyección `@Autowired` por campo | Rompe testabilidad e inmutabilidad | Buscar `@Autowired` fuera de constructores |
| `HashMap` en un `@Service` (Singleton) | Problemas de concurrencia bajo carga | Buscar colecciones mutables en beans Singleton |

### Error 1: `@Transactional` en método privado

```java
@Service
public class OrderService {

    // ❌ MAL — La IA lo genera así. El proxy AOP de Spring
    // no puede interceptar métodos privados. La anotación es ignorada.
    @Transactional
    private void updateOrderStatus(Order order, Status status) {
        order.setStatus(status);
        orderRepository.save(order);
        eventPublisher.publish(new OrderStatusChangedEvent(order));
    }

    // ✅ BIEN — Público o paquete-privado (package-private) para que el proxy funcione.
    // Si debe ser "interno", extráelo a otro @Service.
    @Transactional
    public void updateOrderStatus(Order order, Status status) {
        order.setStatus(status);
        orderRepository.save(order);
        eventPublisher.publish(new OrderStatusChangedEvent(order));
    }
}
```

> ⚠️ Este error no lanza excepción. Simplemente no hay transacción: si `eventPublisher.publish()` falla, el `save()` ya está comiteado y el estado queda inconsistente.

### Error 2: Producto cartesiano con múltiples `JOIN FETCH`

```java
// ❌ MAL — La IA combina dos JOIN FETCH en una sola query.
// Si un Order tiene 10 Items y 5 Payments, JPA devuelve 50 filas (10×5)
// antes de deduplicar. Con miles de órdenes, esto revienta la memoria.
@Query("SELECT o FROM Order o " +
       "JOIN FETCH o.items " +
       "JOIN FETCH o.payments " +
       "WHERE o.customerId = :customerId")
List<Order> findOrdersWithDetails(@Param("customerId") Long customerId);

// ✅ BIEN — Dos queries separadas. Hibernate resuelve la asociación
// con una segunda SELECT IN en lugar de un producto cartesiano.
@Query("SELECT DISTINCT o FROM Order o JOIN FETCH o.items WHERE o.customerId = :customerId")
List<Order> findOrdersWithItems(@Param("customerId") Long customerId);

@Query("SELECT DISTINCT o FROM Order o JOIN FETCH o.payments WHERE o.customerId = :customerId")
List<Order> findOrdersWithPayments(@Param("customerId") Long customerId);
```

### Error 3: `Optional.get()` sin verificación

```java
// ❌ MAL — La IA suele generar esto cuando hay prisa por "completar" el método.
public OrderResponse getOrder(Long id) {
    Optional<Order> order = orderRepository.findById(id);
    return OrderResponse.from(order.get()); // NoSuchElementException si no existe
}

// ✅ BIEN — Opciones idiomáticas según el caso de uso:
public OrderResponse getOrder(Long id) {
    return orderRepository.findById(id)
        .map(OrderResponse::from)
        .orElseThrow(() -> new OrderNotFoundException(id));
}
```

### Error 4: Inyección `@Autowired` por campo

```java
// ❌ MAL — Dificulta los tests (necesitas Reflection o Spring Context),
// oculta las dependencias reales de la clase y permite objetos en estado inválido.
@Service
public class PaymentService {
    @Autowired
    private PaymentGateway paymentGateway;
    @Autowired
    private OrderRepository orderRepository;
}

// ✅ BIEN — Inyección por constructor: dependencias explícitas, clase testeable
// con un simple `new PaymentService(mockGateway, mockRepo)`.
@Service
public class PaymentService {
    private final PaymentGateway paymentGateway;
    private final OrderRepository orderRepository;

    public PaymentService(PaymentGateway paymentGateway, OrderRepository orderRepository) {
        this.paymentGateway = paymentGateway;
        this.orderRepository = orderRepository;
    }
}
```

### Error 5: Estado mutable en un Singleton (`@Service`)

```java
// ❌ MAL — Un @Service es un Singleton: una sola instancia compartida por todos
// los threads. Este HashMap no es thread-safe y causará corrupción de datos bajo carga.
@Service
public class ExchangeRateService {
    private Map<String, BigDecimal> rateCache = new HashMap<>(); // ← BOMBA DE TIEMPO

    public BigDecimal getRate(String currency) {
        if (!rateCache.containsKey(currency)) {
            rateCache.put(currency, fetchFromExternal(currency));
        }
        return rateCache.get(currency);
    }
}

// ✅ BIEN — Usa ConcurrentHashMap, o mejor aún, delega el caché
// a un mecanismo gestionado (Caffeine, Spring @Cacheable).
@Service
public class ExchangeRateService {
    private final Map<String, BigDecimal> rateCache = new ConcurrentHashMap<>();

    @Cacheable("exchange-rates") // alternativa preferida
    public BigDecimal getRate(String currency) {
        return fetchFromExternal(currency);
    }
}
```

### Alucinaciones de API: métodos que no existen

```java
// ❌ MAL — La IA inventa firmas de Spring Data que suenan plausibles
// pero no corresponden a ningún método real.
List<User> users = userRepository.findByStatusAndCreatedDateBetweenOrderByLastName(
    Status.ACTIVE, startDate, endDate // ← compila, pero el método no existe en el repo
);

// ❌ MAL — Parámetros invertidos (la IA confunde el orden)
userRepository.findByCreatedDateBetween(endDate, startDate); // startDate va primero

// ✅ BIEN — Verifica siempre contra la interfaz del repositorio o la documentación JPA.
// En caso de duda, usa @Query explícita para que el error sea en compilación, no en runtime.
@Query("SELECT u FROM User u WHERE u.status = :status AND u.createdDate BETWEEN :start AND :end ORDER BY u.lastName")
List<User> findActiveUsersCreatedBetween(
    @Param("status") Status status,
    @Param("start") LocalDateTime start,
    @Param("end") LocalDateTime end
);
```

> 💡 **Regla de oro:** Si la IA genera código que "suena bien" pero no puedes trazar su origen en la documentación oficial, trátalo como sospechoso hasta que un test lo verifique.

## 3.6. Estrategias para evitar dependencia y degradación de criterio

El riesgo real del autocompletado no es que genere código malo: es que genera código *suficientemente bueno* como para que dejes de pensar. La degradación es gradual e imperceptible.

### El "Modo Crítico": leer como reviewer, no como autor

Antes de aceptar cualquier sugerencia, hazte estas tres preguntas:

1. **¿Entiendo cada línea?** Si hay algo que no puedes explicar en voz alta, no lo aceptes.
2. **¿Qué asume este código?** Toda sugerencia tiene precondiciones implícitas (nunca nulo, siempre una fila, siempre en transacción...).
3. **¿Qué ocurre en el peor caso?** Piensa en el fallo, no en el happy path.

```java
// La IA sugiere esto. ¿Puedes responder las tres preguntas?
public Money calculateRefund(Order order) {
    return order.getPayments().stream()
        .filter(Payment::isRefundable)
        .map(Payment::getAmount)
        .reduce(Money.ZERO, Money::add);
}
// Pregunta 1: ¿Qué ocurre si getPayments() devuelve null en lugar de lista vacía?
// Pregunta 2: ¿Money.ZERO existe en tu modelo de dominio o lo está inventando?
// Pregunta 3: ¿Puede haber pagos parcialmente reembolsables? ¿El filtro los excluye correctamente?
```

### El "Desafío de Simplificación"

La IA tiende a ser **verbosa**: genera el caso general cuando el caso específico es suficiente. Tu tarea es intentar reducir el código generado, no solo aceptarlo.

```java
// ❌ Versión verbosa que la IA genera habitualmente:
public Optional<UserResponse> findUser(String email) {
    Optional<User> userOptional = userRepository.findByEmail(email);
    if (userOptional.isPresent()) {
        User user = userOptional.get();
        UserResponse response = new UserResponse(user.getName(), user.getEmail());
        return Optional.of(response);
    } else {
        return Optional.empty();
    }
}

// ✅ Versión simplificada que deberías ser capaz de escribir tú:
public Optional<UserResponse> findUser(String email) {
    return userRepository.findByEmail(email)
        .map(user -> new UserResponse(user.getName(), user.getEmail()));
}
```

> 💡 Si no puedes simplificar el código de la IA, es señal de que no lo entiendes del todo. Si sí puedes, es señal de que tu criterio técnico sigue activo.

### Práctica de "Whiteboard First"

Para problemas de lógica de negocio compleja (cálculo de precios, reglas de elegibilidad, flujos de estado), **diseña el algoritmo en papel o en comentarios antes de activar el autocompletado**:

```java
// 1. Escribe primero el algoritmo en pseudocódigo como comentarios:
public Money calculateLoyaltyDiscount(Order order, Customer customer) {
    // Si el cliente tiene más de 2 años → descuento del 10%
    // Si además ha gastado más de 1000€ en los últimos 6 meses → descuento adicional del 5%
    // El descuento máximo total es del 15%
    // No aplicar si el pedido contiene productos en liquidación
}

// 2. Ahora activa el autocompletado. Con ese contexto, la sugerencia será
//    mucho más precisa y tú podrás verificarla contra tu diseño previo.
```

### Rotación deliberada: escribe sin IA

Dedica un tiempo semanal (una kata, un módulo pequeño) a escribir código **con el autocompletado desactivado**. No para demostrar nada, sino para mantener el ritmo y detectar dónde has delegado sin darte cuenta.

```
Señales de alerta de dependencia:
  → Sientes ansiedad cuando el autocompletado no sugiere nada
  → No recuerdas la firma de métodos de Spring que usas cada día
  → Aceptas sugerencias sin leerlas porque "suelen estar bien"
  → Tus code reviews se centran en si compila, no en si es correcto
```

### Tests como red de seguridad activa

La forma más efectiva de no depender ciegamente del autocompletado es tener tests que fallen cuando la IA se equivoca. El test no es un paso final: es la verificación que hace posible confiar en la sugerencia.

```java
// El autocompletado generó calculateRefund(). Antes de seguir, escribe el test:
@Test
void refund_should_sum_only_refundable_payments() {
    Order order = OrderTestBuilder.withPayments(
        Payment.refundable(Money.of(100)),
        Payment.nonRefundable(Money.of(50)),  // ← este no debe sumarse
        Payment.refundable(Money.of(75))
    );

    Money refund = service.calculateRefund(order);

    assertThat(refund).isEqualTo(Money.of(175)); // 100 + 75, no 225
}

// Si el test pasa, la sugerencia era correcta.
// Si falla, acabas de evitar un bug en producción.
```

> ⚠️ **Anti-patrón frecuente:** Escribir el test *después* de entender el código generado y ajustarlo para que pase. Ese test no te protege de nada: está verificando lo que la IA decidió, no lo que el negocio requiere.
