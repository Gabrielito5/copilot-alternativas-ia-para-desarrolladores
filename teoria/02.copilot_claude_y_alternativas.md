# Tema 2. COPILOT, CLAUDE Y ALTERNATIVAS EN MODELOS DE IA PARA DESARROLLADORES: ESCOGIENDO LA MEJOR ESTRATEGIA MULTI-HERRAMIENTA PARA SPRING

No todas las IAs son iguales ni sirven para lo mismo. Elegir la herramienta incorrecta para una tarea concreta es tan costoso como no usar IA en absoluto. En este tema construimos un criterio de selección profesional basado en las capacidades reales de cada herramienta y cada versión de modelo en el contexto de desarrollo backend con Spring.

---

## 1. Comparativa entre alternativas existentes

El ecosistema de asistentes de IA para desarrolladores ha madurado rápidamente. Hoy existen categorías bien diferenciadas por su modo de operación y su punto fuerte.

* **Plataformas multi-modelo con integración en IDE:** GitHub Copilot es el ejemplo más representativo. Ya no es un asistente de autocompletado ligado a un único modelo: permite seleccionar el modelo subyacente (GPT-4o, Claude Sonnet, Gemini, etc.) tanto para el completado inline como para el modo agente. JetBrains AI sigue un enfoque similar. La plataforma es la integración; el modelo es intercambiable.
* **Agentes de terminal y sistema de ficheros:** Claude Code, Aider, Cursor Agent. Pueden leer el proyecto completo, ejecutar comandos (`mvn`, `docker`, `git`) e iterar sobre errores de compilación sin salir del terminal. Operan fuera del IDE pero con mayor autonomía.
* **Modelos de razonamiento profundo (vía chat o API):** Claude (claude.ai / API), ChatGPT (OpenAI), Gemini (Google). Óptimos para problemas de diseño, decisiones de arquitectura o análisis de complejidad que requieren razonamiento multi-paso. Se accede a ellos directamente, sin capa IDE.
* **Modelos autohospedados y privados:** Llama 3, Mistral, CodeLlama, Granite (IBM). Permiten operar sin enviar código a terceros. Coste operativo variable; calidad inferior en tareas complejas de Java/Spring.

## 2. Qué aporta Claude vs Copilot en razonamiento y diseño

La comparativa ya no es "Copilot vs Claude" como si fueran dos mundos separados. Copilot puede usar Claude como motor. La diferencia real está en el **modo de operación** y en los **estándares y ecosistema** que cada plataforma ha generado.

* **GitHub Copilot — Fortalezas:**
    * Completado inline de alta velocidad integrado en el flujo de escritura del IDE.
    * Integración nativa en IntelliJ/VS Code sin fricción de configuración.
    * **Selección de modelo en modo agente:** En VS Code, el modo agente de Copilot permite elegir el modelo subyacente (GPT-4o, Claude Sonnet, Gemini Pro, etc.). Esto lo convierte en una plataforma agnóstica al modelo, no en un producto de un único proveedor.
    * Muy efectivo para tareas de "relleno": mappers, getters/setters, DTOs repetitivos, conversiones.
    * `copilot-instructions.md`: fichero de instrucciones para estandarizar el comportamiento del agente en el repositorio.

* **Claude (Anthropic) — Fortalezas:**
    * Razonamiento estructurado y multi-paso: ideal para diseñar una arquitectura, evaluar trade-offs o descomponer un bounded context.
    * Ventana de contexto larga: puede analizar un módulo completo o varios ficheros simultáneamente manteniendo coherencia.
    * **Claude Code como agente de terminal:** lee, escribe, compila y testea en bucle autónomo con capacidad de corrección de errores. Opera directamente en el sistema de ficheros, sin necesidad del IDE.
    * Menor tendencia a "alucinaciones de API" en Spring Boot moderno (3.x / Jakarta EE).
    * **Anthropic como creador de estándares del ecosistema agéntico:** Claude no es solo un modelo; Anthropic ha definido convenciones que se están convirtiendo en estándar de facto en el desarrollo asistido por agentes:
        * **`CLAUDE.md`:** Fichero de instrucciones del repositorio que el agente lee automáticamente al iniciar una sesión. Equivalente a un "briefing" de proyecto. Otros agentes están adoptando el mismo patrón.
        * **Skills / Slash Commands:** Sistema de comandos personalizables que extienden el comportamiento del agente (ej: `/commit`, `/review-pr`). Permite codificar flujos de trabajo del equipo como comandos reutilizables.
        * **MCP (Model Context Protocol):** Protocolo abierto creado por Anthropic para conectar agentes a herramientas externas (bases de datos, APIs, sistemas de ficheros remotos). Adoptado ya por otros proveedores y herramientas del ecosistema.
        * **Hooks:** Mecanismo para ejecutar acciones automáticas antes o después de que el agente realice ciertas operaciones (leer, escribir, ejecutar comandos).

* **La diferencia clave en diseño:** Ante una pregunta como "¿Cómo estructuro los bounded contexts de este sistema de pedidos?", un modelo rápido responderá con código inmediato; Claude con razonamiento profundo responderá con un análisis de opciones, sus pros y contras, y luego propone la implementación. Para decisiones de arquitectura, esa capa de razonamiento previo es imprescindible — independientemente de si se accede a Claude vía Claude Code o vía Copilot en modo agente.

## 3. Eligiendo la versión del modelo: razonamiento profundo vs. modelos rápidos

Una de las decisiones más impactantes — y más ignoradas — no es qué proveedor elegir, sino **qué versión o tier del modelo usar para cada tarea**. Usar siempre el modelo más potente es un error tan grave como usar siempre el más barato.

### El espectro de capacidad y coste

Cada proveedor ofrece una familia de modelos con diferentes compromisos entre velocidad, coste y profundidad de razonamiento:

> **Nota:** Tabla actualizada a marzo 2026. El ecosistema de modelos evoluciona rápidamente; los nombres concretos cambiarán, pero la lógica de tiers es estable.

| Tier | Ejemplos (marzo 2026) | Velocidad | Coste | Cuándo usarlo |
| :--- | :--- | :--- | :--- | :--- |
| **Razonamiento profundo** | Claude Opus 4.6, GPT-5.2 Pro (Thinking), Gemini 3.1 Pro / 3 Deep Think | Lento | Alto | Diseño de arquitectura, análisis de deuda técnica, decisiones sin respuesta obvia |
| **Balanceado** | Claude Sonnet 4.6, GPT-5 / GPT-5.2, Gemini 3 Flash | Medio | Medio | Implementación multi-módulo, code review, generación de especificaciones |
| **Rápido / Económico** | Claude Haiku 4.5, GPT-5.2 Instant, Gemini 3.1 Flash Lite | Muy rápido | Bajo | Completado inline, generación de boilerplate, tests unitarios simples, refactoring mecánico |

## 4. Cuándo sacar el problema del IDE a un modelo "de pensamiento"

No todo problema se resuelve con completado inline. Hay una regla práctica para saber cuándo cambiar de herramienta — y de tier de modelo.

### Política práctica de selección de versión

* **Usa un modelo de razonamiento profundo cuando:**
    * La respuesta incorrecta tiene coste alto (decisión de arquitectura, evaluación de trade-offs de seguridad, diseño de un bounded context).
    * El problema requiere sintetizar múltiples fuentes de contexto y mantener coherencia a lo largo de un razonamiento largo.
    * Estás generando un ADR, una especificación OpenAPI o un documento técnico que otros van a ejecutar.
    * Llevas más de dos turnos de chat y el modelo rápido "no termina de entender" el problema.

* **Usa un modelo balanceado cuando:**
    * Implementas funcionalidad nueva que afecta a varios módulos y necesitas coherencia arquitectónica.
    * Haces code review de un PR con lógica de negocio no trivial.
    * Generates tests de integración con Testcontainers o contratos con Spring Cloud Contract.

* **Usa un modelo rápido/económico cuando:**
    * Completado inline de código repetitivo (mappers, DTOs, getters).
    * Tests unitarios de métodos aislados.
    * Renombrado, reformateo o refactoring mecánico sin lógica de negocio.
    * Consultas rápidas sobre APIs ("¿Cuál es el parámetro de `@Cacheable` para TTL?").
    * Generación de código scaffolding que vas a revisar y ajustar igualmente.

### El coste real de elegir mal el tier

* **Usar siempre el modelo más potente:** Coste económico innecesario en tareas simples. Latencia que interrumpe el flujo. En entornos con límites de API, agota el presupuesto en tareas de poco valor.
* **Usar siempre el modelo más barato:** Razonamientos superficiales en problemas complejos. Soluciones plausibles pero incorrectas que pasan el review inicial y fallan en producción. El coste de arreglarlo supera con creces el ahorro en tokens.

### Recomendación para proyectos Spring

> Establece en el `CLAUDE.md` o en la política del equipo qué tier de modelo se usa por defecto para cada tipo de tarea. No lo dejes a criterio individual de cada desarrollador en cada sesión.

**Aclaración importante:** `CLAUDE.md` es un fichero de instrucciones en lenguaje natural — no cambia el modelo de forma programática. Su función aquí es documentar la política del equipo para que cualquier desarrollador (y el propio agente) sepa qué tier debe usar en cada contexto. El modelo por defecto real se establece en `.claude/settings.json`. Ambos ficheros trabajan juntos: uno define la política, el otro la aplica.

#### Ejemplo: sección de política de modelos en `CLAUDE.md`

```markdown
## Política de selección de modelo

El modelo por defecto del proyecto está configurado en `.claude/settings.json` (Claude Sonnet —
tier balanceado). Antes de iniciar una tarea, selecciona el tier adecuado:

| Tipo de tarea                                      | Tier          | Modelo sugerido     |
|----------------------------------------------------|---------------|---------------------|
| Diseño de arquitectura, ADRs, análisis de deuda    | Alto          | claude-opus-4-6     |
| Especificaciones OpenAPI / AsyncAPI                | Alto          | claude-opus-4-6     |
| Implementación multi-módulo, code review           | Balanceado    | claude-sonnet-4-6   |
| Tests de integración, refactoring con contexto     | Balanceado    | claude-sonnet-4-6   |
| Boilerplate, mappers, DTOs, tests unitarios simples| Rápido        | claude-haiku-4-5    |
| Consultas puntuales de API o sintaxis              | Rápido        | claude-haiku-4-5    |

Cambiar el modelo para una sesión concreta: `claude --model claude-opus-4-6`
Cambiar el modelo dentro de una sesión activa: `/model claude-opus-4-6`
```

#### Ejemplo: `.claude/settings.json` — modelo por defecto del proyecto

El fichero `.claude/settings.json` (en la raíz del repositorio, versionado con git) establece el modelo que Claude Code usa si no se especifica ninguno explícitamente:

```json
{
  "model": "claude-sonnet-4-6"
}
```

Para sobreescribirlo de forma personal sin afectar al equipo, usar `~/.claude/settings.json` (fichero de usuario, fuera del repositorio).

### Cómo seleccionar el modelo en Claude Code

Claude Code ofrece cuatro mecanismos con el siguiente orden de prioridad (de mayor a menor):

1. **Comando `/model` dentro de una sesión activa** — cambia el modelo para el resto de la sesión sin salir:
   ```bash
   /model claude-opus-4-6      # tier alto: arquitectura, ADRs
   /model claude-sonnet-4-6    # tier balanceado: implementación
   /model claude-haiku-4-5     # tier rápido: boilerplate, consultas
   ```

2. **Flag `--model` al lanzar Claude Code** — establece el modelo para esa sesión concreta:
   ```bash
   claude --model claude-opus-4-6
   claude -m sonnet              # forma abreviada también válida
   ```

3. **Variable de entorno** — útil en scripts CI o en la configuración del terminal:
   ```bash
   export ANTHROPIC_MODEL=claude-sonnet-4-6
   claude  # usará el modelo de la variable de entorno
   ```

4. **`.claude/settings.json`** — modelo por defecto del proyecto para todo el equipo (ver ejemplo anterior).

### Cómo seleccionar el modelo en GitHub Copilot

La selección de modelo en Copilot se controla de forma **independiente** para el chat/agente y para las sugerencias inline — son dos configuraciones separadas.

**En VS Code — Chat y modo agente:**
- Abre el panel de Copilot Chat.
- En la parte inferior del panel, haz clic en el selector **CURRENT-MODEL**.
- Elige el modelo: `Claude Sonnet 4.6`, `GPT-5`, `Gemini 3 Flash`, etc.
- Para el modo agente, selecciona primero `Agent` en el desplegable de modo y luego cambia el modelo en el mismo selector.
- El modelo puede cambiarse entre conversaciones; también es posible regenerar la última respuesta con un modelo distinto haciendo clic en el icono de reintento.

**En JetBrains — Sugerencias inline:**
- Ve a `File` → `Settings` (o `Preferences` en macOS).
- Navega a `Tools` → `GitHub Copilot` → `Completions`.
- Despliega `Model for completions` y selecciona el modelo deseado.

**En JetBrains — Chat:**
- Abre el panel de AI Chat.
- Usa el selector de modelo en la barra superior del chat.

> ⚠️ **Anti-patrón frecuente en equipos:** Cada desarrollador usa el modelo que tiene por defecto sin pensar en el tipo de tarea. El resultado es o bien coste innecesario (todo con Opus para hacer un mapper) o bien calidad insuficiente (diseño de arquitectura con Haiku). Una política explícita en `CLAUDE.md` + un `settings.json` con el tier balanceado como defecto es el punto de partida correcto para la mayoría de proyectos.

## 5. Cómo pasar contexto sin filtrar código sensible

El contexto es el activo más valioso para una IA, pero el código de producción puede contener información que no debe salir de la organización.

* **Qué nunca debe enviarse a un modelo externo:**
    * Credenciales, tokens, contraseñas o valores de `application-prod.properties` / `application-prod.yml`.
    * Datos personales de usuarios reales (PII/GDPR): logs con emails, IDs de clientes, datos de tarjetas.
    * Algoritmos de negocio propietarios o lógica de pricing clasificada como secreto comercial.
    * Claves privadas, certificados o configuraciones de seguridad de infraestructura productiva.

* **Técnicas para contextualizar sin exponer:**
    * **Anonimización de ejemplos:** Sustituir valores reales por ficticios antes de pegar en el chat (`user@empresa.com` → `user@example.com`, IDs reales → UUIDs genéricos).
    * **Resumen estructural en lugar del código fuente:** Describir la arquitectura ("Tengo un `OrderService` que depende de un `PaymentGatewayPort`...") sin pegar la implementación completa.
    * **Variables de entorno en los ficheros de configuración:** Asegurarse de que los ficheros que se comparten usan `${DB_PASSWORD}` y no el valor real.
    * **`.gitignore` como filtro:** Verificar que `application-prod.yml`, `.env` y ficheros de secretos están ignorados antes de dar acceso al agente al repositorio.
    * **Modo de revisión offline:** Para módulos especialmente sensibles, usar un modelo autohospedado o Claude Enterprise con acuerdo de no retención de datos.

* **Política práctica para equipos:** Definir una lista de exclusión explícita en el `CLAUDE.md` del proyecto indicando qué paquetes o ficheros el agente no debe leer. Claude Code respeta estas instrucciones si están bien especificadas.

## 6. Alternativas y criterios: privacidad, integración, calidad de Java/Spring, coste

Matriz de decisión para seleccionar herramienta según las restricciones del proyecto o la organización.

> **Nota:** Tabla actualizada a marzo 2026. Precios en USD; pueden variar por región y plan.

| Herramienta | Privacidad | Integración IDE | Calidad Java/Spring | Coste aprox. | Modo Agente |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **GitHub Copilot** | Media (código enviado a GitHub) | ⭐⭐⭐⭐⭐ (nativa VS Code / IntelliJ) | Depende del modelo elegido | Free / Pro $10 / Pro+ $39 / Business $19 por usuario/mes | Sí, multi-modelo (VS Code + IntelliJ) |
| **Claude Code** | Media-Alta (configuración Enterprise) | ⭐⭐⭐⭐ (extensión VS Code nativa + plugin JetBrains) | Muy Alta | Claude Pro $20/mes o API por consumo | Sí, nativo (terminal + IDE) |
| **ChatGPT / GPT-5** | Media (OpenAI) | ⭐⭐ (plugins limitados) | Muy Alta | ~$20/mes o API por consumo | Sí (ChatGPT Tasks; API con tools) |
| **Gemini Code Assist** | Media (Google) | ⭐⭐⭐⭐ (VS Code + IntelliJ nativo) | Alta | Gratuito para individuos; Google AI Pro $20/mes | Sí (modo agente en VS Code + IntelliJ desde ago 2025) |
| **JetBrains AI** | Media (JetBrains Cloud) | ⭐⭐⭐⭐⭐ (IntelliJ nativa) | Alta (multi-modelo + Claude Agent integrado) | ~$10/mes/dev | Sí (Claude Agent integrado de forma nativa) |
| **Llama 4 (local)** | ⭐⭐⭐⭐⭐ (sin salida de datos) | ⭐ (manual) | Media-Alta | Infraestructura propia | Experimental |
| **Granite (IBM/local)** | ⭐⭐⭐⭐⭐ (enterprise) | ⭐ (manual) | Media-Alta (Java) | Licencia empresarial | No |

* **Criterio de privacidad:** Si la organización opera en sectores regulados (banca, salud, administración pública), evaluar primero modelos autohospedados o acuerdos enterprise con cláusula de no retención de datos.
* **Criterio de calidad Java/Spring:** La calidad ya no depende solo de la plataforma sino del modelo seleccionado. Claude Sonnet 4.6 u Opus 4.6 vía Copilot, JetBrains AI o directamente vía Claude Code son actualmente los que mejor comprenden el ecosistema Spring Boot 3.x / Jakarta EE.
* **Criterio de coste:** La mayor trampa es el coste oculto del tiempo de integración y cambio de contexto. Una herramienta con peor integración puede ser más cara en la práctica aunque su licencia sea menor.

## 7. Política de selección por tarea y tier de modelo

Una política explícita de "qué herramienta, qué modelo y qué tier para qué tarea" elimina la parálisis de elección y estandariza el flujo del equipo.

| Tipo de Tarea | Herramienta | Tier de Modelo | Justificación |
| :--- | :--- | :--- | :--- |
| **Coding (boilerplate, mappers, DTOs)** | Copilot inline / Claude Code | Rápido / Económico | Patrón predecible, velocidad > profundidad |
| **Coding (nueva funcionalidad multi-módulo)** | Claude Code (agente terminal) | Balanceado | Necesita contexto amplio y ejecutar validaciones |
| **Code Review (lógica y seguridad)** | Claude (chat) / Copilot agente | Balanceado - Alto | Razonamiento para detectar bugs sutiles |
| **Diseño de arquitectura** | Claude (chat directo) | Alto (razonamiento) | Análisis de trade-offs y generación de ADRs |
| **Documentación técnica (OpenAPI, ADR)** | Claude (chat o API) | Alto (razonamiento) | Especificaciones que otros van a ejecutar: no admiten ambigüedad |
| **Refactoring de legacy** | Claude Code (agente) + revisión humana | Balanceado | Contexto amplio, ejecución de tests, múltiples iteraciones |
| **Tests unitarios simples** | Copilot / Claude Code | Rápido | Tests de métodos aislados: patrón repetitivo |
| **Tests de integración complejos** | Claude Code | Balanceado | Testcontainers, contratos, lógica de orquestación |
| **Análisis de deuda técnica** | Claude (chat, ficheros completos) | Alto (razonamiento) | Visión holística del módulo o servicio |
| **Consultas puntuales de API** | Cualquiera | Rápido | Preguntas factuales sin razonamiento complejo |

## 8. Seleccionando el modelo IA para este curso: Claude

La elección de Claude como herramienta principal de este curso no es aleatoria ni de preferencia personal; responde a criterios técnicos concretos alineados con los objetivos del curso.

* **Razón 1 — Calidad de razonamiento para Spring Boot 3.x:** Claude demuestra comprensión profunda del ecosistema Spring moderno: conoce la diferencia entre `@Bean` en `@Configuration` vs `@Component`, maneja correctamente Jakarta EE, entiende el modelo de programación reactiva de Spring WebFlux y las implicaciones de las versiones del Spring Security actual.

* **Razón 2 — Claude Code como agente de terminal:** Es la herramienta que más se alinea con la filosofía del curso: no solo sugiere código sino que lo ejecuta, lo testa y lo itera. Permite demostrar el paradigma agéntico de forma real, no simulada.

* **Razón 3 — Ventana de contexto larga:** Para trabajar con proyectos Spring reales (con su `pom.xml`, múltiples capas, tests) se necesita contexto amplio. Claude mantiene coherencia a lo largo de conversaciones largas mejor que alternativas con ventanas de contexto más reducidas.

* **Razón 4 — Capacidad de seguir especificaciones estrictas (SDD):** La metodología central del curso es Spec-Driven Development. Claude es especialmente efectivo implementando código a partir de contratos OpenAPI o AsyncAPI sin desviarse de la especificación.

* **Razón 5 — Anthropic como creador de estándares del desarrollo agéntico:** Más allá del modelo en sí, el ecosistema que Anthropic ha construido alrededor de Claude se está convirtiendo en referencia para el sector. Aprender con Claude significa aprender con las herramientas y convenciones que otros agentes están adoptando: `CLAUDE.md`, MCP, Skills, Hooks. No son características propietarias — son estándares emergentes.

* **Nota sobre Copilot:** Se usará Copilot como referencia comparativa en los primeros temas. Los patrones de prompting y los flujos de trabajo aprendidos son transferibles entre herramientas; la diferencia está en el modo de operación, las capacidades del agente y el ecosistema de estándares alrededor de cada plataforma.
