# Tema 1. El Rol de la IA en el desarrollo profesional con Spring

## 1. Evolución del desarrollo asistido por IA y "Agentic Development"

No estamos ante una mejor versión de Google o ante un autocompletar con esteroides. Estamos ante un cambio de paradigma en la relación humano-máquina donde **la abstracción sube de nivel**.

La integración de la IA en el desarrollo de software no ha sido un salto repentino, sino una **evolución gradual** en tres eras claramente diferenciadas. Cada era ha incrementado la **autonomía** de la IA y ha cambiado el **rol del desarrollador**.

```
Desarrollador escribe todo → IA sugiere líneas → IA conversa → IA ejecuta y actúa
         (2020)                   (2021)           (2023)         (2025+)
```

---

### Era 1: Autocomplete — "Ghost Text" (2021-2022)

#### ¿Qué es?
El **Autocomplete** (popularizado por GitHub Copilot) es un modelo de IA que predice las **siguientes líneas de código** basándose en el contexto inmediato del archivo abierto.

#### Características
| Aspecto | Detalle |
|---|---|
| **Input** | Las líneas anteriores y el nombre del archivo |
| **Output** | Una o varias líneas de código sugeridas ("ghost text") |
| **Contexto** | Limitado al archivo actual y, como mucho, archivos abiertos en tabs |
| **Acción** | Ninguna. Solo sugiere, tú aceptas o rechazas |

#### Ejemplo Real: Autocompletado en Spring Boot

Imagina que estás en un archivo `UserService.java` y escribes:

```java
// Método para buscar un usuario por email
public Optional<User> findByEmail(
```

El autocomplete sugeriría algo como:

```java
public Optional<User> findByEmail(String email) {
    return userRepository.findByEmail(email);
}
```

**¿Está bien?** A veces sí. Pero el autocomplete:
- ❌ No sabe si `userRepository` tiene ese método.
- ❌ No sabe si el tipo de retorno debería ser `Optional` o lanzar una excepción.
- ❌ No puede verificar que compila.
- ❌ No puede ejecutar los tests para ver si rompe algo.

#### Limitaciones Clave
1. **Sin contexto global**: No entiende la arquitectura de tu proyecto.
2. **Sin ejecución**: No puede correr `mvn compile` para verificar su sugerencia.
3. **Sin memoria**: Cada sugerencia es independiente; no recuerda lo que sugirió antes.
4. **Sesgo hacia lo popular**: Tiende a generar código "promedio" de Stack Overflow, no código adaptado a TU proyecto.

#### Mentalidad del Desarrollador en esta Era

> "Ayúdame a escribir este método más rápido."

El desarrollador sigue siendo el **conductor**. La IA es un **copiloto pasivo** que solo completa frases.

### Era 2: Chat — Razonamiento Conversacional (2023-2024)

#### ¿Qué es?
Los modelos conversacionales como ChatGPT, Claude Web o Copilot Chat permiten hacer **preguntas en lenguaje natural** y recibir explicaciones, código, y razonamiento detallado.

#### Características
| Aspecto | Detalle |
|---|---|
| **Input** | Pregunta en lenguaje natural + código copiado manualmente |
| **Output** | Explicaciones, bloques de código, alternativas |
| **Contexto** | Solo lo que tú le pegas en el chat |
| **Acción** | Ninguna. Tú copias y pegas el resultado |

#### Ejemplo Real: Pidiendo Código por Chat

Tú preguntas en el chat:

> "Necesito un endpoint REST en Spring Boot que reciba un JSON con `name` y `email`, valide los campos, y guarde un usuario en la base de datos. Usa arquitectura hexagonal."

La IA responde con un bloque de código largo:

```java
// UserController.java
@RestController
@RequestMapping("/api/users")
public class UserController {

    private final CreateUserUseCase createUserUseCase;

    public UserController(CreateUserUseCase createUserUseCase) {
        this.createUserUseCase = createUserUseCase;
    }

    @PostMapping
    public ResponseEntity<UserResponse> createUser(
            @Valid @RequestBody CreateUserRequest request) {
        User user = createUserUseCase.execute(
            new CreateUserCommand(request.name(), request.email())
        );
        return ResponseEntity.status(HttpStatus.CREATED)
            .body(UserResponse.from(user));
    }
}
```

```java
// CreateUserUseCase.java (Puerto de Entrada)
public interface CreateUserUseCase {
    User execute(CreateUserCommand command);
}
```

```java
// CreateUserService.java (Implementación del caso de uso)
@Service
public class CreateUserService implements CreateUserUseCase {

    private final UserRepository userRepository;

    public CreateUserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    @Override
    public User execute(CreateUserCommand command) {
        // Lógica de negocio
        if (userRepository.existsByEmail(command.email())) {
            throw new EmailAlreadyExistsException(command.email());
        }
        User user = User.create(command.name(), command.email());
        return userRepository.save(user);
    }
}
```

#### El Problema: "The Copy-Paste Gap"

Este código puede estar **muy bien**, pero:

1. **Tú** tienes que copiar cada bloque al archivo correcto.
2. **Tú** tienes que crear las clases que faltan (`CreateUserCommand`, `UserResponse`, `User`, etc.).
3. **Tú** tienes que verificar que compila.
4. **Tú** tienes que correr los tests.
5. Si algo no funciona, tienes que **volver al chat**, pegar el error, esperar otra respuesta, y repetir.

```
┌──────────────────────────────────────────────────────────┐
│                    THE COPY-PASTE GAP                     │
│                                                          │
│  Navegador (Chat)  ──copy──>  IDE  ──paste──>  Terminal  │
│       │                                          │       │
│       │       Error! ← ← ← ← ← ← ← ← ← ← ← ←│       │
│       │                                                  │
│       └────── copy error, paste back to chat ────────────│
│                                                          │
│                    💀 CICLO INFINITO                       │
└──────────────────────────────────────────────────────────┘
```

#### Limitaciones Clave
1. **Sin acceso a tu proyecto**: No puede leer tu `pom.xml`, tu estructura de paquetes, ni tus convenciones.
2. **Sin ejecución**: No puede ejecutar `mvn test` ni ver si su código compila.
3. **Pérdida de contexto**: Si la conversación se alarga, olvida lo que dijo al principio.
4. **El humano es el integrador**: Tú eres el "pegamento" entre el código generado y tu proyecto real.

#### Mentalidad del Desarrollador en esta Era

> "Explícame cómo hacer esto y dame el código para copiar."

El desarrollador es un **integrador manual**. La IA es un **consultor externo** que da consejos pero no se ensucia las manos.

### Era 3: Agentes — Ejecución Autónoma (2025+)

#### ¿Qué es?
Un **Agente de IA** es un modelo que no solo genera código, sino que **opera directamente sobre tu proyecto**: lee archivos, crea carpetas, edita código, ejecuta comandos en terminal, interpreta errores, y corrige iterativamente.

#### Características
| Aspecto | Detalle |
|---|---|
| **Input** | Instrucciones en lenguaje natural (prompts) |
| **Output** | Archivos creados/modificados, comandos ejecutados, logs de resultados |
| **Contexto** | Todo tu sistema de archivos, terminal, y herramientas CLI |
| **Acción** | **Directa**. Ejecuta, verifica, y corrige |

#### Herramientas que Definen esta Era

| Herramienta | Tipo | Descripción |
|---|---|---|
| **Claude Code** | CLI Agent | Opera en terminal, acceso total a archivos y comandos |
| **Copilot Agent Mode** | IDE Agent | Integrado en VS Code, ejecuta acciones dentro del editor |
| **Cursor / Windsurf** | IDE Agent | IDEs con agentes nativos integrados |
| **Aider** | CLI Agent | Agente CLI open-source para Git |
| **Cline / Roo Code** | IDE Extension | Extensiones que convierten VS Code en un entorno agéntico |

#### Ejemplo Real: El Mismo Caso, con un Agente

En lugar de copiar y pegar, le dices al agente directamente en terminal:

```bash
# Prompt al agente (en tu terminal, dentro de tu proyecto)
> Crea un endpoint POST /api/users que reciba name y email,
  valide los campos, y guarde un usuario en la base de datos.
  Usa arquitectura hexagonal siguiendo la estructura existente.
  Escribe tests unitarios y de integración.
  Ejecuta los tests y asegúrate de que todos pasan.
```

**Lo que el agente hace automáticamente:**

```
1. 🔍 Lee tu proyecto
   - Analiza pom.xml → Detecta Spring Boot 3.4, Java 21
   - Analiza src/main/ → Detecta estructura hexagonal existente
   - Analiza tests existentes → Detecta patrón de testing

2. ⌨️ Crea archivos
   - src/main/java/.../adapter/in/web/UserController.java
   - src/main/java/.../application/port/in/CreateUserUseCase.java
   - src/main/java/.../application/service/CreateUserService.java
   - src/main/java/.../domain/model/User.java
   - src/main/java/.../adapter/out/persistence/UserJpaEntity.java
   - (... y todos los archivos necesarios)

3. 🖥️ Ejecuta
   $ mvn compile → ✅ BUILD SUCCESS
   $ mvn test    → ❌ FAILURE: UserRepositoryTest - table 'users' not found

4. 🔄 Corrige
   - Lee el error → Falta el script de migración Flyway
   - Crea: src/main/resources/db/migration/V2__create_users_table.sql
   $ mvn test    → ✅ BUILD SUCCESS (12 tests passed)

5. ✅ Reporta
   "He creado el módulo de usuarios con 8 archivos.
    12 tests pasan (4 unitarios, 4 integración, 4 controller).
    ¿Quieres que revise algo?"
```

#### El Cambio Fundamental

```
┌──────────────────────────────────────────────────────┐
│              ANTES (Chat)          AHORA (Agente)     │
│                                                      │
│  Tú escribes código       Tú escribes especificación │
│  Tú copias y pegas        El agente implementa       │
│  Tú compilas              El agente compila           │
│  Tú debuggeas             El agente debuggea          │
│  Tú corres tests          El agente corre tests       │
│  Tú eres el obrero        Tú eres el arquitecto       │
│                                                      │
│  Tu rol:  IMPLEMENTAR  →  Tu rol:  REVISAR y GUIAR   │
└──────────────────────────────────────────────────────┘
```

#### Mentalidad del Desarrollador en esta Era

> "Aquí tienes la especificación. Implementa el módulo y avísame cuando los tests pasen."

El desarrollador es un **arquitecto y revisor**. La IA es un **desarrollador junior incansable** que ejecuta las instrucciones.

## 2. Qué problemas reales resuelve en backend y microservicios

En el desarrollo de backend con Spring, el problema no es "escribir código", sino gestionar la **complejidad de las abstracciones** y la **coherencia del ecosistema**.

---

#### 2.1. Gestión de la Carga Cognitiva en Arquitecturas Distribuidas
* **Contextualización rápida:** Uso de la IA para explicar flujos de peticiones entre servicios, interceptores y trazabilidad.
* **Identificación de dependencias:** Resolución de conflictos en el classpath (`pom.xml` / `build.gradle`).

#### 2.2. Automatización de la "Burocracia del Framework"
* **Mapeos Complejos:** Implementación de mappers (MapStruct) entre JPA, Dominio y DTOs.
* **Spring Security 6.x:** Generación de `SecurityFilterChain` sin memorizar APIs fluidas cambiantes.

#### 2.3. Modernización y Refactorización de Legacy
* **Migraciones:** Automatización de Java 8 a 21 y Spring Boot 2.x a 3.x (Jakarta EE).
* **Sustitución de API:** De `RestTemplate` a `RestClient` o `WebClient`.

#### 2.4. Estrategias de Testing de Integración
* **Infraestructura:** Generación de código para **TestContainers** y mocks con **WireMock**.
* **Contratos:** Creación de tests de Spring Cloud Contract o Pact.

## 3. Productividad vs calidad: métricas útiles (lead time, defect rate, PR cycle time)

El error común en el management es medir la productividad por "líneas de código". Para un Senior de Spring, la IA no es una fábrica de líneas (lo que aumentaría la deuda técnica), sino un acelerador de la entrega de valor y un optimizador de la carga cognitiva.

---

#### 3.1. Métricas de Flujo (Velocidad de entrega)
* **PR Cycle Time (Tiempo de ciclo de Pull Request):** Es la métrica donde la IA tiene un impacto más inmediato.
    * **Fase de Autoría:** Reducción drástica del tiempo para generar el esqueleto de controladores, servicios y DTOs.
    * **Fase de Revisión:** Uso de la IA para resumir cambios complejos en PRs de compañeros, permitiendo que el Senior actúe como "Editor Jefe" de forma más ágil.
* **Lead Time for Changes:** Tiempo total desde que se inicia un commit hasta que llega a producción. Al automatizar la configuración de beans, integraciones y scripts de despliegue, el cuello de botella técnico desaparece.

#### 3.2. Métricas de Calidad (Estabilidad y Robustez)
* **Defect Rate (Tasa de defectos):** ¿La IA introduce más errores?
    * *Enfoque Senior:* La IA debe usarse para aumentar la **cobertura de tests** (unitarios, integración con TestContainers, tests de mutación). Si se usa para generar código sin tests, la tasa de defectos sube; si se usa para generar tests robustos, la tasa baja.
* **Mean Time to Recovery (MTTR):** Capacidad de usar la IA para diagnosticar fallos en producción analizando trazas de Stacktraces de Spring o logs de microservicios en segundos.

#### 3.3. La paradoja de la "Falsa Productividad" (Code Bloat)
* **Métrica de Densidad de Código:** Un Senior debe vigilar que la IA no genere 500 líneas para algo que se resuelve con una anotación de Spring o una librería existente.
* **Eficiencia de Revisión:** Medir cuánto tiempo dedica el Senior a validar el código generado vs. escribirlo. Si la validación tarda más que la escritura manual, la herramienta está restando productividad.

---

| Métrica DORA | Impacto de la IA | Riesgo para el Senior |
| :--- | :--- | :--- |
| **Deployment Frequency** | **Aumenta:** Menos fricción en el setup de nuevos microservicios. | Generar "basura" técnica más rápido. |

## 4. Qué no debe delegarse a la IA (decisiones de dominio, riesgo, seguridad, compliance)

Para un Senior de Spring, la IA es un motor de ejecución, pero nunca el juez de la arquitectura. Delegar ciegamente el "qué" y el "por qué" (frente al "cómo") puede comprometer la integridad del sistema a largo plazo.

---

#### 4.1. El Diseño del Modelo de Dominio (Domain-Driven Design)
* **El contexto del negocio:** La IA no conoce la evolución futura de tu producto ni los matices del lenguaje ubicuo de tu empresa.
* **Límites de los Bounded Contexts:** Decidir si dos servicios deben compartir una entidad o si deben estar desacoplados es una decisión estratégica que la IA tiende a simplificar erróneamente.
* **Invariantes de negocio:** Las reglas que aseguran que los datos son consistentes según la lógica de negocio deben ser definidas y supervisadas por el humano.

#### 4.2. Arquitectura de Seguridad y Criptografía
* **Gestión de Secretos:** Nunca permitas que la IA sugiera dónde o cómo guardar claves API o certificados (evitar `application.properties` en texto plano).
* **Protocolos de Auth:** La IA puede sugerir configuraciones de OAuth2 o JWT funcionales pero inseguras (ej. algoritmos de firma débiles o falta de validación de claims).
* **Modelado de Amenazas:** La identificación de vectores de ataque específicos en tu infraestructura es una tarea puramente humana.

#### 4.3. Compliance, Privacidad y Aspectos Legales
* **GDPR/RGPD:** La IA no sabe qué campos en tu entidad `User` son datos de carácter personal que requieren cifrado en reposo o políticas de borrado.
* **Auditoría:** La lógica de trazabilidad y auditoría legal (quién hizo qué y cuándo) debe ser diseñada para cumplir con normativas específicas (fintech, salud, etc.).

#### 4.4. Decisiones de Riesgo Tecnológico
* **Selección de Stack:** La IA suele estar sesgada hacia lo más popular en su dataset de entrenamiento, no hacia lo más estable o lo que mejor encaja en tu infraestructura actual.
* **Trade-offs de Rendimiento:** La IA puede generar un código "limpio" pero con problemas de N+1 en JPA que solo un Senior detecta al analizar el volumen de datos real.

---

| Área | ¿Delegable? | Razón |
| :--- | :---: | :--- |
| **Escritura de DTOs/Mappers** | ✅ Verde | Tarea mecánica, bajo riesgo lógico. |
| **Configuración de Persistencia** | ⚠️ Ámbar | Riesgo de eficiencia (Lazy vs Eager, Índices). |

| **Política de CORS/CSP** | ❌ Rojo | Puerta abierta a ataques XSS/CSRF si se configura "para que funcione". |

## 5. Riesgos de uso acrítico y "overtrust"

El "overtrust" (exceso de confianza) es el fenómeno donde un desarrollador Senior empieza a validar código por "vibe" en lugar de por ejecución o análisis. En Spring, donde muchas cosas ocurren mediante "magia" de anotaciones y proxys en tiempo de ejecución, este riesgo se multiplica.

Lo peligroso del overtrust no es que la IA genere código incorrecto de forma obvia. Es que genera código **plausible**: código que compila, que parece razonable, que pasa una revisión rápida, y que falla semanas después en producción bajo condiciones específicas que nadie reprodujo en desarrollo.

---

### 5.1. El Sesgo de Automatización y la Ceguera del Revisor

La psicología del trabajo asistido por automatización lleva décadas documentada en ingeniería aeronáutica y de procesos industriales. El fenómeno es conocido: cuando un sistema automatizado funciona bien el 95% del tiempo, el operador humano deja de prestar atención al 5% restante.

En el desarrollo de software ocurre exactamente lo mismo. Cuando la IA acierta de forma consistente durante semanas, el revisor empieza a leer el código de forma diagonal, buscando confirmación en lugar de búsqueda activa de errores.

**El caso del proxy de Spring:** Esta trampa tiene una manifestación concreta y silenciosa en Spring. El mecanismo de proxys AOP de Spring (que gestiona `@Transactional`, `@Async`, `@Cacheable`, etc.) solo intercepta llamadas desde **fuera** del bean. Las llamadas internas dentro de la misma clase saltan el proxy.

La IA genera este error con frecuencia porque el código resultante es sintácticamente impecable:

```java
@Service
public class OrderService {

    // ❌ MAL — La IA genera esto porque "parece lógico"
    // El proxy de Spring NO intercepta esta llamada cuando createOrder()
    // es invocado desde otro método de la MISMA clase.
    @Transactional
    private void validateAndPersistOrder(Order order) {
        if (order.getTotal().compareTo(BigDecimal.ZERO) <= 0) {
            throw new InvalidOrderException("El total debe ser positivo");
        }
        orderRepository.save(order);
    }

    public Order createOrder(CreateOrderCommand command) {
        Order order = Order.from(command);
        validateAndPersistOrder(order); // ← llamada interna: @Transactional ignorado
        eventPublisher.publish(new OrderCreatedEvent(order));
        return order;
    }
}
```

```java
// ✅ BIEN — La transacción debe estar en el método público que el proxy sí intercepta,
// o la lógica interna debe extraerse a otro @Service independiente.
@Service
public class OrderService {

    @Transactional
    public Order createOrder(CreateOrderCommand command) {
        Order order = Order.from(command);
        if (order.getTotal().compareTo(BigDecimal.ZERO) <= 0) {
            throw new InvalidOrderException("El total debe ser positivo");
        }
        orderRepository.save(order);
        eventPublisher.publish(new OrderCreatedEvent(order));
        return order;
    }
}
```

El error no lanza excepción. La transacción sencillamente no existe. Si `orderRepository.save()` falla, no hay rollback. Si `eventPublisher.publish()` falla después del save, el pedido queda persistido en un estado inconsistente sin que nadie lo sepa. Este tipo de bug aparece en producción bajo carga, de forma intermitente, y es extremadamente difícil de rastrear.

**La misma trampa con `@Async`:**

```java
@Service
public class NotificationService {

    // ❌ El proxy tampoco intercepta @Async en llamadas internas
    @Async
    private void sendWelcomeEmail(String email) {
        emailClient.send(email, "Bienvenido");
    }

    public void onUserRegistered(UserRegisteredEvent event) {
        // Esta llamada es directa, no pasa por el proxy → se ejecuta síncronamente
        sendWelcomeEmail(event.email()); 
    }
}
```

> ⚠️ **Regla práctica:** Cualquier anotación de Spring que dependa de proxys AOP (`@Transactional`, `@Async`, `@Cacheable`, `@PreAuthorize`) solo funciona cuando el método es invocado desde **fuera** del bean. Verifica siempre el punto de entrada real de la llamada, no donde está puesta la anotación.

### 5.2. Alucinaciones Técnicas Sutiles

Las alucinaciones más peligrosas de la IA no son las que inventan una API que no existe (eso lo detecta el compilador). Son las que inventan **atributos o comportamientos en APIs que sí existen**.

#### Anotaciones con atributos fantasma

```java
// ❌ La IA genera esto. @Scheduled NO tiene un atributo "enabled"
@Scheduled(fixedRate = 60000, enabled = true)
public void processScheduledJobs() {
    // ...
}
// Error de compilación: Unknown attribute 'enabled' for annotation type Scheduled.
// Este error sí es visible. Pero no siempre el atributo inventado da error de compilación.

// ❌ Más peligroso: atributos que existen pero con semántica distinta a la asumida
@Cacheable(value = "products", unless = "#result == null", sync = true)
public Product findProduct(String id) {
    // La IA asume que sync=true evita thundering herd en Caffeine.
    // En realidad, sync=true solo funciona con proveedores de caché que lo soporten.
    // Con un ConcurrentHashMap simple, se ignora silenciosamente.
    return productRepository.findById(id).orElse(null);
}
```

#### Dependencias con versiones que no existen o son incompatibles

```xml
<!-- ❌ La IA puede sugerir starters con nombres plausibles que no existen -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-kafka-streams</artifactId>
</dependency>
<!-- El artefacto correcto es spring-kafka, que incluye Kafka Streams support -->

<!-- ❌ O versiones donde la API cambió y el código generado no compila -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-contract-verifier</artifactId>
    <version>4.0.0</version>
    <!-- La IA puede generar código de la API de la versión 3.x que cambió en 4.x -->
</dependency>
```

#### Métodos de Spring Data que no existen (o con parámetros invertidos)

```java
// ❌ La IA inventa firmas derivadas de nombre que Spring Data no puede parsear
List<Order> orders = orderRepository.findByStatusAndCreatedDateBetweenOrderByLastName(
    Status.ACTIVE, startDate, endDate
); // Error: Spring Data no puede parsear "AndOrderBy" en este contexto

// ❌ Parámetros en orden incorrecto (el error ocurre en runtime, no en compilación)
orderRepository.findByCreatedDateBetween(endDate, startDate);
// El orden correcto es (startDate, endDate). Devuelve resultados vacíos o incorrectos.

// ✅ Ante la duda, @Query explícita: el error aparece en el arranque de la aplicación
@Query("SELECT o FROM Order o WHERE o.status = :status " +
       "AND o.createdDate BETWEEN :start AND :end " +
       "ORDER BY o.customer.lastName")
List<Order> findActiveOrdersInPeriod(
    @Param("status") Status status,
    @Param("start") LocalDateTime start,
    @Param("end") LocalDateTime end
);
```

> 💡 **Regla de oro:** Si la IA genera un método derivado de nombre en Spring Data con más de tres condiciones, trata la firma como sospechosa y sustitúyela por `@Query` explícita. El coste de escribir el JPQL es mínimo; el coste de debuggear un método que devuelve datos incorrectos en producción es muy alto.

### 5.3. Erosión de la Agudeza Mental (Skill Decay)

El riesgo más silencioso del uso acrítico de la IA no es un bug en producción. Es la degradación gradual e imperceptible de las habilidades de diagnóstico del propio desarrollador.

#### La trampa del "funciona"

Cuando la IA resuelve sistemáticamente un tipo de problema (configuración de `@Async`, gestión de transacciones, problemas de concurrencia), el desarrollador deja de ejercitar ese músculo mental. En seis meses, cuando el agente no esté disponible o cuando el problema sea lo suficientemente inusual como para que la IA no lo resuelva correctamente, el desarrollador habrá perdido la capacidad de atacarlo solo.

```java
// Problema clásico de concurrencia que la IA resuelve "bien"
// pero que el dev ya no sabe diagnosticar sin ella:

@Service
public class InventoryService {
    // La IA detecta que hay una race condition aquí y sugiere @Lock o ConcurrentHashMap.
    // ¿Pero sabes explicar POR QUÉ este código falla bajo carga?
    // ¿Sabes cuándo usar @Version (optimistic) vs @Lock (pessimistic)?
    // ¿Sabes cuál es el coste en throughput de cada opción?
    
    private Map<String, Integer> stockCache = new HashMap<>(); // ← bomba de tiempo
    
    public void reserveStock(String productId, int quantity) {
        Integer current = stockCache.get(productId);
        if (current != null && current >= quantity) {
            stockCache.put(productId, current - quantity);
        }
    }
}
```

Si delegas siempre en la IA para este tipo de análisis, perderás la capacidad de explicar el problema a tu equipo, de evaluarlo en una entrevista técnica o de detectarlo cuando aparezca en una forma ligeramente diferente.

#### El efecto anclaje en el diseño

Cuando la IA propone la primera solución a un problema de diseño, esa propuesta se convierte en el punto de referencia mental del desarrollador. A partir de ahí, el trabajo es refinamiento de lo que la IA propuso, no diseño desde cero.

```
Problema: "Sistema de notificaciones para eventos de pedido"

IA propone:
  → NotificationService con inyección de múltiples providers
  → Patrón Strategy para cada canal (Email, SMS, Push)
  → Configuración via @ConditionalOnProperty

Tú refinás:
  → "Añade retry con exponential backoff"
  → "Añade template de mensaje configurable"

Lo que quizás no consideraste porque la IA no lo propuso:
  → ¿Debería ser un servicio separado con su propia BD de estado de notificaciones?
  → ¿Debería el emisor del evento ni siquiera conocer al receptor (Event Sourcing)?
  → ¿Necesitamos idempotencia en el envío?
```

> ⚠️ **Señales de que el efecto anclaje está ocurriendo:** Si tu arquitectura de los últimos tres meses tiene siempre el mismo patrón estructural (independientemente del dominio del problema), es probable que estés refinando las propuestas de la IA en lugar de diseñar.

#### Práctica de mantenimiento de agudeza

La mejor defensa contra el skill decay no es usar menos la IA, sino ser deliberado sobre cuándo no usarla:

- Dedica un tiempo semanal (una kata, un módulo pequeño) a escribir código **sin asistencia de IA**, sin buscar en Stack Overflow, con solo la documentación oficial como referencia.
- Cuando la IA resuelva un problema de concurrencia o transacciones por ti, tómate diez minutos después para escribir en papel **por qué** la solución es correcta. Si no puedes, no aceptes el código.

### 5.4. El Problema de la "Masa Crítica" de Código

Generar código es barato con la IA. Mantenerlo, entenderlo y modificarlo seis meses después sigue siendo caro, con o sin IA.

El uso acrítico lleva a un crecimiento exponencial de la base de código donde nadie tiene un modelo mental completo del sistema. Cada feature nueva se construye sobre abstracciones que nadie comprende del todo, y cada refactor potencial se convierte en una operación de riesgo alto.

```
Sin IA (ritmo de generación humano):
  Mes 1: 10 clases → el equipo las entiende todas
  Mes 3: 30 clases → el equipo entiende ~25
  Mes 6: 60 clases → hay zonas que nadie toca porque "funciona"

Con IA sin criterio (ritmo de generación acelerado):
  Mes 1: 40 clases → el equipo "entiende" 30 (las otras las generó el agente)
  Mes 3: 120 clases → hay módulos enteros que nadie puede explicar en detalle
  Mes 6: 300 clases → el proyecto es opaco. Cada bug requiere arqueología.
```

La estrategia correcta no es frenar la generación de código, sino imponer **criterios de aceptación más estrictos** al código generado:

1. **El test es la especificación, no la verificación.** Antes de que la IA genere el código, escribe el test que define el comportamiento esperado. Si no puedes escribir el test, no sabes lo que quieres.
2. **No aceptes código que no puedas explicar.** Si ves una clase generada por la IA y no puedes explicar en dos frases qué hace y por qué existe, no la integres hasta que lo entiendas.
3. **El código más corto es el que menos hay que mantener.** Si la IA genera 80 líneas y tú puedes reducirlas a 30 sin perder claridad, hazlo. Esa simplificación es evidencia de que has entendido el código.

### 💡 Tabla: Errores comunes de "Overtrust" en Spring

| Sugerencia de la IA | Riesgo Oculto | Consecuencia en producción |
| :--- | :--- | :--- |
| `@Transactional` en método privado | El proxy AOP no intercepta métodos privados ni llamadas internas | La transacción no existe: no hay rollback en caso de error |
| Query JPA con múltiples `JOIN FETCH` | Producto cartesiano en memoria (N×M filas) | `OutOfMemoryError` con volúmenes reales de datos |
| Inyección de dependencias por campo (`@Autowired`) | Oculta las dependencias reales de la clase | Tests unitarios requieren Spring Context o Reflection |
| `Optional.get()` sin chequeo | Asume que el dato siempre existe | `NoSuchElementException` silenciosa en producción |
| `@Async` en método privado | El proxy no intercepta: se ejecuta síncronamente | El endpoint bloquea hasta que termina la tarea "asíncrona" |
| `HashMap` en bean `@Service` (Singleton) | No es thread-safe bajo carga concurrente | Corrupción de datos intermitente y difícil de reproducir |
| `@CrossOrigin(origins = "*")` | Deshabilita protección CORS por completo | Superficie de ataque CSRF ampliada |

### 🧪 Ejercicio: "Encuentra la alucinación" (15 min)

El siguiente código fue generado por una IA para un caso de uso de procesamiento reactivo con Spring WebFlux. A primera vista parece correcto. Encuentra el error.

```java
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    private final OrderService orderService;

    @GetMapping("/{orderId}/status")
    public Mono<OrderStatusResponse> getOrderStatus(@PathVariable String orderId) {
        return Mono.fromCallable(() -> {
            // La IA genera esto porque "funciona en los tests"
            Order order = orderService.findById(orderId).block(); // ← ¿lo ves?
            return OrderStatusResponse.from(order);
        }).subscribeOn(Schedulers.boundedElastic());
    }
}
```

**El problema:** `.block()` dentro de un operador reactivo bloquea el hilo del scheduler de Reactor. En producción, bajo carga sostenida, esto agota el pool de threads de `boundedElastic()` y el servicio deja de responder. En un test con una sola petición, funciona perfectamente.

**La corrección:**

```java
@GetMapping("/{orderId}/status")
public Mono<OrderStatusResponse> getOrderStatus(@PathVariable String orderId) {
    return orderService.findById(orderId)  // ← findById debe devolver Mono<Order>
        .map(OrderStatusResponse::from);
}
```

**Objetivo del ejercicio:** Identificar que "funcionar en local" y "funcionar en tests" no garantiza corrección bajo carga real, especialmente cuando la IA genera código que mezcla estilos imperativos y reactivos.

## 6. Cambios en el rol del desarrollador (más diseño, validación, revisión y automatización)

El rol del Senior está sufriendo una metamorfosis: de ser un "constructor de componentes" a ser un "director de sistemas". La habilidad de escribir sintaxis Java está perdiendo valor frente a la habilidad de orquestar flujos de trabajo y validar la integridad de sistemas complejos.

Este cambio no es gradual ni opcional. Es estructural. La IA ya genera DTOs, mappers, controllers y tests unitarios simples con una calidad suficiente para producción. El Senior que siga compitiendo en esa capa está compitiendo en un área donde su ventaja diferencial desaparece rápidamente.

### 6.1. El Desarrollador como "Editor Jefe" (Reviewer-in-Chief)

El trabajo de un editor jefe de una publicación no es escribir los artículos: es definir la línea editorial, revisar que cada pieza cumple los estándares, detectar los errores que el autor no ve y devolver lo que no llega al nivel. Ese es exactamente el nuevo rol del Senior en la era agéntica.

**De la creación a la curación:** La mayor parte del tiempo ya no se gasta frente a una hoja en blanco. Se gasta revisando propuestas del agente, evaluando su corrección arquitectónica y detectando los errores que el agente no puede ver porque no tiene acceso al contexto del negocio ni al historial de decisiones del equipo.

**Leer código es ahora más importante que escribirlo.** Y leer código generado por IA requiere un nivel de atención diferente al de leer código de un compañero. El código de un compañero tiene errores humanos predecibles: olvidos, shortcuts, malentendidos de requisitos. El código de la IA tiene errores de una categoría diferente: correcto en la superficie, incorrecto en las suposiciones implícitas.

```java
// Este código generado por la IA "parece perfecto" en la revisión rápida.
// El editor jefe debe detectar que hay dos suposiciones implícitas problemáticas:

@Service
public class ReportService {

    public ReportSummary generateMonthlyReport(int year, int month) {
        List<Order> orders = orderRepository.findByYearAndMonth(year, month);
        // Suposición 1: findByYearAndMonth devuelve una lista finita.
        // Con un año de datos históricos, puede ser un millón de registros en memoria.

        BigDecimal total = orders.stream()
            .map(Order::getTotal)
            .reduce(BigDecimal.ZERO, BigDecimal::add);

        Map<String, Long> byCategory = orders.stream()
            .collect(Collectors.groupingBy(
                order -> order.getCategory().getName(),
                // Suposición 2: getCategory() nunca devuelve null.
                // Si un pedido migrado de un sistema legacy no tiene categoría, NullPointerException.
                Collectors.counting()
            ));

        return new ReportSummary(total, byCategory);
    }
}
```

El editor jefe no solo ve que compila. Ve que hay una query sin paginación en un contexto de datos históricos, y que hay una dependencia de datos limpios que los datos reales de producción pueden no cumplir.

**El protocolo de revisión del editor jefe:**

Antes de aprobar cualquier bloque de código generado por la IA, hazte estas tres preguntas:
1. **¿Qué asume este código sobre los datos?** (nunca nulo, siempre paginado, siempre transaccional...)
2. **¿Qué ocurre cuando esto falla a las 3 de la madrugada con el doble de carga?**
3. **¿Si este código introduce un bug en producción, podré encontrarlo leyendo los logs?**

Si no puedes responder estas tres preguntas, no apruebes el código.

### 6.2. Énfasis en el Diseño de Sistemas y Contratos

En la era del agente, el cuello de botella del desarrollo ha dejado de ser la velocidad de tecleo y se ha convertido en la **calidad de la especificación**. La IA puede implementar cualquier especificación clara. No puede inferir lo que no está especificado.

**La habilidad principal del Senior pasa a ser el diseño de contratos:**

| Artefacto | Qué especifica | Por qué es el input del agente |
| :--- | :--- | :--- |
| **OpenAPI 3.1** | Endpoints, schemas, códigos HTTP, validaciones | El agente implementa el controller y los tests de contrato |
| **AsyncAPI 2.x/3.x** | Eventos, canales, payloads de Kafka/RabbitMQ | El agente implementa producers, consumers y handlers |
| **Esquema de BD (Flyway)** | Tablas, tipos exactos, constraints, índices | El agente genera entidades JPA con tipos correctos |
| **ADR (Architecture Decision Record)** | Decisiones de diseño y sus justificaciones | El agente implementa respetando las restricciones documentadas |

Un endpoint definido con precisión en OpenAPI produce una implementación correcta. Un endpoint descrito vagamente en lenguaje natural produce una implementación que requiere múltiples iteraciones de corrección.

**Antes (el Senior como implementador):**
```
Requisito: "Necesito un endpoint para crear pedidos"
Senior: escribe UserController, CreateOrderRequest, servicio, repositorio...
Resultado: 2-4 horas de trabajo, decisiones tomadas on-the-fly
```

**Ahora (el Senior como arquitecto de contratos):**
```yaml
# orders-api.yaml — La especificación que el agente implementa
paths:
  /orders:
    post:
      summary: Crear un nuevo pedido
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/CreateOrderRequest'
      responses:
        '201':
          description: Pedido creado exitosamente
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/OrderResponse'
        '422':
          description: Datos de entrada inválidos
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/ValidationErrorResponse'
        '409':
          description: Conflicto (cliente con deuda pendiente)
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/ErrorResponse'
```

Esta especificación ya toma decisiones que antes se tomaban durante la implementación: qué código HTTP para cada caso de error, qué schema exacto para las respuestas de error, si la validación es 400 o 422. El agente implementa esto sin ambigüedad. Y si la implementación no cumple el contrato, los tests de contrato fallan automáticamente.

**Prompt al Agente:**
> "Implementa el endpoint `POST /orders` siguiendo exactamente el contrato definido en `src/main/resources/openapi/orders-api.yaml`. Usa arquitectura hexagonal: el puerto de entrada en `domain/port/in/CreateOrderUseCase.java`, el adaptador REST en `infrastructure/rest/OrderController.java`. Genera también el test de contrato con MockMvc que verifique los tres casos de respuesta definidos en el OpenAPI. Ejecuta los tests y corrígelos si fallan."

### 6.3. El Ingeniero de Automatización Integral

El Senior de la era agéntica no solo usa herramientas de IA: **diseña los flujos donde la IA trabaja**. Esto incluye configurar qué hace el agente, cómo recibe contexto, qué puede tocar y qué no, y cómo se integra con el resto del ciclo de desarrollo.

**CLAUDE.md como política de equipo:** El fichero `CLAUDE.md` en la raíz del repositorio es el "briefing" que el agente lee automáticamente. Un Senior que sabe configurarlo bien hace que cada sesión del agente arranque ya orientada al contexto del proyecto:

```markdown
# CLAUDE.md — Proyecto Orders Service

## Stack
- Java 21, Spring Boot 3.4, PostgreSQL 16
- Arquitectura hexagonal estricta

## Reglas de código
- Inyección SOLO por constructor, nunca @Autowired en campos
- DTOs como Java Records (no Lombok @Data)
- El paquete domain/ NO puede importar nada de infrastructure/

## Política de modelos
- Tier balanceado (Sonnet) para implementación multi-módulo y code review
- Tier alto (Opus) para diseño de nuevos bounded contexts o análisis de deuda técnica
- Tier rápido (Haiku) para generación de DTOs, mappers y tests unitarios simples

## Qué NO tocar
- src/main/resources/db/migration/ → las migraciones de Flyway las revisa siempre un humano
- SecurityConfig.java → los cambios de seguridad requieren aprobación explícita
```

**Hooks para automatizar el ciclo de validación:** Los hooks permiten ejecutar acciones automáticamente antes o después de que el agente realice ciertas operaciones. Por ejemplo, ejecutar `mvn verify` después de cada modificación de fichero Java, o ejecutar un linter de seguridad antes de permitir que el agente edite configuraciones de Spring Security.

**Prompt Engineering Técnico:** No se trata de "pedir cosas", sino de inyectar el contexto que el agente necesita para no tomar decisiones por defecto que no se alinean con el proyecto. Un prompt técnico preciso tiene:

```
Contexto del proyecto (stack, versiones, restricciones)
    ↓
Objetivo concreto (qué ficheros crear o modificar)
    ↓
Restricciones explícitas (qué NO puede hacer)
    ↓
Criterio de éxito verificable (cuándo está "terminado")
```

```
# Ejemplo de prompt técnico bien estructurado:

"Añade el campo `discountCode` (String, nullable, máximo 50 caracteres) 
a la entidad Order.

Stack: Java 21, Spring Boot 3.4, JPA con Flyway para migraciones.

Actualiza en cascada:
- La entidad JPA OrderEntity con la nueva columna
- El DTO CreateOrderRequest (Record, añade el campo como Optional<String>)
- El DTO OrderResponse (Record, incluye el campo)
- El mapper OrderMapper respetando el patrón existente
- La migración Flyway: src/main/resources/db/migration/V[N]__add_discount_code_to_orders.sql

Restricciones:
- No toques la lógica de cálculo de descuentos en OrderService (ese campo es solo trazabilidad)
- No modifiques los tests existentes para que pasen: si fallan, el campo está mal implementado
- El campo es nullable en BD y en el DTO

Criterio de éxito: mvn test pasa sin errores."
```

### 6.4. Mentoring y Liderazgo Técnico en la Era de la IA

El Senior no solo usa la IA de forma más sofisticada que un Junior: tiene la responsabilidad de definir **cómo el equipo entero usa la IA**, qué guardrails existen y cómo se forman los perfiles menos experimentados.

**Formar a Juniors en el uso responsable:** El mayor riesgo de los desarrolladores junior con acceso a agentes de IA no es que generen código malo (eso es detectable). Es que generen código que ellos mismos no comprenden y que nadie revisa porque "los tests pasan". El Senior debe establecer una cultura donde el criterio de aceptación no sea "compila y los tests pasan", sino "puedes explicar cada línea y su alternativa".

**Definir la política de delegación del equipo:** Qué se delega al agente, con qué tier de modelo, y qué siempre decide un humano. Esta política no debe existir en la cabeza del Senior: debe estar documentada en el `CLAUDE.md` del repositorio, versionada con el código.

**El Senior como última línea de defensa arquitectónica:** En un equipo donde todos usan agentes, la velocidad de generación de código sube drásticamente. También sube la velocidad a la que se introducen decisiones arquitectónicas incorrectas que parecen correctas. El Senior es quien tiene el contexto histórico del sistema, quien conoce los fallos de producción del pasado, quien ha vivido las consecuencias de las malas decisiones de tres años atrás. Ese conocimiento no puede ser delegado a ningún agente.

## 7. Uso responsable: propiedad intelectual, privacidad y datos sensibles

El uso de IA en entornos corporativos no es solo una cuestión técnica, sino de cumplimiento legal y protección de activos. Cuando el código de producción pasa por los servidores del proveedor de la IA, la información que contiene sale de la infraestructura controlada de la empresa. Un Senior debe entender exactamente qué sale, dónde va y qué consecuencias tiene.

---

### 7.1. Privacidad y Protección de Datos (GDPR/RGPD)

El problema no es solo enviar datos explícitamente. El problema es que el contexto que el agente recopila automáticamente puede contener información sensible que el desarrollador ni siquiera recuerda que estaba ahí.

**Qué puede contener un archivo de contexto sin que el dev lo note:**

```java
// Un @Entity que parece inocuo...
@Entity
@Table(name = "users")
public class User {
    private String email;           // PII — dato personal identificable
    private String fullName;        // PII
    private String nationalId;      // PII — DNI/NIF — altamente sensible
    private String creditCardLast4; // PII financiero
    private String ipAddress;       // PII según GDPR
    private LocalDate birthDate;    // PII
}
```

Si el agente lee este fichero para generar tests, los datos de ejemplo que sugiera pueden incluir datos reales que están en los comentarios, en los seeds de BD que también ha leído, o en los logs de error que has pegado en el chat.

**Sanitización de logs antes de pasarlos a la IA:**

```
# ❌ Log real que NO debes pegar en el chat:
ERROR 2026-03-15 14:23:01 [http-nio-8080-exec-3] OrderService - 
Error processing order for user juan.garcia@empresa.com (ID: 847293)
Card: **** **** **** 4521, Amount: 1247.50 EUR
Connection URL: jdbc:postgresql://prod-db.internal:5432/orders?user=app_user&password=s3cur3P4ss!

# ✅ Log sanitizado para compartir con la IA:
ERROR [OrderService] - Error processing order for user user@example.com (ID: 000000)
Card: **** **** **** 0000, Amount: XXX.XX EUR
Connection URL: jdbc:postgresql://localhost:5432/orders (credenciales omitidas)
```

**Regla práctica:** Antes de pegar cualquier log, traza de error o fragmento de código en el chat de la IA, busca manualmente: emails, nombres propios, IPs, tokens, passwords, UUIDs que puedan ser IDs reales de clientes, y cualquier URL que contenga parámetros de conexión. Sustitúyelos por placeholders antes de enviar.

---

### 7.2. Propiedad Intelectual y Licenciamiento

**El riesgo del código "envenenado":** Los modelos de IA se entrenan con código público que incluye código bajo licencias GPL, AGPL y otras licencias copyleft. Cuando el modelo sugiere un fragmento que ha memorizado de ese corpus, puede estar sugiriendo código que técnicamente tiene restricciones de licencia incompatibles con un proyecto comercial cerrado.

En la práctica, el riesgo principal no es que la IA copie bloques completos con copyright (los proveedores tienen filtros para esto), sino que sugiera implementaciones algorítmicas cuya única fuente conocida es código bajo licencia restrictiva.

**Mitigación:** Para algoritmos críticos o lógica de negocio propietaria, verifica siempre que la implementación sugerida tiene referencias en documentación oficial (RFC, spec de la librería, documentación de Spring) antes de integrarla. Si la única explicación que encuentras para "por qué funciona así" es el código de otra librería GPL, no uses esa implementación.

**Copyright del código generado:** En la mayoría de jurisdicciones (incluida la UE), el código generado íntegramente por IA sin intervención creativa humana no es protegible por copyright. La autoría legalmente reconocida recae en el diseño, la arquitectura y las decisiones que el desarrollador tomó. Esto tiene implicaciones prácticas: si tu empresa vende software y una parte relevante fue generada por IA sin revisión humana significativa, la protección intelectual de esa parte puede ser cuestionable.

**Políticas de retención por plan:** La diferencia más crítica entre los planes de uso de herramientas de IA es qué ocurre con el contenido de los prompts:

| Plan | Uso para entrenamiento | Retención de prompts | Controles |

### 7.4. Responsabilidad Técnica: El Desarrollador como Último Responsable

Si la IA sugiere una librería con una vulnerabilidad conocida (CVE publicado en la NVD), la responsabilidad legal y profesional del fallo en producción recae en el desarrollador que aceptó el `pom.xml`, no en el proveedor de la IA.

Este principio se extiende a todo el código generado:

```xml
<!-- ❌ La IA sugiere esta dependencia. Parece razonable. -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>some-utility-library</artifactId>
    <version>1.2.3</version>
</dependency>
<!-- Si 1.2.3 tiene un CVE de RCE conocido y tú lo aceptas sin verificar,
     la responsabilidad es tuya. La IA no consulta la NVD antes de sugerir versiones. -->
```

**Práctica obligatoria:** Integra un scanner de vulnerabilidades en el pipeline de CI que revise las dependencias introducidas. Para Maven, `org.owasp:dependency-check-maven` es el estándar. Configúralo para que falle el build ante CVEs con severidad CRITICAL o HIGH sin excepción documentada.

```xml
<!-- pom.xml — plugin de auditoría de dependencias -->
<plugin>
    <groupId>org.owasp</groupId>
    <artifactId>dependency-check-maven</artifactId>
    <configuration>
        <failBuildOnCVSS>7</failBuildOnCVSS> <!-- Falla con severidad >= HIGH -->
        <suppressionFile>dependency-check-suppressions.xml</suppressionFile>
    </configuration>
</plugin>
```

Las excepciones (falsos positivos o CVEs con mitigación documentada) deben estar en el fichero de supresiones con justificación escrita. Eso crea un rastro auditable de decisiones de riesgo tomadas conscientemente.

## 8. Expectativas realistas: dónde acelera mucho y dónde no

Para un Senior de Spring, la IA es un multiplicador de fuerza. Un multiplicador por cero sigue siendo cero: si no hay criterio técnico sólido sobre qué pedir y cómo validar la respuesta, la IA solo acelera la generación de problemas.

El error de expectativas más frecuente no es pensar que la IA lo hace todo mal. Es pensar que lo hace todo bien en todos los contextos. La diferencia entre dónde la IA acelera genuinamente y dónde introduce más fricción de la que elimina es una de las habilidades prácticas más importantes de este curso.

---

### 8.1. Zonas de Alta Aceleración

Estas son las tareas donde el uso de la IA tiene un ROI claro y consistente. El patrón común: tareas con una estructura predecible, un resultado verificable y bajo coste de error.

**Generación de boilerplate y configuración inicial:**

```bash
# En lugar de buscar en la documentación de Spring Initializr y copiar configuración:
> "Crea un proyecto Spring Boot 3.4 con Java 21 para un microservicio de gestión
  de pedidos. Incluye: spring-data-jpa, postgresql, spring-web, validation,
  spring-kafka y testcontainers. Estructura de paquetes en arquitectura hexagonal.
  Genera también el docker-compose.yml con PostgreSQL 16 y Kafka."
```

Lo que antes requería 20-30 minutos de configuración manual se convierte en un punto de partida funcional en 2 minutos. El Senior revisa y ajusta; no construye desde cero.

**Mappers, DTOs y conversiones de formato:**

```java
// Dado este dominio:
public record Order(OrderId id, CustomerId customerId, List<OrderItem> items,
                    Money total, OrderStatus status, LocalDateTime createdAt) {}

// La IA genera correctamente el mapper y el DTO de respuesta en segundos:
public record OrderResponse(String id, String customerId, List<OrderItemResponse> items,
                            BigDecimal total, String status, LocalDateTime createdAt) {
    public static OrderResponse from(Order order) {
        return new OrderResponse(
            order.id().value(),
            order.customerId().value(),
            order.items().stream().map(OrderItemResponse::from).toList(),
            order.total().amount(),
            order.status().name(),
            order.createdAt()
        );
    }
}
```

**Tests unitarios con patrón claro (AAA):**

```java
// La IA genera tests unitarios con Mockito de forma fiable cuando el patrón es predecible:
@Test
void createOrder_shouldThrowException_whenCustomerHasPendingDebt() {
    // Arrange
    String customerId = "customer-123";
    when(customerRepository.findById(customerId))
        .thenReturn(Optional.of(Customer.withDebt(customerId)));

    // Act & Assert
    assertThrows(CustomerWithPendingDebtException.class,
        () -> orderService.createOrder(new CreateOrderCommand(customerId, List.of())));
    verify(orderRepository, never()).save(any());
}
```

**Configuración de infraestructura de test con Testcontainers:**

```java
// Setup de Testcontainers que la IA genera correctamente y que manualmente requiere
// leer documentación de Testcontainers + Spring Boot integration:
@Testcontainers
@SpringBootTest
class OrderRepositoryIntegrationTest {

    @Container

        .withDatabaseName("orders_test")

    @DynamicPropertySource

        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
    }
}
```

**Documentación y explicación de código legacy:**

```
> "Lee la clase LegacyBillingCalculator.java y explícame:
   1. Qué hace cada método público
   2. Qué dependencias externas tiene
   3. Qué casos de negocio cubre según los nombres de los métodos
   4. Qué partes parecen más frágiles o candidatas a refactoring"
```

Este caso de uso tiene un ROI especialmente alto con código legacy donde la documentación no existe o está desactualizada.

### 8.2. Zonas de Baja Aceleración

Estas son las tareas donde la IA proporciona un punto de partida pero el trabajo real sigue siendo del Senior. El patrón común: requieren contexto sistémico que la IA no tiene, o la consecuencia de un error es lo suficientemente alta como para que la validación sea más costosa que la implementación.

**Debugging de problemas de concurrencia y condiciones de carrera:**

La IA puede explicar qué es una race condition y proponer soluciones genéricas. Lo que no puede hacer es analizar el comportamiento específico de tu sistema bajo la carga concreta de tu producción, con tus patrones de acceso y tu configuración de pool de conexiones.

```java
// La IA propone la "solución fácil" ante una LazyInitializationException:
// ❌ Opción 1 que la IA sugiere frecuentemente (incorrecta):
spring.jpa.properties.hibernate.enable_lazy_load_no_trans=true
// Resultado: queries N+1 silenciosas que destrozan el rendimiento en producción.

// ❌ Opción 2 que la IA sugiere frecuentemente (incorrecta en este contexto):
// Cambiar todas las relaciones a FetchType.EAGER
// Resultado: se cargan todas las relaciones siempre, aunque no se necesiten.

// ✅ La solución correcta (que requiere análisis del caso concreto):
// Usar Entity Graphs o Projections específicos según el caso de uso:
@EntityGraph(attributePaths = {"items", "items.product"})
@Query("SELECT o FROM Order o WHERE o.customerId = :customerId")
List<Order> findOrdersWithItemsByCustomerId(@Param("customerId") String customerId);
```

La IA te dará las tres opciones si le preguntas con el contexto correcto. Pero elegir cuál aplica en tu sistema, con tu volumen de datos y tus patrones de acceso, sigue siendo una decisión del Senior.

**Tuning de rendimiento específico:**

```yaml
# La IA genera configuraciones de HikariCP "estándar". Los valores correctos
# para tu sistema requieren medir, no asumir:
spring:
  datasource:
    hikari:
      maximum-pool-size: 10      # ← ¿Por qué 10? ¿Has medido tus conexiones activas?
      minimum-idle: 5            # ← ¿Cuál es tu patrón de carga en horas valle?
      connection-timeout: 30000  # ← ¿Cuánto tarda tu BD en responder en P99?
      idle-timeout: 600000       # ← ¿Cuánto tiempo mantiene conexiones inactivas tu proxy?
```

Estos valores no los puede conocer la IA porque dependen de la telemetría de tu sistema específico. La IA puede explicar qué hace cada parámetro; tú decides el valor.

**Decisiones de arquitectura evolutiva:**

La decisión de cuándo extraer un bounded context de un monolito a un microservicio, o cuándo dos microservicios deben fusionarse, depende de factores que la IA no puede ver: la estructura del equipo (Ley de Conway), el historial de fallos de producción, los SLAs contractuales con clientes, la deuda técnica acumulada y la madurez operativa del equipo. La IA puede ayudar a estructurar el análisis; la decisión final siempre es humana.

### 8.3. La Barrera del Contexto y el Conocimiento Tácito

Aunque las ventanas de contexto de los modelos actuales son grandes (200k tokens en Opus), hay dos tipos de contexto que la IA estructuralmente no puede tener:

**Contexto histórico del sistema:** Por qué se tomó la decisión de usar JDBC plano en `BillingRepository` en lugar de JPA (hubo un bug de rendimiento grave hace dos años con una versión específica de Hibernate que ya está corregida pero nadie lo sabe). Por qué `OrderService` tiene ese método de 200 líneas que nadie toca (un incidente de producción en 2023 cuyo fix está en esas líneas y no está documentado en el código). La IA no tiene acceso a ese conocimiento tácito que vive en las conversaciones del equipo y en la memoria de los desarrolladores más veteranos.

**Contexto de negocio no codificado:** Las reglas de negocio que el equipo conoce de forma oral, que están en emails de hace tres años o en conversaciones con el área de negocio. La IA implementa lo que hay en el código y lo que tú le describes explícitamente. Lo que no está escrito en ningún sitio, no existe para el agente.

```
# Ejemplo real de consecuencia:
Prompt: "Refactoriza DiscountService para simplificar la lógica de cálculo"

Lo que la IA no sabe: ese if() de 40 líneas en calculateLoyaltyDiscount()
existe porque hay un contrato firmado con un cliente enterprise que estipula
condiciones de descuento específicas que están fuera de la lógica estándar.
El contrato está en un PDF en Confluence que la IA no ha leído.

Resultado: la IA simplifica el código, los tests pasan, y el cliente enterprise
recibe facturas incorrectas durante tres semanas hasta que alguien lo detecta.
```

La defensa es documentar el conocimiento tácito en el código mismo (`// IMPORTANT: this branch handles the Acme Corp enterprise contract - see Confluence/contracts/acme-2023.pdf`) o en el `CLAUDE.md` del repositorio.

### 📂 Esqueleto del Contenido

#### 9.1. Metodología: "Hands-on with Guardrails"
* **Enfoque Práctico:** El curso se basa en proyectos de Spring Boot reales (microservicios, persistencia, seguridad).
* **El "Guardrail" (Barandilla):** Cada tarea generada con IA debe pasar por un proceso de revisión obligatoria basado en un checklist de calidad (Clean Code, Performance, Seguridad).

#### 9.2. Progresión del Aprendizaje
1. **Nivel 1 (Asistido):** Uso de Copilot para eliminar tareas mecánicas y boilerplate.
2. **Nivel 2 (Estratégico):** Uso de la IA para refactorización masiva y modernización de código legacy.
3. **Nivel 3 (Agéntico/SDD):** Definición de especificaciones técnicas (OpenAPI/Specs) y supervisión de la implementación completa por parte de la IA.

#### 9.3. Criterios de Éxito (KPIs del Alumno)
* **Autonomía Crítica:** El alumno identifica y corrige al menos el 90% de las alucinaciones o malas prácticas propuestas por la IA.
* **Reducción del Lead Time:** Capacidad de implementar una feature completa (desde el contrato hasta el test de integración) en un 40-60% menos de tiempo que el flujo tradicional.
* **Calidad del Código:** El código final debe cumplir con los estándares de producción (cobertura de tests >80%, sin vulnerabilidades críticas, y arquitectura desacoplada).

#### 🧪 Ejercicio Final: "The Senior Challenge"
> **Reto:** Se entrega un requisito de negocio complejo (ej: "Implementar un sistema de cuotas para una API con Redis y Spring Cloud Gateway").
> **Condición:** El alumno debe documentar el proceso:
> 1. Qué prompt/especificación inicial dio.
> 2. Qué errores detectó en la respuesta de la IA.
> 3. Cómo guió a la IA para llegar a la solución óptima.
     > **Evaluación:** No se evalúa el código final, sino el **diario de decisiones** y las correcciones técnicas realizadas.
