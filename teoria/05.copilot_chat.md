# Tema 5. COPILOT CHAT EN EL IDE: PROMPTING PROFESIONAL Y FLUJO CONVERSACIONAL

El Chat en IntelliJ no es un buscador; es un motor de ejecución de lógica. Para un Senior, la clave está en cómo estructurar el diálogo para que la IA actúe como un revisor crítico o un arquitecto asistente.
Un prompt profesional no es una pregunta, es una **orden técnica estructurada** con objetivos, restricciones y criterios de aceptación.

---

## 1. Chat vs Completions: cuándo usar cada uno

No todo se resuelve en el editor ni todo se resuelve en el chat. El Senior elige según la economía de esfuerzo:

| Modalidad | Ideal Para | Ejemplo Spring |
|---|---|---|
| **Completions (En línea)** | Implementación local de métodos, boilerplate inmediato | Escribir un mapper, completar un DTO |
| **Chat (Panel lateral)** | Refactorizaciones multi-archivo, diseño, explicaciones | Rediseñar la capa de seguridad |
| **Inline Chat** (`Ctrl+I` / `Cmd+I`) | Modificaciones rápidas sobre bloques existentes | Cambiar un método para usar Records |
| **Agente** | Cambios multi-archivo con ejecución | Migrar de RestTemplate a WebClient en todo el proyecto |

---

## 2. Técnicas de prompting para devs: objetivos, restricciones y criterios de aceptación

Un prompt profesional no es una pregunta, es una **orden técnica estructurada**.

* **Definición de Objetivos:** Qué queremos lograr (ej: "Sustituir la persistencia de JDBC a Spring Data JPA").
* **Inyección de Restricciones:** Lo que NO puede hacer (ej: "No uses Lombok", "No cambies la firma del método público", "Mantén la compatibilidad con Java 17").
* **Criterios de Aceptación:** Cómo validaremos el éxito (ej: "Genera el código y un test unitario que cubra el caso de base de datos vacía").

### 1. Objetivo (QUÉ quiero)
```
Sustituir la persistencia de JDBC plano a Spring Data JPA 
en el servicio de usuarios.
```

### 2. Restricciones (QUÉ NO puede hacer)
```
- No uses Lombok
- No cambies las firmas de los métodos públicos
- Mantén compatibilidad con Java 17
- Sigue la arquitectura hexagonal existente
```

### 3. Criterios de Aceptación (CÓMO validamos el éxito)
```
- Genera el código y un test de integración con Testcontainers
- El test debe cubrir: caso exitoso, caso de base de datos vacía, 
  caso de entidad duplicada
- Los tests existentes deben seguir pasando
```

### Prompt Completo de Ejemplo

```
Refactoriza OrderService para que use Spring Data JPA en lugar de JDBC directo.

Restricciones:
- No uses Lombok, usa Java Records para los DTOs
- No modifiques la firma de createOrder(CreateOrderCommand)
- Mantén la arquitectura hexagonal actual (dominio separado de infraestructura)
- Usa inyección por constructor

Criterios de aceptación:
- Crea un test de integración con Testcontainers (PostgreSQL)
- El test debe cubrir: creación exitosa, pedido duplicado (error), 
  y búsqueda de pedido inexistente
- Ejecuta los tests y confirma que pasan
```

## 3. Pedir alternativas comparadas (trade-offs), no “una solución”

El Senior no acepta la primera idea; evalúa opciones.

* **Prompting comparativo:** "Dame tres formas de implementar este sistema de caché en Spring Boot y compáralas en términos de latencia y uso de memoria".
* **Análisis de Trade-offs:** Forzar a la IA a explicar por qué propone Redis frente a un `ConcurrentHashMap` local en un escenario específico.
* **Evaluación de patrones:** Comparar una implementación con Eventos de Spring vs. una llamada síncrona directa.

### Prompting Comparativo

```
Dame tres formas de implementar caché para el servicio de catálogo 
de productos en Spring Boot y compáralas:

1. Spring @Cacheable con ConcurrentHashMap
2. Spring @Cacheable con Redis
3. Caché manual con Caffeine

Para cada una, evalúa:
- Latencia esperada
- Uso de memoria
- Complejidad de configuración
- Comportamiento en un cluster de 3 nodos
- Invalidación de caché

Presenta la comparación en una tabla.
```

### Análisis de Patrones

```
Compara implementar la comunicación entre OrderService y PaymentService usando:
1. Llamada REST síncrona con Feign
2. Eventos de dominio con Spring ApplicationEventPublisher
3. Mensajería asíncrona con Kafka

¿Cuál recomendarías para un sistema con 1000 pedidos/minuto 
y qué efecto tiene cada opción en la consistencia de datos?
```

## 4. Pedir “preguntas antes de codificar” para reducir ambigüedad

Evitar que la IA asuma cosas que no conoce de tu negocio.

* **El patrón de "Pregunta primero":** Finalizar los prompts con "Antes de generar código, hazme cualquier pregunta necesaria para entender mejor las restricciones de mi infraestructura".
* **Clarificación de requisitos:** Cómo la IA puede ayudarte a detectar vacíos en tus propios requisitos de backend (ej: "¿Qué debe pasar si la cola de mensajes está llena?").

### El Patrón "Pregunta Primero"

```
Necesito implementar un sistema de notificaciones para mi aplicación 
Spring Boot de comercio electrónico.

ANTES de generar código, hazme las preguntas necesarias para entender:
- La infraestructura actual
- Los requisitos de entrega (en tiempo real o eventual)
- Las restricciones de rendimiento
- Los canales de notificación necesarios
```

**¿Por qué funciona?** La IA puede ayudarte a detectar **vacíos en tus propios requisitos**:

> "¿Qué debe pasar si la cola de mensajes está llena?"  
> "¿Las notificaciones necesitan persistencia o pueden perderse?"  
> "¿Hay requisitos de internacionalización para los mensajes?"

Preguntas que quizá tú no te habías planteado.

## 5. Pedir plan de cambio incremental (Safe Refactor)

Evitar el "Big Bang Refactoring" que rompe el proyecto.

* **Planificación por pasos:** Pedir a la IA que desglose una refactorización compleja en 4 o 5 pasos seguros y verificables.
* **Estrategia de coexistencia:** Cómo mantener el código antiguo y el nuevo funcionando en paralelo durante la migración (ej: `@Deprecated` y nuevos adaptadores).
* **Validación de impacto:** Preguntar a la IA: "¿Qué otros componentes de este paquete se verán afectados si cambio este Bean?".

### Prompt de Planificación

```
Necesito migrar el servicio de facturación de una clase monolítica 
de 800 líneas a arquitectura hexagonal.

NO ejecutes cambios todavía. Dame un plan en 5-6 pasos incrementales 
donde cada paso:
1. Sea un cambio autocontenido
2. No rompa la compilación
3. Mantenga los tests existentes en verde
4. Pueda ser revertido con un solo git revert

Para cada paso, indica qué archivos se crean/modifican y por qué.
```

### Validación de Impacto

```
Si cambio la interfaz InvoicePort para añadir un método 
calculateTaxes(InvoiceId), ¿qué otros componentes de este proyecto 
se verán afectados? 

Analiza los imports y las dependencias para darme una lista 
de archivos que necesitarán cambios.
```

## 6. Pedir validación: edge cases, concurrencia, transacciones y consistencia

Usar el Chat como un "Linter" lógico avanzado.

* **Análisis de Edge Cases:** "Busca posibles `NullPointerException` o estados inconsistentes en este servicio".
* **Auditoría de Transacciones:** Preguntar si `@Transactional` es suficiente en un flujo que involucra llamadas a APIs externas y bases de datos.
* **Revisión de Concurrencia:** Validar si el uso de un `HashMap` en un Bean `@Service` (Singleton) causará problemas de hilos bajo carga.

### Análisis de Edge Cases

```
Revisa el siguiente servicio y busca:
1. Posibles NullPointerException o estados inconsistentes
2. Problemas de concurrencia si dos hilos ejecutan esto simultáneamente
3. ¿Es @Transactional suficiente aquí considerando que hay una 
   llamada HTTP externa dentro de la transacción?
4. ¿Qué pasa si el evento Kafka no se puede publicar?
```

### Auditoría de Transacciones

```
Tengo un servicio con @Transactional que:
1. Lee un pedido de PostgreSQL
2. Llama a un servicio externo de pagos via REST
3. Actualiza el estado del pedido en PostgreSQL
4. Publica un evento en Kafka

¿Qué pasa si el paso 3 tiene éxito pero el paso 4 falla?
¿El @Transactional protege todo esto? Propón una solución robusta.
```

## 7. Convertir conversaciones en tareas accionables en el repo

Cerrar el ciclo: de la charla al código productivo.

* **Generación de TODOs:** Pedir a la IA que resuma la conversación en comentarios `// TODO` o tareas para el backlog.
* **Documentación de decisiones:** Usar el chat para generar el texto de la Pull Request explicando los cambios realizados y por qué se eligió esa solución.
* **Creación de tickets:** Cómo formatear la salida del chat para pegarla directamente en Jira o GitHub Issues.

### Generación de TODOs en el Código

```
Hemos discutido varios problemas en OrderService:
1. El método createOrder() mezcla lógica de negocio con persistencia
2. Falta validación del stock antes de confirmar el pedido
3. El cálculo de descuentos debería delegarse a una clase CouponStrategy

No hagas ningún cambio. Añade comentarios // TODO en el código
marcando exactamente dónde hay que intervenir en cada caso.
Incluye en cada TODO: qué hay que hacer, por qué, y el riesgo
si no se hace.
```

Resultado esperado en el código:

```java
public Order createOrder(CreateOrderCommand command) {
    // TODO [REFACTOR - MEDIA]: Extraer validación de negocio a un caso de uso
    // separado (CreateOrderUseCase). El servicio no debería conocer los detalles
    // de persistencia. Riesgo: la lógica de negocio crecerá aquí haciéndola
    // imposible de testear sin BD.
    validateCustomer(command.customerId());

    // TODO [BUG - ALTA]: Validar stock disponible antes de confirmar el pedido.
    // Actualmente se puede confirmar un pedido aunque no haya stock.
    // Riesgo: overselling en producción.
    Order order = Order.create(command.customerId());

    // TODO [REFACTOR - MEDIA]: Delegar cálculo de descuentos a CouponStrategy.
    // Riesgo: añadir nuevos tipos de descuento implica modificar este método.
    applyDiscounts(order, command.couponCode());

    return orderRepository.save(order);
}
```

### Ciclo de Prompting Iterativo
```
Prompt Inicial (amplio) → Respuesta genérica
    ↓ Refinar con restricciones
Prompt 2 (específico) → Respuesta más ajustada
    ↓ Pedir edge cases
Prompt 3 (validación) → Versión robusta
    ↓ Pedir tests
Prompt 4 (verificación) → Código + Tests listos
```

## 8. Biblioteca de Prompts: Referencia por Categoría

Estos prompts aplican la estructura aprendida en este tema: **objetivo + restricciones + criterio de aceptación**. Úsalos como plantillas y adáptalos a tu contexto.

### 🔐 Seguridad

**Auditoría de endpoints:**
```
Audita los siguientes endpoints REST de mi aplicación Spring Boot:

Restricciones:
- Usa Spring Security 6.x como referencia
- No propongas cambios en la lógica de negocio, solo en la capa de seguridad
- El proyecto usa JWT con Keycloak como proveedor

Busca y reporta:
1. Endpoints sin autenticación que deberían tenerla
2. Roles mal asignados o ausentes (@PreAuthorize)
3. Exposición de datos sensibles en las respuestas (contraseñas, tokens, PII)
4. Cabeceras de seguridad HTTP que faltan (CORS, CSP, X-Frame-Options)

Clasifica cada hallazgo por severidad: Crítica / Media / Baja.
```

**Configuración de Spring Security:**
```
Implementa la configuración de Spring Security para una API REST con los siguientes requisitos:

Restricciones:
- Spring Boot 3.4, Spring Security 6.x
- Autenticación JWT sin estado (stateless)
- No uses WebSecurityConfigurerAdapter (está deprecated)
- Usa inyección por constructor, no @Autowired

Criterios de aceptación:
- Los endpoints bajo /api/public/** son accesibles sin token
- Los endpoints bajo /api/admin/** requieren rol ADMIN
- El resto requiere autenticación básica con rol USER
- Crea un test de integración con @WebMvcTest que valide los tres escenarios
```

**Análisis de vulnerabilidades OWASP:**
```
Analiza este servicio Spring Boot buscando las siguientes vulnerabilidades OWASP Top 10:

1. Inyección SQL (aunque use JPA, busca uso de queries nativas)
2. Broken Access Control (¿se valida que el usuario solo accede a sus datos?)
3. Security Misconfiguration (¿hay endpoints de actuator expuestos sin protección?)
4. Sensitive Data Exposure (¿se loguean datos sensibles?)

Para cada vulnerabilidad encontrada, muestra el fragmento de código afectado
y propón la corrección concreta.
```

### ♻️ Refactorización

**Extracción a arquitectura hexagonal:**
```
Refactoriza [NombreServicio] para separar dominio de infraestructura
siguiendo arquitectura hexagonal.

Restricciones:
- No cambies los contratos públicos (firmas de métodos de la interfaz)
- No uses Lombok, usa Java Records para los DTOs
- Mantén Java 21 y Spring Boot 3.4
- No migres la persistencia, solo reorganiza la estructura de paquetes

Estructura destino:
- domain/model/ → entidades y value objects
- domain/port/in/ → casos de uso (interfaces)
- domain/port/out/ → puertos de salida (interfaces)
- infrastructure/rest/ → adaptador REST
- infrastructure/persistence/ → adaptador JPA

Criterios de aceptación:
- El dominio no importa nada de org.springframework
- Los tests existentes siguen compilando y pasando
```

**Eliminar lógica de negocio del Controller:**
```
Este Controller tiene lógica de negocio que no le corresponde.

NO ejecutes los cambios todavía. Dame un plan en 4 pasos donde:
1. Cada paso sea un commit autocontenido
2. No rompa la compilación en ningún momento
3. Los tests existentes sigan en verde tras cada paso
4. Indiques qué archivos se crean y cuáles se modifican en cada paso

Al final del plan, señala qué tests nuevos habría que añadir
para cubrir la lógica que estás moviendo al servicio.
```

**Modernización a Java 21:**
```
Moderniza este código Java 8/11 para aprovechar las características de Java 21.

Restricciones:
- No cambies la lógica de negocio, solo la sintaxis y estructuras de datos
- No introduzcas dependencias nuevas

Aplica donde corresponda:
- Records en lugar de clases con solo getters/setters
- Sealed classes si hay jerarquías cerradas de tipos
- Pattern matching en instanceof y switch expressions
- Text blocks para strings multilínea (SQL, JSON)
- Optional correctamente (sin .get() sin isPresent())

Para cada cambio, añade un comentario indicando qué característica
de Java 21 se está usando y por qué mejora el código.
```

```
Genera el adaptador REST en Spring Boot 3.4 para el recurso [Recurso]
siguiendo el contrato definido en [archivo.yaml].

Restricciones:
```

**Diseño de API REST con trade-offs:**
```
Necesito diseñar el endpoint para [describir operación de negocio].

Dame tres alternativas de diseño REST y compáralas:
1. [Opción A, ej: recurso clásico con PUT]
2. [Opción B, ej: sub-recurso]

Para cada una evalúa:

```

**Gestión de errores y respuestas:**
```
Implementa un manejador global de excepciones para la API REST con:

Restricciones:
- Spring Boot 3.4, usa @RestControllerAdvice
- Formato de error: RFC 7807 (ProblemDetail), no inventes un formato propio
- No expongas stack traces ni mensajes internos al cliente

Cubre al menos:
- ConstraintViolationException → 400 con detalle de campos inválidos
- EntityNotFoundException → 404
- OptimisticLockException → 409
- Excepciones no controladas → 500 genérico (sin detalle interno)
```
