# Tema 6. AGENT MODE EN JETBRAINS: TRABAJO SEMI-AUTÓNOMO CON CONTROL DEL DEV

El **Agent Mode** en IntelliJ marca la diferencia entre "chatear con una IA" y "colaborar con un ejecutor". En este módulo, aprenderemos a delegar tareas complejas que afectan a múltiples archivos, tratando a la IA como un agente que propone planes de acción sobre nuestro sistema de archivos.

---

## 1. Qué es Agent Mode y qué cambia respecto a Chat normal

* **Capacidad de Acción:** A diferencia del Chat (que solo sugiere texto), el Agente puede leer múltiples archivos del proyecto, proponer ediciones reales y, en entornos compatibles, interactuar con el terminal del IDE.
* **Ciclo de Razonamiento:** El agente no responde de inmediato; primero "piensa", busca archivos relevantes, construye un plan y luego lo ejecuta.
* **Autocorrección:** Si el agente propone un cambio que rompe la compilación, puede (en teoría) detectar el error y proponer una corrección antes de que tú intervengas.
* **El cambio de rol:** Pasas de ser un "escritor de código" a un "supervisor de misiones".

### Chat vs Agent: la diferencia en la práctica

| Dimensión | Chat (Panel lateral) | Agent Mode |
|---|---|---|
| **Acceso a archivos** | Lee los que tú pegas o referencias | Navega el proyecto autónomamente |
| **Capacidad de escritura** | Solo sugiere texto | Edita archivos directamente |
| **Ejecución de comandos** | No | Puede ejecutar tests y comandos del terminal |
| **Alcance** | Un archivo / fragmento | Multi-archivo, multi-paquete |
| **Supervisión requerida** | Media (revisas texto) | Alta (revisas cambios reales en el sistema de archivos) |
| **Cuándo usarlo** | Diseño, consultas, explicaciones | Refactors estructurales, migraciones, scaffolding |

### El ciclo interno del agente

Cuando lanzas una tarea al Agent Mode, este no actúa como un chat normal. Ejecuta un bucle de razonamiento:

```
1. PLAN    → "¿Qué archivos necesito leer para entender el sistema?"
2. OBSERVE → Lee OrderService.java, OrderRepository.java, OrderDTO.java...
3. ACT     → Propone y aplica cambios en el sistema de archivos
4. VERIFY  → Ejecuta `mvn test` para comprobar que no ha roto nada
5. ITERATE → Si hay errores, vuelve al paso 3
```

> ⚠️ Este ciclo puede ejecutarse varias veces antes de que veas un resultado. Es normal. No interrumpas el agente en mitad de un ciclo; espera a que se detenga para revisar.

## 2. Qué tareas delegar: El catálogo de misiones

Un Senior no usa al agente para un método, lo usa para cambios estructurales. La regla práctica: **si el cambio toca más de 3 archivos, el agente es más eficiente que tú.**

### Generación de cambios multi-archivo

**Escenario:** Añades un campo `discountCode` a la entidad `Order`. Manualmente tendrías que actualizar la entidad JPA, el DTO de request, el DTO de response, el mapper, el controlador, la query JPQL y los tests.

**Misión para el agente:**
```
Añade el campo `discountCode` (String, nullable) a la entidad Order.
Actualiza en cascada:
- La entidad JPA con la columna correspondiente
- OrderRequest y OrderResponse (DTOs)
- OrderMapper (MapStruct o manual, respeta el patrón existente)
- El controlador si es necesario para exponer el campo
- Los tests unitarios existentes para que sigan compilando
No toques la lógica de negocio de descuentos: ese campo es solo de trazabilidad.
```

### Refactors guiados

**Escenario:** Tienes un `OrderService` de 600 líneas donde la lógica de negocio, la persistencia y las llamadas a servicios externos están mezcladas.

**Misión para el agente:**
```
Extrae la lógica de cálculo de precio de OrderService hacia un 
nuevo servicio de dominio PricingService.
Restricciones:
- PricingService no puede tener dependencias de infraestructura (sin @Repository)
- Los tests existentes de OrderService deben seguir pasando
- Usa inyección por constructor
Dame primero el plan de archivos a crear/modificar antes de ejecutar.
```

### Testing de integración completo

**Escenario:** Tienes un repositorio Spring Data sin ningún test de integración.

**Misión para el agente:**
```
Crea una suite de tests de integración para OrderRepository usando Testcontainers.
La suite debe incluir:
- Setup con PostgreSQL via Testcontainers
- Script SQL de datos de semilla en src/test/resources/db/seed.sql
- Tests para: findById (existente y no existente), save, findByCustomerId
- Usa @DataJpaTest con la configuración de Testcontainers
```

### Docs-as-Code

**Misión para el agente:**
```
Analiza los @RestController del paquete com.example.orders.infrastructure.rest
y genera la especificación OpenAPI 3.1 en src/main/resources/openapi/orders-api.yaml.
Incluye: todos los endpoints, esquemas de request/response y los códigos HTTP posibles.
```

## 3. Estrategia "brief → plan → execute → review → iterate"

Para mantener el control, el Senior debe imponer un protocolo de trabajo. El error más común es lanzar la misión y pulsar "aceptar todo" al final. Eso convierte al Senior en un "copiar-pegar" de lujo.

### 1. Brief (Instrucción)

Define el objetivo con **tres elementos obligatorios**: qué quieres, qué no puede hacer, y cómo vas a validar el éxito.

```
# Misión: Añadir Auditoría a los Servicios de Facturación

## Objetivo
Añadir logging de auditoría a todos los métodos públicos de los 
servicios en el paquete com.example.billing.domain.service.

## Restricciones
- No usar librerías externas (ni AspectJ, ni Spring AOP avanzado)
- No modificar las firmas de los métodos públicos
- El log debe incluir: nombre del método, parámetros de entrada, timestamp

## Criterio de éxito
- Los tests existentes siguen en verde tras el cambio
- Puedo ver el log al ejecutar BillingServiceTest manualmente
```

### 2. Plan (Validación)

Antes de ejecutar, pide siempre el plan:

```
Antes de hacer ningún cambio, muéstrame:
1. Qué archivos vas a crear
2. Qué archivos vas a modificar y qué cambio exacto vas a hacer en cada uno
3. Qué archivos vas a leer pero NO modificar

No ejecutes hasta que yo te diga "adelante".
```

El plan te permite detectar errores arquitectónicos **antes** de que el agente toque el código. Un agente que va a modificar `UserRepository` cuando la misión era sobre facturación es una señal de alerta.

### 3. Execute (Ejecución)

Una vez validado el plan, das el "adelante". El agente aplica los cambios. Durante la ejecución:
- No modifiques archivos manualmente: créarías conflictos.
- Si el agente se atasca en un bucle de errores, para y reinicia con más restricciones.

### 4. Review (Revisión)

**El paso más importante.** Abre el panel de cambios de IntelliJ (`Git > Commit`) y revisa el diff archivo por archivo.

| Qué revisar | Señal de alerta |
|---|---|
| Lógica de negocio modificada | El agente cambió comportamiento que no pediste |
| Imports añadidos | Nuevas dependencias que no reconoces |
| Tests modificados para que pasen | El agente "arregló" el test en lugar de el código |
| Cambios en archivos fuera del scope | El agente tocó cosas que no debía |

### 5. Iterate (Refinamiento)

Tras la revisión, el ciclo no termina. Los ajustes finos son parte del proceso:

```
Bien, los cambios en BillingService son correctos. Pero en el diff veo que:
1. El log en createInvoice está imprimiendo el objeto completo del parámetro — 
   puede contener datos sensibles. Cambia el log para que solo registre el ID.
2. En el método deleteInvoice no se ha añadido el log de auditoría. Inclúyelo.
```

## 4. Manejo de limitaciones conocidas en JetBrains

### Contexto finito: el agente no ve todo el repo

El agente no carga todo tu proyecto en memoria. Trabaja con una ventana de contexto que se llena con los archivos más relevantes para la tarea. Si tu proyecto tiene 200 clases, el agente verá quizás 15-20.

**Estrategia para ayudarle:**

```
# ❌ MAL — Misión vaga, el agente no sabe por dónde empezar
"Refactoriza la capa de persistencia"

# ✅ BIEN — Misión acotada con archivos concretos
"Refactoriza OrderJpaRepository para que implemente la interfaz 
OrderRepository del dominio (actualmente no la implementa). 
Los archivos relevantes son:
- domain/port/out/OrderRepository.java (la interfaz)
- infrastructure/persistence/OrderJpaRepository.java (la implementación actual)
- infrastructure/persistence/OrderEntity.java (la entidad JPA)"
```

**El símbolo `@workspace`** (disponible en algunos modos) le indica al agente que debe escanear más allá de los archivos abiertos. Úsalo cuando la misión sea transversal a varios paquetes.

### Terminal y comandos: saber qué puede ejecutar el agente

| Comando | Puede ejecutarlo el agente | Qué debes hacer tú |
|---|---|---|
| `mvn test` | ✅ Sí | Revisar el output en el terminal del IDE |
| `mvn verify` | ✅ Sí | — |
| `docker-compose up` | ⚠️ A veces | Levantarlo manualmente antes de lanzar la misión |
| `git commit` | ❌ No (por diseño) | Commitear manualmente tras revisar el diff |
| `psql` / consultas directas a BD | ❌ No | Verificar estado de BD manualmente si es necesario |

> 💡 **Tip de Senior:** Si tu misión requiere que la base de datos esté levantada (por ejemplo, para ejecutar tests de integración con Testcontainers), lanza `docker-compose up -d` antes de iniciar el Agent Mode. El agente no gestiona el ciclo de vida de Docker.

### Foco en la tarea: el problema de las misiones "demasiado grandes"

Un agente que recibe una misión demasiado amplia no falla de forma obvia: genera código que parece correcto pero tiene inconsistencias sutiles.

```
# ❌ MAL — Degradación de precisión garantizada
"Refactoriza toda la persistencia del proyecto para usar Spring Data JPA 
en lugar de JDBC directo"

# ✅ BIEN — Misión acotada, resultado predecible
"Migra únicamente UserJdbcRepository a Spring Data JPA. 
No toques otros repositorios. Una vez terminado, te pediré el siguiente."
```

La regla práctica: **una misión de agente, un agregado del dominio.** Orders, Users, Invoices... por separado.

## 5. Control de riesgos: Seguridad y Reversibilidad

### Commit antes de empezar: la regla de oro

El Agent Mode debe trabajar siempre sobre un estado limpio de Git. Si el agente genera cambios incorrectos y tienes trabajo sin commitear, mezclarás sus cambios con los tuyos y perderás trazabilidad.

```bash
# Antes de cualquier misión de agente:
git status          # Debe estar limpio
git stash           # Si tienes cambios que no quieres perder
git checkout -b feature/agent-refactor-pricing  # Rama limpia para el agente
```

### Local History de IntelliJ: el salvavidas

IntelliJ guarda un historial local de cambios independiente de Git. Si el agente entra en un bucle y te deja el proyecto en un estado inconsistente:

1. Clic derecho en el archivo → `Local History > Show History`
2. Busca el estado previo a la ejecución del agente
3. `Revert` al estado anterior

> ⚠️ El Local History es por archivo, no por proyecto completo. Si el agente modificó 15 archivos, tendrás que revertirlos uno a uno (o usar `git checkout .` si tienes el commit limpio).

### Diffs atómicos: no aceptes cambios masivos sin entenderlos

La trampa más común del Agent Mode es la siguiente: el agente modifica 20 archivos, todo parece funcionar (los tests pasan), y el Senior hace clic en "Accept All". Semanas después aparece un bug difícil de trazar porque nadie leyó ese cambio en `PaymentMapper` que el agente introdujo de paso.

**Protocolo de revisión:**

```
Para una misión que toca N archivos:
- N ≤ 5:   Revisa el diff de cada archivo línea a línea
- 5 < N ≤ 15: Revisa todos los archivos, haz zoom en los de lógica de negocio
- N > 15:  Para el agente. La misión era demasiado grande. Divídela.
```

## 6. Revisión obligatoria: Los 4 pilares del Senior

Aunque el agente diga "Success" y los tests pasen, el trabajo del Senior no ha terminado. Los tests verifican comportamiento funcional, no estos cuatro pilares.

### 1. Invariantes del Dominio

¿El agente ha respetado las reglas de negocio que no están en los tests?

```java
// ❌ El agente puede generar esto: parece correcto, compila y los tests pasan
public Order createOrder(CreateOrderCommand command) {
    Order order = new Order();
    order.setCustomerId(command.customerId());
    order.setItems(command.items());
    order.setStatus(OrderStatus.PENDING);
    return orderRepository.save(order);
}

// ⚠️ Pero el dominio requería validar que el cliente no tenga deuda pendiente.
// Esa regla no estaba en los tests → el agente la ignoró silenciosamente.
// Pregunta obligatoria: ¿dónde está la validación del dominio?
```

### 2. Seguridad

¿El agente ha introducido configuraciones inseguras o expuesto datos sensibles?

```java
// ❌ El agente añade logging "útil" que expone datos sensibles
log.info("Processing payment for user {} with card {}",
         userId, creditCardNumber); // ← PAN en texto plano en los logs

// ❌ El agente puede relajar restricciones de seguridad para que los tests pasen
@CrossOrigin(origins = "*") // ← Añadido para resolver un error de CORS en test
@RestController
public class OrderController { ... }
```

**Lista de verificación de seguridad post-agente:**
- Buscar `origins = "*"` en anotaciones `@CrossOrigin`
- Buscar `log.info/debug` con parámetros que parezcan datos de usuario
- Revisar si se han añadido `@Transactional(readOnly = false)` donde no corresponde
- Verificar que no se hayan expuesto endpoints sin `@PreAuthorize`

### 3. Rendimiento

¿El agente ha introducido el problema N+1 o bucles ineficientes?

```java
// ❌ El agente puede generar esto en un mapper o en un servicio
public List<OrderResponse> findAllOrders() {
    return orderRepository.findAll().stream()
        .map(order -> {
            // Cada order dispara una query a la BD para cargar el customer
            Customer customer = customerRepository.findById(order.getCustomerId())
                .orElseThrow();
            return OrderResponse.from(order, customer);
        })
        .toList();
    // Con 1000 pedidos → 1001 queries a la BD
}

// ✅ Lo que debería generar (o lo que tú debes exigirle):
@Query("SELECT o FROM Order o JOIN FETCH o.customer WHERE o.status = :status")
List<Order> findAllWithCustomer(@Param("status") OrderStatus status);
```

**Señales de alerta de rendimiento en el diff:**
- Llamadas a repositorios dentro de un `stream().map()`
- `findAll()` sin paginación en un servicio de producción
- Más de un `JOIN FETCH` en la misma query JPQL

### 4. Observabilidad

¿El agente ha mantenido (o eliminado) los logs, métricas y trazas necesarios?

```java
// ⚠️ El agente puede eliminar logs existentes al refactorizar
// porque "el código es más limpio sin ellos"
// Pero ese log era la única señal de alerta en producción para ese flujo

// Antes del refactor (el agente eliminó esto):
log.info("Invoice {} created for customer {} - amount: {}", 
    invoice.getId(), customerId, invoice.getAmount());

// Después del refactor: el agente simplificó el método y el log desapareció.
// Resultado: perdiste visibilidad en producción sin saberlo.
```

## 7. Patrones avanzados de misiones

### El patrón "Scaffolding Primero"

Para proyectos nuevos o módulos desde cero, el agente es especialmente eficiente si le pides primero la estructura y luego la implementación.

**Fase 1 — Estructura:**
```
Crea el scaffolding del módulo de notificaciones siguiendo arquitectura hexagonal.
Solo crea los archivos vacíos con las clases/interfaces declaradas (sin implementación):
- domain/model/Notification.java
- domain/port/in/SendNotificationUseCase.java
- domain/port/out/NotificationSenderPort.java
- domain/service/NotificationService.java
- infrastructure/email/SmtpNotificationAdapter.java
No rellenes ningún método todavía. Solo la declaración de clases.
```

**Fase 2 — Implementación:**
```
Ahora implementa NotificationService según la interfaz SendNotificationUseCase.
El servicio debe delegar el envío a NotificationSenderPort.
Añade un test unitario con Mockito para el caso exitoso y para el caso 
en que el puerto lanza una excepción.
```

### El patrón "Agente como Revisor"

El agente no solo genera código; puede revisar el código que él mismo ha generado con un rol diferente.

```
Acabas de refactorizar OrderService. Ahora actúa como un Senior reviewer 
y busca en los cambios que has hecho:
1. ¿Has introducido algún problema de concurrencia?
2. ¿Hay algún @Transactional que no sea necesario o que esté mal colocado?
3. ¿Los nombres de los métodos nuevos son coherentes con el resto del código base?

Dame la revisión antes de que yo haga el commit.
```

## 🧪 Ejercicio: "La Migración Estructural"

> **Contexto:** Tienes un servicio Spring Boot donde todos los beans usan `@Autowired` por campo. El equipo ha decidido migrar a inyección por constructor en todo el paquete `com.example.orders.service`.

**Fase 1 — Preparación (2 min):**
1. Asegúrate de que `git status` está limpio.
2. Crea una rama: `git checkout -b exercise/constructor-injection`

**Fase 2 — Brief al agente (5 min):**
```
Misión: Migrar todos los @Autowired de campo a inyección por constructor 
en el paquete com.example.orders.service.

Restricciones:
- Si la clase tiene Lombok, usa @RequiredArgsConstructor en lugar de 
  generar el constructor manualmente
- Si la clase NO tiene Lombok, genera el constructor explícitamente
- No toques ningún otro paquete
- Los campos deben ser final tras el cambio

Criterio de éxito:
- mvn test pasa sin errores
- No queda ningún @Autowired en el paquete service/
```

**Fase 3 — Validar el plan (3 min):**
- El agente debe decirte qué archivos va a tocar antes de ejecutar.
- Verifica que la lista de archivos es coherente con el paquete que pediste.

**Fase 4 — Ejecutar y revisar (10 min):**
- Lanza la ejecución.
- Abre el panel `Git > Commit` y revisa el diff.
- Responde estas preguntas para cada archivo:
    - ¿Los constructores tienen los tipos correctos?
    - ¿Se ha eliminado `@Autowired`?
    - ¿Los campos son `final`?

**Fase 5 — Reflexión (5 min):**
- ¿Ha usado `@RequiredArgsConstructor` donde correspondía?
- ¿Ha tocado algún archivo fuera del paquete `service/`?
- ¿Ha generado el constructor en alguna clase que ya tenía uno?

## 💡 Tips del IDE

### "The Reviewer Layout": configura IntelliJ para el Agent Mode

Cuando trabajas con el Agent Mode, el diff es tu herramienta más importante. Configura el workspace para tenerlo siempre visible:

1. Abre `View > Tool Windows > Commit` y ancla el panel en la parte inferior.
2. En el panel Commit, activa `Show Diff` para ver los cambios en tiempo real.
3. Usa `Ctrl+D` / `Cmd+D` sobre cada archivo para abrir el diff en pantalla completa.

### Comando `@workspace` para misiones transversales

Si la misión afecta a múltiples paquetes que el agente no puede inferir del contexto abierto, empieza el prompt con:

```
@workspace Analiza todos los @Service del proyecto y...
```

Esto le indica al agente que debe escanear más allá de los archivos abiertos. Úsalo con moderación: amplía el contexto pero también el tiempo de procesamiento.

### Checkpoints manuales con Local History

Antes de lanzar una misión larga, crea un checkpoint manual:
`Clic derecho en el proyecto > Local History > Put Label`

Ponle una etiqueta como `"Before agent mission: constructor injection"`. Esto te permite revertir el proyecto completo a ese estado desde `Local History > Show History` si algo va mal.

## 8. Skills: Flujos de Trabajo Codificados como Slash Commands

Los prompts complejos tienen un problema de escala: cuando un flujo de trabajo tiene 4 fases, restricciones específicas y un criterio de validación, repetirlo en cada sesión es ineficiente y propenso a omisiones. Las **Skills** (slash commands personalizados) resuelven esto: conviertes ese flujo en un comando de una palabra que cualquier miembro del equipo puede invocar con consistencia garantizada.

### Qué es una Skill

Una Skill es un fichero Markdown cuyo contenido se convierte en el prompt del agente cuando el usuario escribe `/nombre-del-comando`. Nada más. No hay código especial ni configuración de IDE: el agente lee el fichero y lo ejecuta como si el usuario hubiera escrito ese texto.

La diferencia con escribir el prompt manualmente en cada sesión:

| Dimensión | Prompt manual | Skill |
|---|---|---|
| **Consistencia** | Varía según quién lo escribe | Idéntico en cada ejecución |
| **Descubribilidad** | Solo quien lo sabe lo usa | Listado con `/help` en el IDE |
| **Versionado** | Vive en la cabeza o en Slack | Controlado en Git junto al código |
| **Mantenimiento** | Se actualiza por cada persona | Un cambio en el fichero actualiza para todos |
| **Onboarding** | Transmisión oral | El nuevo dev lo usa el día 1 |

Una Skill codifica el **proceso del equipo**, no solo la tarea. Cuando defines que antes de refactorizar hay que generar tests de regresión y esperar confirmación del desarrollador, esa decisión queda en Git y se aplica siempre.

### Cómo se crea una Skill

Las Skills se crean como ficheros Markdown en `.claude/commands/` dentro del repositorio. Para skills disponibles en todos los proyectos, usa `~/.claude/commands/`.

```
.claude/
└── commands/
    └── refactor-class.md
```

El nombre del fichero (sin extensión) es el slash command: `refactor-class.md` → `/refactor-class`.

El fichero puede tener un frontmatter YAML opcional con metadatos y el cuerpo es directamente el prompt:

```markdown
---
description: Refactoriza una clase Java con análisis de impacto previo y tests de regresión
---

El prompt que el agente ejecutará cuando se invoque este comando...
```

El campo `description` aparece en el listado de comandos disponibles. Si no hay frontmatter, el fichero funciona igual: solo el cuerpo como prompt.

### Estructura de un buen prompt de Skill

Un prompt de Skill efectivo tiene tres partes:

1. **Contexto de entrada**: qué información necesita el agente para empezar y cómo obtenerla del usuario si no se proporciona.
2. **Fases de ejecución**: los pasos del flujo de trabajo, cada uno con sus instrucciones y sus criterios de avance o parada.
3. **Puntos de decisión humana**: dónde debe detenerse el agente y esperar confirmación explícita. Sin estos, una Skill se convierte en un agente que actúa sin supervisión.

---

### Ejemplo práctico: La Skill `/refactor-class`

Vamos a crear una skill que implementa el flujo completo de refactorización segura: analiza el impacto, genera tests de regresión antes de tocar el código, aplica el refactor en pasos verificables y produce un informe final.

#### Paso 1 — Crear el fichero de la Skill

Crea el fichero `.claude/commands/refactor-class.md`:

```markdown
---
description: Refactoriza una clase Java de forma segura: análisis de impacto, tests de regresión y refactor paso a paso con verificación continua.
---

Actúa como responsable de un refactor seguro en Java/Spring Boot.

El usuario te indicará la clase a refactorizar y el objetivo. Si no lo han especificado, pregúntales antes de continuar:
- ¿Qué clase quieres refactorizar? (ruta completa)
- ¿Cuál es el objetivo del refactor? (extraer servicio, separar responsabilidades, introducir patrón, etc.)

## Fase 1 — Análisis de impacto

Lee la clase indicada por el usuario. Luego busca todas las clases que la referencian con grep sobre el directorio `src/`.

Con esa información, produce un informe estructurado con estas secciones:

**Resumen ejecutivo** (máximo 3 líneas): qué se pide y cuál es el riesgo global (BAJO / MEDIO / ALTO).

**Mapa de impacto**: tabla con columnas `Archivo | Tipo de cambio | Riesgo` para cada fichero afectado.

**Riesgos detectados**: revisa explícitamente estas categorías y señala lo que encuentres:
- *Transaccionalidad*: ¿hay `@Transactional` en los métodos a mover? ¿el refactor puede romper los límites transaccionales?
- *Rendimiento*: ¿el refactor puede introducir el problema N+1 o llamadas a repositorios dentro de bucles?
- *Seguridad*: ¿hay `@PreAuthorize` o lógica de autorización inline que podría desplazarse accidentalmente?
- *API pública*: ¿algún método afectado forma parte de un endpoint REST? Señálalo como breaking change potencial.

**Orden de ejecución recomendado**: pasos numerados para hacer el refactor de forma segura y reversible.

Una vez generado el informe, muéstraselo al usuario y espera su confirmación explícita antes de continuar:
**"¿Procedo con el refactor? (sí/no)"**

Si el riesgo global es ALTO, añade antes de la pregunta:
> ⚠️ El análisis detecta riesgo ALTO. Verifica que tienes un commit limpio en Git antes de continuar.

No avances a la Fase 2 sin recibir "sí" del usuario.

## Fase 2 — Tests de regresión

Antes de modificar ningún fichero de producción, genera tests que capturen el comportamiento actual de la clase.

Lee los tests existentes en el proyecto para reproducir sus patrones (nomenclatura, organización, uso de fixtures, anotaciones). Aplica estas reglas de selección de capa:
- Lógica de dominio pura → test unitario con JUnit 5 + Mockito
- Repositorio Spring Data → `@DataJpaTest` + Testcontainers
- Controlador REST → `@WebMvcTest` + MockMvc

Para cada método público relevante genera al menos: caso feliz, caso de entrada inválida o vacía, y caso de error esperado.

Los tests generados deben pasar con el código **sin modificar**. Aplícalos y ejecuta:
    ```bash
        mvn test -pl <módulo>
    ```
Si algún test falla antes del refactor, detente y muestra el error al usuario. No continúes hasta que los tests pasen con el código original.

## Fase 3 — Refactorización

Con los tests en verde, aplica el refactor siguiendo el orden de ejecución del informe de la Fase 1, un paso a la vez.

Después de cada paso:
1. Ejecuta `mvn test -pl <módulo>`
2. Si los tests fallan: detente, muestra el error y espera instrucciones del usuario.
3. Si los tests pasan: continúa con el siguiente paso.

No agrupes varios pasos entre ejecuciones de tests.

## Fase 4 — Informe final

Al completar todos los pasos, genera un resumen con:
- Archivos creados (con descripción de su propósito)
- Archivos modificados (con descripción del cambio aplicado)
- Archivos eliminados
- Resultado final de `mvn test` (número de tests, failures, skipped)
- Próximos pasos recomendados si el refactor es parte de una migración mayor
```

### ✅ Puntos Clave sobre Skills

- Una Skill no es un atajo de teclado: es una especificación ejecutable del proceso del equipo.
- Los **puntos de decisión humana** son el elemento más importante de una Skill. Sin ellos, el agente actúa sin freno. Cada `"¿Procedo? (sí/no)"` es un checkpoint de supervisión intencionado.
- Una Skill bien escrita hace que el onboarding sea trivial: el nuevo desarrollador no necesita conocer el proceso porque la Skill lo codifica.
- Empieza con una Skill simple (4-6 pasos) y añade complejidad solo cuando la uses lo suficiente como para saber qué falta. La sobreingeniería de prompts produce agentes frágiles.
- El prompt de la Skill es código del equipo: revísalo, mejóralo y trátalo con el mismo rigor que el código de producción.

## 9. Subagentes: Especialización y Orquestación

En el punto anterior hemos creado la skill `/refactor-class` con instrucciones inline para el análisis de impacto y la generación de tests. Este enfoque funciona bien desde el primer día y no requiere configuración adicional. Sin embargo, tiene un límite: las instrucciones inline son genéricas. El agente aplica el criterio que le das en el prompt, pero no acumula especialización ni conocimiento profundo del dominio de testing o arquitectura de tu proyecto.

Cuando el mismo flujo se ejecuta repetidamente, cuando el equipo crece y necesita consistencia más estricta, o cuando la calidad de cada fase necesita elevarse, el siguiente paso es extraer esas instrucciones inline a **subagentes especializados**.

### Qué es un subagente

Un subagente es un agente con un **system prompt dedicado** que define su rol, conocimiento y restricciones con mucho más detalle del que cabe en una fase de una Skill. La diferencia con el agente general es la misma que hay entre pedirle algo a un compañero de equipo cualquiera y pedírselo al experto de seguridad, al especialista en tests o al arquitecto.

Cuando Claude Code lanza un subagente, este:

1. Hereda el contexto del proyecto (lee el `CLAUDE.md` del repo)
2. Aplica el system prompt de su especialidad **encima** de ese contexto
3. Trabaja de forma aislada: no mezcla su razonamiento con el del agente principal
4. Devuelve un resultado estructurado que el agente orquestador puede procesar

```
┌─────────────────────────────────────────────────────────────┐
│              SKILL /refactor-class (orquestador)            │
│                                                             │
│   Recibe la misión → lanza subagentes especializados       │
│   → consolida los resultados → actúa con más criterio      │
└─────────────────────────────────────────────────────────────┘
          │                          │
          ▼                          ▼
┌──────────────────────┐      ┌──────────────────────────┐
│ java-test-specialist │      │ java-refactor-analyst    │
│ (reemplaza Fase 2)   │      │ (reemplaza Fase 1)        │
└──────────────────────┘      └──────────────────────────┘
```

La Skill del punto 8 sigue siendo la interfaz de usuario. Lo que cambia es que las Fases 1 y 2, en lugar de ejecutarse con instrucciones inline, se delegan a subagentes que aplican un criterio más profundo y consistente.

### Por qué importa esta separación

El agente general sabe de todo pero profundiza en nada. Un subagente especializado en tests conoce los patrones de tu proyecto, sabe qué hacer con Testcontainers, distingue un test de integración de uno unitario y aplica las convenciones del equipo sin que se las recuerdes en cada prompt.

La separación también mejora la **trazabilidad**: sabes exactamente qué razonamiento produjo cada parte del resultado y puedes mejorar cada subagente de forma independiente sin tocar los demás.

### Dónde se definen los subagentes

Los subagentes se crean como ficheros Markdown en `.claude/agents/` dentro del repositorio (o en `~/.claude/agents/` para que estén disponibles en todos los proyectos):

```
.claude/
├── agents/
│   ├── java-test-specialist.md
│   └── java-refactor-analyst.md
└── commands/
    └── refactor-class.md
```

Cada fichero tiene un frontmatter YAML con los metadatos del agente y el cuerpo es el system prompt:

```markdown
---
name: nombre-del-agente
description: Descripción concisa — usada por el orquestador para decidir cuándo invocar este agente
model: claude-sonnet-4-6
---

System prompt del agente: rol, reglas, conocimiento especializado...
```

El campo `description` es crítico: el agente orquestador lo usa para decidir cuándo delegar en este subagente. Escríbelo pensando en cuándo **no** debe usarse este agente, no solo cuándo sí.

### Ejemplo práctico: Evolucionando `/refactor-class` con subagentes especializados

#### Paso 1 — Crear el subagente especialista en testing

Crea el fichero `.claude/agents/java-test-specialist.md`:

```markdown
---
name: java-test-specialist
description: Specialist in JVM testing strategy. Use when you need to generate, review or improve tests for Java/Spring Boot code. Knows JUnit 5, Mockito, Testcontainers, @DataJpaTest, @WebMvcTest, and Spring Boot test slices. Applies the testing patterns already present in the project.
model: claude-sonnet-4-6
---

Eres un especialista en testing para proyectos Java con Spring Boot 3.4+ y Java 21+.

## Tu rol

Analizas código Java y generas o mejoras tests siguiendo estas prioridades:
1. **Respetar los patrones existentes**: antes de generar nada, lees los tests existentes en el proyecto y reproduces su estructura (nomenclatura, organización, uso de fixtures).
2. **Elegir la capa de test correcta**:
    - Lógica de dominio pura → test unitario con JUnit 5 + Mockito
    - Repositorios Spring Data → `@DataJpaTest` + Testcontainers
    - Controladores REST → `@WebMvcTest` + MockMvc
    - Flujos completos → `@SpringBootTest` + Testcontainers
3. **Cobertura de casos límite**: para cada método público, identificas al menos: caso feliz, caso nulo/vacío, caso de error esperado.

## Restricciones

- No generas tests que dependan de datos hardcodeados en BD de desarrollo.
- No usas `@SpringBootTest` si un test slice es suficiente (coste de arranque innecesario).
- Si el proyecto usa Lombok, usas sus anotaciones en las clases de test también (`@Builder` para construir fixtures).
- Los tests deben compilar en el primer intento: lees las dependencias del `pom.xml` antes de añadir imports.

## Formato de respuesta

Devuelve siempre:
1. **Estrategia de testing**: qué tipo de test usarás y por qué.
2. **Lista de casos de test**: en formato tabla (nombre del test, escenario, resultado esperado).
3. **Código de los tests**: completo y compilable.
4. **Cobertura estimada**: qué ramas del código cubren estos tests.
```

#### Paso 2 — Crear el subagente especialista en análisis de refactorizaciones

Crea el fichero `.claude/agents/java-refactor-analyst.md`:

```markdown
---
name: java-refactor-analyst
description: Specialist in Java refactoring risk analysis. Use when you need to evaluate the impact and risks of a refactoring before applying it. Identifies affected classes, architectural violations, N+1 risks, and transaction boundary issues. Does NOT write code — produces analysis reports only.
model: claude-sonnet-4-6
---

Eres un arquitecto senior especializado en análisis de impacto de refactorizaciones en proyectos Java/Spring Boot.

## Tu rol

**Solo analizas. No escribes código de producción.**

Cuando recibes una clase o fragmento para refactorizar:

1. **Mapa de dependencias**: identificas qué otras clases dependen de la clase objetivo (quién la inyecta, quién hereda de ella, qué interfaces implementa).
2. **Análisis de riesgos por categoría**:
    - *Transaccionalidad*: ¿hay `@Transactional` en los métodos afectados? ¿el refactor puede romper los límites transaccionales?
    - *Concurrencia*: ¿hay estado mutable compartido? ¿locks explícitos?
    - *Rendimiento*: ¿el refactor puede introducir el problema N+1? ¿llamadas a repositorios dentro de bucles?
    - *Seguridad*: ¿hay `@PreAuthorize` o lógica de autorización inline que puede desplazarse accidentalmente?
3. **Lista de archivos a modificar**: ordenada por riesgo (alto / medio / bajo).
4. **Criterio de reversibilidad**: qué cambios son seguros de hacer en una sola iteración y cuáles requieren dos pasos (p.ej., introduce nueva clase → migra usos → elimina antigua).

## Restricciones

- No propones cambios de nombres de métodos públicos que formen parte de una API REST expuesta sin señalarlo explícitamente como breaking change.
- Si detectas que el refactor propuesto viola el principio de responsabilidad única en la dirección equivocada (agrupa en lugar de separar), lo señalas.

## Formato de respuesta

Devuelve siempre un informe estructurado con estas secciones:
1. **Resumen ejecutivo** (3 líneas máximo): qué se pide, qué riesgo global tiene (BAJO / MEDIO / ALTO).
2. **Mapa de impacto**: tabla con archivos afectados, tipo de cambio y nivel de riesgo.
3. **Riesgos detallados**: una subsección por cada riesgo encontrado.
4. **Orden de ejecución recomendado**: pasos numerados para hacer el refactor de forma segura.
5. **Criterios de validación**: cómo verificar que el refactor no ha roto nada (qué tests ejecutar, qué endpoints probar).
```

## Fase 3 — Refactorización

Con los tests en verde, aplica el refactor siguiendo el orden de ejecución recomendado en el informe de la Fase 1, un paso a la vez.

Después de cada paso:
1. Ejecuta `mvn test -pl <módulo>`
2. Si los tests fallan: detente, muestra el error y espera instrucciones del usuario.
3. Si los tests pasan: continúa con el siguiente paso.

## Fase 4 — Informe final

Al completar todos los pasos, genera un resumen con:
- Archivos creados (con descripción de su propósito)
- Archivos modificados (con descripción del cambio aplicado)
- Archivos eliminados
- Resultado final de `mvn test`
- Próximos pasos recomendados si el refactor es parte de una migración mayor

### ✅ Puntos Clave sobre Subagentes

- Los subagentes no son agentes "más potentes"; son agentes **más enfocados**. Su valor está en el system prompt, no en el modelo.
- El campo `description` del frontmatter es el contrato entre el subagente y el orquestador. Escríbelo pensando en cuándo NO debe usarse este agente, no solo cuándo sí.
- Un subagente que solo analiza (sin escribir código) es igual de valioso que uno que genera código. La separación de análisis y ejecución reduce el riesgo de cambios incorrectos.
- La progresión natural es: **Skill con instrucciones inline → Skill con subagentes**. Empieza simple; extrae a subagente cuando la calidad de una fase necesite subir o cuando el mismo rol aparezca en varias Skills.
- **Empieza simple**: un subagente con 20 líneas de system prompt bien pensadas supera a uno con 200 líneas genéricas. Itera el system prompt como iteras el código.
