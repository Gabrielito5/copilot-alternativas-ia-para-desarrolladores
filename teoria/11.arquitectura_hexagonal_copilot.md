# Tema 11. ARQUITECTURA HEXAGONAL CON COPILOT (SIN MEZCLAR CAPAS)

La arquitectura hexagonal busca proteger el núcleo del negocio de las tecnologías externas (Bases de datos, APIs, Frameworks). En este tema aprenderemos a usar Copilot para mantener esta disciplina arquitectónica, evitando que el código de infraestructura "contamine" nuestro dominio.

Aunque el título mantiene la referencia histórica a Copilot, el enfoque práctico de este tema es **agente multi-modelo**: puedes ejecutar el mismo flujo con Copilot Agent, Claude Code u otras herramientas equivalentes, aplicando selección de tier según el tipo de decisión:

- Tier alto (razonamiento profundo): diseño de puertos, ADR de arquitectura, decisiones de dominio.
- Tier balanceado: implementación de adaptadores, mappers, refactorizaciones guiadas.
- Tier rápido: boilerplate de Value Objects, tests unitarios simples, scaffolding de clases vacías.

Stack de referencia:

- Java 21+
- Spring Boot 3.4+
- Maven 3.9+
- ArchUnit (validación de reglas arquitectónicas)
- JUnit 5

## 1. Generación guiada de Puertos y Adaptadores

La arquitectura hexagonal descansa sobre dos conceptos fundamentales: **puertos** (interfaces) y **adaptadores** (implementaciones). El error más frecuente al usar un agente sin instrucciones estrictas es que genera clases que mezclan ambos roles o que hacen que el dominio dependa de la infraestructura.

### Estructura de paquetes de referencia

Antes de pedir cualquier código al agente, establece la estructura de paquetes como contrato irrenunciable:

```
src/main/java/com/example/orders/
├── domain/
│   ├── model/          # Entidades, Value Objects, Aggregates — Java puro
│   ├── port/
│   │   ├── in/         # Puertos de entrada (casos de uso, interfaces)
│   │   └── out/        # Puertos de salida (interfaces de repositorio, etc.)
│   └── service/        # Servicios de dominio (lógica sin orquestación de casos de uso)
└── infrastructure/
    ├── rest/            # Adaptador REST (Controllers, DTOs web)
    ├── persistence/     # Adaptador JPA/JDBC (entidades JPA, mappers)
    └── messaging/       # Adaptador Kafka/RabbitMQ
```

Incluir esta estructura en el contexto del agente antes de pedir código elimina el 80% de las violaciones arquitectónicas habituales.

### Definición de Puertos (Interfaces)

Un puerto de salida es una interfaz que el dominio define pero no implementa. Describe **qué** necesita el dominio (persistir un pedido) sin mencionar **cómo** (JPA, JDBC, Redis).

**Prompt al Agente:**
> "Crea el puerto de salida `OrderRepository` en `domain/port/out/`. Es una interfaz Java pura. Debe declarar las operaciones que el dominio necesita: `save`, `findById`, `findByCustomerId`. No uses ningún tipo de Spring, JPA ni librería de infraestructura. Los tipos de retorno deben ser entidades de dominio (`Order`) o `Optional<Order>`, nunca entidades JPA."

Ejemplo de puerto de salida correcto:

```java
package com.example.orders.domain.port.out;

import com.example.orders.domain.model.Order;
import java.util.List;
import java.util.Optional;
import java.util.UUID;

public interface OrderRepository {
    Order save(Order order);
    Optional<Order> findById(UUID id);
    List<Order> findByCustomerId(UUID customerId);
}
```

Ejemplo de puerto de entrada (caso de uso) correcto:

```java
package com.example.orders.domain.port.in;

import com.example.orders.domain.model.Order;

public interface CreateOrderUseCase {
    Order execute(CreateOrderCommand command);
}
```

```java
package com.example.orders.domain.port.in;

import java.math.BigDecimal;
import java.util.List;
import java.util.UUID;

public record CreateOrderCommand(
    UUID customerId,
    List<OrderLineCommand> lines
) {
    public record OrderLineCommand(
        String productName,
        int quantity,
        BigDecimal unitPrice
    ) {}
}
```

**Regla para el agente**: si el puerto importa algo que no sea `java.*`, `com.example.*.domain.*` o tipos primitivos, la IA ha cometido un error.

### Implementación de Adaptadores

Los adaptadores son la única capa que puede conocer detalles de infraestructura. Implementan los puertos de salida definidos en el dominio.

**Prompt al Agente:**
> "Implementa el adaptador de persistencia `OrderPersistenceAdapter` en `infrastructure/persistence/`. Debe implementar `OrderRepository` (puerto de salida del dominio). Usa Spring Data JPA con entidad `OrderJpaEntity`. Incluye un mapper `OrderMapper` para convertir entre `Order` (dominio) y `OrderJpaEntity` (JPA). Restricciones: no pongas lógica de negocio en el adaptador; solo persistencia y mapeo."

Ejemplo de adaptador de persistencia:

```java
package com.example.orders.infrastructure.persistence;

import com.example.orders.domain.model.Order;
import com.example.orders.domain.port.out.OrderRepository;
import org.springframework.stereotype.Component;

import java.util.List;
import java.util.Optional;
import java.util.UUID;

@Component
public class OrderPersistenceAdapter implements OrderRepository {

    private final SpringDataOrderRepository springDataRepo;
    private final OrderJpaMapper mapper;

    public OrderPersistenceAdapter(
            SpringDataOrderRepository springDataRepo,
            OrderJpaMapper mapper) {
        this.springDataRepo = springDataRepo;
        this.mapper = mapper;
    }

    @Override
    public Order save(Order order) {
        OrderJpaEntity entity = mapper.toJpaEntity(order);
        OrderJpaEntity saved = springDataRepo.save(entity);
        return mapper.toDomain(saved);
    }

    @Override
    public Optional<Order> findById(UUID id) {
        return springDataRepo.findById(id).map(mapper::toDomain);
    }

    @Override
    public List<Order> findByCustomerId(UUID customerId) {
        return springDataRepo.findByCustomerId(customerId)
            .stream()
            .map(mapper::toDomain)
            .toList();
    }
}
```

Ejemplo de adaptador REST (entrada):

```java
package com.example.orders.infrastructure.rest;

import com.example.orders.domain.model.Order;
import com.example.orders.domain.port.in.CreateOrderCommand;
import com.example.orders.domain.port.in.CreateOrderUseCase;
import jakarta.validation.Valid;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

@RestController
@RequestMapping("/api/orders")
public class OrderController {

    private final CreateOrderUseCase createOrderUseCase;

    public OrderController(CreateOrderUseCase createOrderUseCase) {
        this.createOrderUseCase = createOrderUseCase;
    }

    @PostMapping
    public ResponseEntity<OrderResponse> createOrder(
            @Valid @RequestBody CreateOrderRequest request) {

        CreateOrderCommand command = new CreateOrderCommand(
            request.customerId(),
            request.lines().stream()
                .map(l -> new CreateOrderCommand.OrderLineCommand(
                    l.productName(), l.quantity(), l.unitPrice()))
                .toList()
        );

        Order order = createOrderUseCase.execute(command);
        return ResponseEntity.status(HttpStatus.CREATED).body(OrderResponse.from(order));
    }
}
```

### Mapeo de Datos

El mapper es el puente entre capas. Sin él, se cae en el anti-patrón de compartir la misma clase entre JPA y el dominio.

**Prompt al Agente:**
> "Crea `OrderJpaMapper` en `infrastructure/persistence/`. Convierte entre `Order` (dominio, Java puro) y `OrderJpaEntity` (JPA). El mapper debe ir en `infrastructure/`, nunca en `domain/`. Usa `Order.reconstitute()` para reconstruir el agregado desde la base de datos sin pasar por el constructor de creación."

```java
package com.example.orders.infrastructure.persistence;

import com.example.orders.domain.model.Order;
import com.example.orders.domain.model.OrderLine;
import com.example.orders.domain.model.OrderStatus;
import org.springframework.stereotype.Component;

import java.util.stream.Collectors;

@Component
public class OrderJpaMapper {

    public OrderJpaEntity toJpaEntity(Order domain) {
        OrderJpaEntity entity = new OrderJpaEntity();
        entity.setId(domain.getId());
        entity.setCustomerId(domain.getCustomerId());
        entity.setStatus(domain.getStatus().name());
        entity.setCreatedAt(domain.getCreatedAt());
        entity.setLines(
            domain.getLines().stream()
                .map(this::toJpaLine)
                .collect(Collectors.toList())
        );
        return entity;
    }

    public Order toDomain(OrderJpaEntity entity) {
        return Order.reconstitute(
            entity.getId(),
            entity.getCustomerId(),
            entity.getLines().stream().map(this::toDomainLine).toList(),
            OrderStatus.valueOf(entity.getStatus()),
            entity.getCreatedAt()
        );
    }

    private OrderLineJpaEntity toJpaLine(OrderLine line) {
        OrderLineJpaEntity entity = new OrderLineJpaEntity();
        entity.setProductName(line.productName());
        entity.setQuantity(line.quantity());
        entity.setUnitPrice(line.unitPrice());
        return entity;
    }

    private OrderLine toDomainLine(OrderLineJpaEntity entity) {
        return new OrderLine(
            entity.getProductName(),
            entity.getQuantity(),
            entity.getUnitPrice()
        );
    }
}
```

**Por qué `Order.reconstitute()` y no el constructor de creación**: el método de fábrica `Order.create()` genera un nuevo `UUID`, fija el estado como `DRAFT` y aplica invariantes de creación. Al reconstruir desde base de datos, ya tienes todos los valores y no debes volver a aplicar esas reglas. Disponer de dos entradas distintas hace explícita esta diferencia.

```java
// Dentro de Order.java en domain/model/
public static Order reconstitute(
        UUID id, UUID customerId, List<OrderLine> lines,
        OrderStatus status, LocalDateTime createdAt) {
    // Sin validaciones de creación — los datos ya son de BD
    return new Order(id, customerId, lines, status, createdAt);
}
```

### Checklist de Puertos y Adaptadores

- Puertos definidos como interfaces Java puras en `domain/port/`.
- Adaptadores implementan puertos, nunca al revés.
- Mappers en `infrastructure/`, nunca en `domain/`.
- `Order.reconstitute()` separado de `Order.create()`.
- Ningún adaptador importa otro adaptador directamente.

## 2. Cómo "promptear" límites de capas

Sin restricciones explícitas, el agente tomará el camino más corto: mezclar capas. La tarea del Senior es establecer las reglas como parte del contexto antes de pedir cualquier implementación.

### Prompts con restricciones de dependencia

La forma más efectiva de controlar la separación de capas es incluir las restricciones directamente en el prompt, como si fueran "leyes de importación".

**Prompt al Agente:**
> "Implementa `CreateOrderService` en `domain/service/`. Este servicio de aplicación implementa `CreateOrderUseCase` y usa `OrderRepository` (puerto de salida). Restricciones estrictas: solo puedes importar de `com.example.orders.domain.*`. Está prohibido importar `org.springframework.*`, `jakarta.persistence.*`, `infrastructure.*` o cualquier librería que no sea Java estándar. Si necesitas Spring `@Service`, ponlo en `infrastructure/config/` como Bean de configuración, no en la clase del servicio."

Ejemplo de servicio de aplicación correcto (sin anotaciones de framework):

```java
package com.example.orders.domain.service;

import com.example.orders.domain.model.Order;
import com.example.orders.domain.port.in.CreateOrderCommand;
import com.example.orders.domain.port.in.CreateOrderUseCase;
import com.example.orders.domain.port.out.OrderRepository;

public class CreateOrderService implements CreateOrderUseCase {

    private final OrderRepository orderRepository;

    public CreateOrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    @Override
    public Order execute(CreateOrderCommand command) {
        Order order = Order.create(command.customerId());

        command.lines().forEach(line ->
            order.addLine(line.productName(), line.quantity(), line.unitPrice())
        );

        order.confirm();
        return orderRepository.save(order);
    }
}
```

Registro del servicio en la capa de configuración de infraestructura:

```java
package com.example.orders.infrastructure.config;

import com.example.orders.domain.port.in.CreateOrderUseCase;
import com.example.orders.domain.port.out.OrderRepository;
import com.example.orders.domain.service.CreateOrderService;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class OrdersConfiguration {

    @Bean
    public CreateOrderUseCase createOrderUseCase(OrderRepository orderRepository) {
        return new CreateOrderService(orderRepository);
    }
}
```

Con este patrón, `CreateOrderService` es Java puro testeable sin Spring. La anotación `@Bean` vive donde corresponde: en la capa de infraestructura.

### Uso de anotaciones

El agente tiende a poner anotaciones de framework en cualquier clase que crea. El Senior debe instruirle sobre qué anotaciones pertenecen a qué capa.

Tabla de anotaciones permitidas y prohibidas por capa:

| Capa | Permitido | Prohibido |
| :--- | :--- | :--- |
| `domain/model/` | Ninguna anotación de framework | `@Entity`, `@Service`, `@Component`, `@JsonProperty`, `@NotNull` (Bean Validation) |
| `domain/port/` | Ninguna | Cualquier anotación de Spring o Jakarta |
| `domain/service/` | Ninguna (registro como `@Bean` externo) | `@Service`, `@Transactional`, `@Autowired` |
| `infrastructure/rest/` | `@RestController`, `@RequestMapping`, `@Valid`, `@NotBlank` | Lógica de dominio |
| `infrastructure/persistence/` | `@Entity`, `@Table`, `@Column`, `@Component` | Lógica de negocio |
| `infrastructure/config/` | `@Configuration`, `@Bean` | Lógica de negocio |

**Prompt al Agente:**
> "Revisa las clases en `domain/` y elimina cualquier anotación que no sea de Java estándar. Si encuentras `@Entity`, `@Service`, `@Transactional` o `@JsonProperty` en clases dentro de `domain/`, muévelas a la capa correcta o extrae el código a un adaptador. Lista los cambios antes de aplicarlos."

### Contexto por archivo

Cuando el agente tiene acceso al puerto mientras genera el adaptador, la tasa de error se reduce drásticamente. No pidas adaptadores sin tener el puerto abierto en el contexto.

Flujo recomendado con el agente:

1. Abre `domain/port/out/OrderRepository.java` (el contrato).
2. Pide la implementación del adaptador refiriéndote explícitamente a ese fichero.
3. El agente genera código que implementa exactamente esa interfaz.

**Prompt al Agente:**
> "Tengo abierto `OrderRepository.java` (el puerto de salida). Genera `OrderPersistenceAdapter` que lo implemente usando Spring Data JPA. No añadas métodos que no estén en la interfaz. El adaptador va en `infrastructure/persistence/`."

### Checklist de restricciones de capa

- El prompt incluye la lista de imports permitidos y prohibidos.
- El agente conoce la interfaz del puerto antes de generar el adaptador.
- Los servicios de dominio no tienen anotaciones de Spring.
- La configuración Spring está en `infrastructure/config/`.
- Tras cada generación, verificas con `grep -r "org.springframework" domain/` que no haya fugas.

## 3. Refactorización estructural incremental (No Big-Bang)

La refactorización hacia arquitectura hexagonal debe hacerse en pasos pequeños y verificables. Un agente sin guía tenderá a reescribir todo de golpe, romper tests y crear más deuda de la que elimina.

### De Monolito a Hexagonal

El punto de partida habitual es un servicio "gordo" de Spring que hace todo: consulta BD, aplica reglas de negocio, llama APIs externas y devuelve JSON.

Ejemplo de servicio monolítico (punto de partida):

```java
// ANTES: todo mezclado — dominio, infraestructura y web
@Service
@Transactional
public class OrderService {

    @Autowired
    private OrderJpaRepository orderJpaRepository;

    @Autowired
    private CustomerJpaRepository customerJpaRepository;

    public OrderDto createOrder(CreateOrderDto dto) {
        // Validación de negocio junto a consulta JPA
        CustomerEntity customer = customerJpaRepository.findById(dto.customerId())
            .orElseThrow(() -> new RuntimeException("Customer not found"));

        if (!customer.isActive()) {
            throw new RuntimeException("Customer is not active");
        }

        OrderEntity entity = new OrderEntity();
        entity.setId(UUID.randomUUID());
        entity.setCustomerId(dto.customerId());
        entity.setStatus("DRAFT");

        dto.lines().forEach(line -> {
            OrderLineEntity lineEntity = new OrderLineEntity();
            lineEntity.setProductName(line.productName());
            lineEntity.setQuantity(line.quantity());
            lineEntity.setUnitPrice(line.unitPrice());
            entity.getLines().add(lineEntity);
        });

        entity.setStatus("CONFIRMED");

        OrderEntity saved = orderJpaRepository.save(entity);

        // Mapping manual hacia DTO de respuesta
        return new OrderDto(saved.getId(), saved.getStatus(), saved.getCustomerId());
    }
}
```

La estrategia incremental es extraer en tres pasos: primero el dominio, luego los puertos, finalmente los adaptadores.

**Paso 1 — Extraer la lógica de dominio:**

**Prompt al Agente:**
> "En `OrderService.createOrder()` hay lógica de negocio mezclada con JPA. Extrae las reglas de negocio (validación de cliente activo, estados del pedido, añadir líneas) a una clase `Order` en `domain/model/`. Esta clase debe ser Java puro sin anotaciones de framework. Mantén `OrderService` como está de momento — solo mueve la lógica. Asegúrate de que los tests existentes siguen verdes."

Resultado esperado del paso 1:

```java
// domain/model/Order.java — Java puro, sin framework
public class Order {

    private final UUID id;
    private final UUID customerId;
    private final List<OrderLine> lines = new ArrayList<>();
    private OrderStatus status;

    public static Order create(UUID customerId) {
        return new Order(UUID.randomUUID(), customerId, OrderStatus.DRAFT);
    }

    private Order(UUID id, UUID customerId, OrderStatus status) {
        this.id = id;
        this.customerId = customerId;
        this.status = status;
    }

    public void addLine(String productName, int quantity, BigDecimal unitPrice) {
        if (status != OrderStatus.DRAFT) {
            throw new OrderAlreadyConfirmedException(id);
        }
        lines.add(new OrderLine(productName, quantity, unitPrice));
    }

    public void confirm() {
        if (lines.isEmpty()) {
            throw new EmptyOrderException(id);
        }
        this.status = OrderStatus.CONFIRMED;
    }

    // Getters...
}
```

**Paso 2 — Definir puertos:**

**Prompt al Agente:**
> "Ahora que la lógica está en `Order`, define los puertos. Crea `OrderRepository` (puerto de salida) en `domain/port/out/`. No implementes nada todavía. Solo la interfaz con los métodos que `CreateOrderService` necesitará."

**Paso 3 — Mover infraestructura al adaptador:**

**Prompt al Agente:**
> "Mueve el acceso a `OrderJpaRepository` desde `OrderService` al nuevo `OrderPersistenceAdapter` que implementa `OrderRepository`. El servicio debe quedar usando solo el puerto. Borra los imports de JPA de `OrderService` una vez terminado. Ejecuta los tests después de cada cambio y reporta si alguno se rompe."

### Extracción de infraestructura

El agente puede identificar dependencias de infraestructura dentro del dominio de forma sistemática.

**Prompt al Agente:**
> "Analiza todas las clases en `domain/` y lista qué imports son de librerías externas (Spring, Hibernate, Jackson, Apache Commons, etc.). Para cada uno, indica qué clase lo usa, qué línea importa y sugiere cómo moverlo a `infrastructure/`. No hagas cambios todavía — solo el informe."

Ejemplo de respuesta esperada del agente:

```
Análisis de imports externos en domain/:

1. domain/model/Customer.java:35 — import com.fasterxml.jackson.annotation.JsonIgnore
   Sugerencia: eliminar y crear CustomerResponse.java en infrastructure/rest/

2. domain/model/Order.java:12 — import jakarta.persistence.Entity
   Sugerencia: mover a OrderJpaEntity.java en infrastructure/persistence/

3. domain/service/NotificationService.java:8 — import org.springframework.mail.javamail.JavaMailSender
   Sugerencia: extraer interfaz EmailPort en domain/port/out/ e implementar en infrastructure/messaging/
```

Una vez revisado el informe, aplica los cambios de forma gradual, un import a la vez, verificando tests entre cada paso.

### Testeo durante la transición

El agente debe mantener los tests verdes durante toda la refactorización. Para ello, el orden importa.

Estrategia de testing durante refactoring:

1. Escribe tests para la lógica que vas a extraer **antes** de extraerla.
2. Extrae la lógica.
3. Verifica que los tests siguen pasando.
4. Elimina el código original solo cuando los tests del nuevo código están verdes.

**Prompt al Agente:**
> "Antes de refactorizar `OrderService`, escribe tests unitarios para las reglas de negocio que vas a mover a `Order`. Los tests deben cubrir: creación de pedido, añadir líneas, confirmar pedido vacío (debe fallar), añadir líneas a pedido confirmado (debe fallar). No cambies aún `OrderService`. Ejecuta los tests y confirma que están verdes."

Ejemplo de test de dominio antes de la extracción:

```java
class OrderTest {

    @Test
    void shouldNotConfirmAnEmptyOrder() {
        Order order = Order.create(UUID.randomUUID());
        assertThatThrownBy(order::confirm)
            .isInstanceOf(EmptyOrderException.class);
    }

    @Test
    void shouldNotAddLinesToConfirmedOrder() {
        Order order = Order.create(UUID.randomUUID());
        order.addLine("Laptop", 1, new BigDecimal("999.99"));
        order.confirm();

        assertThatThrownBy(() -> order.addLine("Mouse", 1, new BigDecimal("29.99")))
            .isInstanceOf(OrderAlreadyConfirmedException.class);
    }
}
```

Si estos tests existen antes de la extracción, sabes exactamente qué comportamiento preservar.

### Checklist de refactorización incremental

- Tests escritos para la lógica antes de moverla.
- Cambios en pasos pequeños verificables, no big-bang.
- `mvn test` en verde después de cada paso.
- Informe de imports externos antes de hacer cambios.
- Código original eliminado solo cuando el nuevo está cubierto por tests.

## 4. Separación del Dominio: Servicios, Aplicación e Infraestructura

Una duda frecuente es dónde poner cada clase. El agente, sin instrucciones, mezcla las responsabilidades. Este punto define las reglas de forma precisa.

### Entidades y Objetos de Valor (Domain)

El dominio contiene la lógica que no cambia independientemente de la tecnología: invariantes, reglas de negocio, comportamiento de entidades.

Reglas para el agente al generar código de dominio:

- Ninguna anotación de framework.
- Getters pero no setters públicos (la mutación ocurre por métodos con intención de negocio).
- Excepciones de dominio propias (no `RuntimeException` genérico).
- Value Objects como `record` de Java 21.

Ejemplo de entidad de dominio:

```java
package com.example.orders.domain.model;

import java.math.BigDecimal;
import java.time.LocalDateTime;
import java.util.*;

public class Order {

    private final UUID id;
    private final UUID customerId;
    private final List<OrderLine> lines;
    private OrderStatus status;
    private final LocalDateTime createdAt;

    public static Order create(UUID customerId) {
        Objects.requireNonNull(customerId, "customerId is required");
        return new Order(
            UUID.randomUUID(), customerId, new ArrayList<>(),
            OrderStatus.DRAFT, LocalDateTime.now()
        );
    }

    public static Order reconstitute(UUID id, UUID customerId, List<OrderLine> lines,
                                      OrderStatus status, LocalDateTime createdAt) {
        return new Order(id, customerId, new ArrayList<>(lines), status, createdAt);
    }

    private Order(UUID id, UUID customerId, List<OrderLine> lines,
                  OrderStatus status, LocalDateTime createdAt) {
        this.id = id;
        this.customerId = customerId;
        this.lines = lines;
        this.status = status;
        this.createdAt = createdAt;
    }

    public void addLine(String productName, int quantity, BigDecimal unitPrice) {
        if (status != OrderStatus.DRAFT) {
            throw new OrderAlreadyConfirmedException(id);
        }
        if (quantity <= 0) {
            throw new InvalidQuantityException(quantity);
        }
        lines.add(new OrderLine(productName, quantity, unitPrice));
    }

    public void confirm() {
        if (lines.isEmpty()) {
            throw new EmptyOrderException(id);
        }
        this.status = OrderStatus.CONFIRMED;
    }

    public BigDecimal calculateTotal() {
        return lines.stream()
            .map(OrderLine::subtotal)
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }

    public UUID getId() { return id; }
    public UUID getCustomerId() { return customerId; }
    public List<OrderLine> getLines() { return Collections.unmodifiableList(lines); }
    public OrderStatus getStatus() { return status; }
    public LocalDateTime getCreatedAt() { return createdAt; }
}
```

Ejemplo de Value Object con `record`:

```java
package com.example.orders.domain.model;

import java.math.BigDecimal;

public record OrderLine(String productName, int quantity, BigDecimal unitPrice) {

    public OrderLine {
        Objects.requireNonNull(productName, "productName is required");
        if (quantity <= 0) throw new InvalidQuantityException(quantity);
        if (unitPrice.compareTo(BigDecimal.ZERO) <= 0) throw new InvalidPriceException(unitPrice);
    }

    public BigDecimal subtotal() {
        return unitPrice.multiply(BigDecimal.valueOf(quantity));
    }
}
```

**Prompt al Agente:**
> "Crea la entidad de dominio `Product` con los campos: `id`, `name`, `price`, `category`, `stockQuantity`. Incluye validaciones de negocio en el constructor privado y métodos `reserve(int quantity)` y `restock(int quantity)` con sus invariantes. Sin anotaciones de framework. Usa `record` para el Value Object `Money` que encapsula `amount` y `currency`."

### Servicios de Aplicación (Use Cases)

Los servicios de aplicación orquestan el flujo: cargan entidades, aplican lógica de dominio, persisten. No contienen lógica de negocio propia.

Regla de oro: si el agente añade lógica de negocio en el servicio de aplicación, hay que moverla al dominio.

```java
package com.example.orders.domain.service;

import com.example.orders.domain.model.Order;
import com.example.orders.domain.port.in.CreateOrderCommand;
import com.example.orders.domain.port.in.CreateOrderUseCase;
import com.example.orders.domain.port.out.CustomerRepository;
import com.example.orders.domain.port.out.OrderRepository;

public class CreateOrderService implements CreateOrderUseCase {

    private final OrderRepository orderRepository;
    private final CustomerRepository customerRepository;

    public CreateOrderService(
            OrderRepository orderRepository,
            CustomerRepository customerRepository) {
        this.orderRepository = orderRepository;
        this.customerRepository = customerRepository;
    }

    @Override
    public Order execute(CreateOrderCommand command) {
        // 1. Cargar entidades necesarias
        Customer customer = customerRepository.findById(command.customerId())
            .orElseThrow(() -> new CustomerNotFoundException(command.customerId()));

        // 2. Validar precondiciones (regla de negocio en entidad, no aquí)
        customer.assertIsActive();

        // 3. Crear y ejecutar lógica de dominio
        Order order = Order.create(command.customerId());
        command.lines().forEach(line ->
            order.addLine(line.productName(), line.quantity(), line.unitPrice())
        );
        order.confirm();

        // 4. Persistir y devolver
        return orderRepository.save(order);
    }
}
```

**Prompt al Agente:**
> "Implementa el caso de uso `CancelOrderUseCase` en `domain/service/`. Debe cargar el pedido por id, llamar a `order.cancel()` y persistirlo. Si el pedido no existe, lanza `OrderNotFoundException`. No pongas en el servicio la lógica de qué estados permiten cancelación — eso va en `Order.cancel()`."

### Servicios de Infraestructura

Los servicios de infraestructura encapsulan detalles técnicos: enviar emails, llamar APIs externas, publicar eventos.

El patrón correcto: el dominio define un puerto de salida (`EmailNotifier`), la infraestructura lo implementa.

```java
// domain/port/out/EmailNotifier.java — el contrato en dominio
package com.example.orders.domain.port.out;

import java.util.UUID;

public interface EmailNotifier {
    void sendOrderConfirmation(UUID orderId, String customerEmail);
}
```

```java
// infrastructure/messaging/SmtpEmailNotifier.java — la implementación con Spring
package com.example.orders.infrastructure.messaging;

import com.example.orders.domain.port.out.EmailNotifier;
import org.springframework.mail.SimpleMailMessage;
import org.springframework.mail.javamail.JavaMailSender;
import org.springframework.stereotype.Component;

import java.util.UUID;

@Component
public class SmtpEmailNotifier implements EmailNotifier {

    private final JavaMailSender mailSender;

    public SmtpEmailNotifier(JavaMailSender mailSender) {
        this.mailSender = mailSender;
    }

    @Override
    public void sendOrderConfirmation(UUID orderId, String customerEmail) {
        SimpleMailMessage message = new SimpleMailMessage();
        message.setTo(customerEmail);
        message.setSubject("Order confirmed: " + orderId);
        message.setText("Your order has been confirmed.");
        mailSender.send(message);
    }
}
```

**Prompt al Agente:**
> "Necesito notificar al cliente cuando se confirma un pedido. Crea el puerto de salida `EmailNotifier` en `domain/port/out/` con un solo método. Implementa `SmtpEmailNotifier` en `infrastructure/messaging/` usando `JavaMailSender`. El dominio solo debe conocer la interfaz, nunca la implementación SMTP."

### Checklist de separación de capas

- `domain/model/` sin anotaciones de framework.
- Servicios de aplicación: solo orquestación, sin lógica de negocio propia.
- Servicios de infraestructura: implementan puertos, nunca se llaman directamente desde el dominio.
- Las excepciones de dominio viven en `domain/`, no en `infrastructure/`.
- Cada capa tiene tests de su propio nivel sin necesitar la capa de abajo.

## 5. Anti-patrones típicos del código generado y corrección

El agente comete estos errores porque optimiza para "funcionar", no para respetar arquitectura. Reconocerlos y corregirlos es la habilidad diferencial del Senior.

### Fuga de Anotaciones

El agente pone `@Entity` o `@Table` en la entidad de dominio porque es la forma más rápida de persistir. Esto acopla el dominio a JPA.

```java
// MALO — dominio contaminado con JPA
@Entity
@Table(name = "orders")
public class Order {
    @Id
    private UUID id;

    @Enumerated(EnumType.STRING)
    private OrderStatus status;

    @OneToMany(cascade = CascadeType.ALL)
    private List<OrderLine> lines;
    // ...
}
```

Corrección:

```java
// CORRECTO — dominio puro
public class Order {
    private final UUID id;
    private OrderStatus status;
    private final List<OrderLine> lines;
    // ...
}

// La anotación @Entity va en la entidad JPA, en infrastructure/
@Entity
@Table(name = "orders")
public class OrderJpaEntity {
    @Id
    private UUID id;

    @Enumerated(EnumType.STRING)
    private String status;

    @OneToMany(cascade = CascadeType.ALL, orphanRemoval = true)
    @JoinColumn(name = "order_id")
    private List<OrderLineJpaEntity> lines = new ArrayList<>();
    // Getters y setters necesarios para JPA...
}
```

**Prompt de corrección al Agente:**
> "La clase `Order` en `domain/model/` tiene anotaciones `@Entity`, `@Id` y `@OneToMany`. Esto es incorrecto. Crea `OrderJpaEntity` en `infrastructure/persistence/` con esas anotaciones. Limpia `Order` dejándola como Java puro. Crea `OrderJpaMapper` para traducir entre ambas."

### Lógica de Negocio en Adaptadores

El agente añade validaciones de negocio en el controlador REST porque parece lo más directo. Esta lógica debe vivir en el dominio.

```java
// MALO — validación de negocio en el controlador
@PostMapping
public ResponseEntity<OrderResponse> createOrder(@RequestBody CreateOrderRequest request) {
    // Esta validación de negocio NO pertenece aquí
    if (request.lines().stream().anyMatch(l -> l.quantity() > 1000)) {
        return ResponseEntity.badRequest().body(null);
    }
    if (request.lines().size() > 50) {
        return ResponseEntity.badRequest().body(null);
    }
    // ...
}
```

Corrección:

```java
// CORRECTO — el controlador solo mapea y delega
@PostMapping
public ResponseEntity<OrderResponse> createOrder(@Valid @RequestBody CreateOrderRequest request) {
    CreateOrderCommand command = toCommand(request);
    Order order = createOrderUseCase.execute(command);  // Las validaciones están en Order
    return ResponseEntity.status(HttpStatus.CREATED).body(OrderResponse.from(order));
}

// La validación va en el dominio
public void addLine(String productName, int quantity, BigDecimal unitPrice) {
    if (quantity > 1000) {
        throw new InvalidQuantityException("Quantity cannot exceed 1000, got: " + quantity);
    }
    // ...
}
```

**Prompt de corrección al Agente:**
> "El controlador `OrderController` valida que la cantidad no supere 1000 y que haya menos de 50 líneas. Mueve esas validaciones a `Order.addLine()` como invariantes de dominio. El controlador debe quedar sin ifs de negocio. Actualiza los tests para probar esas reglas en `OrderTest`, no en `OrderControllerTest`."

### Dependencia Inversa

El dominio conoce detalles de la base de datos o de la web, violando la regla de dependencia.

```java
// MALO — el dominio importa Spring Data
import org.springframework.data.jpa.repository.JpaRepository;

public interface OrderRepository extends JpaRepository<Order, UUID> {
    // La entidad de dominio pasa directamente a Spring Data
}
```

Corrección:

```java
// CORRECTO — el puerto de dominio es Java puro
// domain/port/out/OrderRepository.java
public interface OrderRepository {
    Order save(Order order);
    Optional<Order> findById(UUID id);
}

// infrastructure/persistence/SpringDataOrderRepository.java — Spring Data oculto en infraestructura
interface SpringDataOrderRepository extends JpaRepository<OrderJpaEntity, UUID> {
    List<OrderJpaEntity> findByCustomerId(UUID customerId);
}

// infrastructure/persistence/OrderPersistenceAdapter.java — implementa el puerto
@Component
public class OrderPersistenceAdapter implements OrderRepository {
    // usa SpringDataOrderRepository internamente
}
```

**Prompt de corrección al Agente:**
> "El puerto `OrderRepository` en `domain/` extiende `JpaRepository`. Esto hace que el dominio dependa de Spring Data — es incorrecto. Crea una interfaz Java pura `OrderRepository` en `domain/port/out/` con solo los métodos necesarios. Mueve la extensión de `JpaRepository` a `SpringDataOrderRepository` en `infrastructure/persistence/`. Usa el adaptador para conectar ambas."

### Uso de librerías externas en el Core

El agente sugiere Apache Commons, Guava u otras librerías de utilidad dentro del dominio porque las conoce bien. El dominio debe ser independiente de estas dependencias.

```java
// MALO — Guava en el dominio
import com.google.common.collect.ImmutableList;

public class Order {
    private ImmutableList<OrderLine> lines;
    // ...
}
```

```java
// MALO — Apache Commons en el dominio
import org.apache.commons.lang3.StringUtils;

public class Customer {
    public boolean hasValidEmail() {
        return StringUtils.isNotBlank(email);  // dependencia externa en dominio
    }
}
```

Corrección:

```java
// CORRECTO — Java puro
public class Order {
    private final List<OrderLine> lines;

    public List<OrderLine> getLines() {
        return Collections.unmodifiableList(lines);  // Java estándar
    }
}

public class Customer {
    public boolean hasValidEmail() {
        return email != null && !email.isBlank();  // Java 11+ sin dependencias
    }
}
```

**Prompt de corrección al Agente:**
> "Revisa las clases en `domain/` y elimina cualquier uso de Guava, Apache Commons, Lombok u otras librerías externas. Reemplaza `ImmutableList` por `Collections.unmodifiableList()`, `StringUtils.isNotBlank` por `String.isBlank()` de Java 11+, y `@lombok.Value` por `record`. El dominio debe compilar con solo dependencias de Java estándar."

### Prompt de auditoría completa

Úsalo al final de cualquier sesión de generación para detectar violaciones antes de hacer commit:

**Prompt al Agente:**
> "Audita todas las clases del proyecto y responde en tabla con columnas: archivo, violación, severidad (alta/media/baja), corrección propuesta.\n> Verifica:\n> 1. Clases en `domain/` no importan `org.springframework.*`, `jakarta.persistence.*`, `com.fasterxml.*`.\n> 2. Lógica de negocio no está en controladores ni adaptadores.\n> 3. El dominio no conoce la implementación de sus puertos.\n> 4. Los mappers están en `infrastructure/`, no en `domain/`.\n> 5. No hay librerías de terceros en `domain/model/`."

## 6. Plugin del Curso: Guardian de la Arquitectura Hexagonal

Los cinco puntos anteriores definen un estándar preciso: el dominio no conoce a nadie, los adaptadores implementan puertos, los mappers viven en infraestructura, las anotaciones de framework no contaminan el núcleo. El problema es que el agente viola este estándar con frecuencia, a menudo de forma sutil. Este punto lo convierte en un guardian automatizado.

El subagente ejecuta la auditoría de arquitectura que en el punto 5 hacíamos con un prompt manual. La skill orquesta la refactorización incremental del punto 3 — la más arriesgada de las operaciones de este tema — con análisis previo, checkpoints humanos y validación ArchUnit al final.

### Subagente: `spring-architecture-guard`

Audita el cumplimiento de la Regla de Dependencia de la Arquitectura Hexagonal en cualquier módulo Spring Boot. Va más allá de un `grep` manual: entiende la semántica de cada capa y detecta violaciones sutiles (como un servicio de dominio que usa `@Transactional` de Spring, o un adaptador REST que instancia directamente un servicio de dominio saltándose el puerto).

Crea el fichero `.claude/agents/spring-architecture-guard.md`:

name: spring-architecture-guard
description: Architecture auditor for Hexagonal Architecture (Ports & Adapters) in Spring Boot projects. Use when reviewing code generated by an AI agent to verify the Dependency Rule is respected, or before committing changes that touch domain/, application/ or infrastructure/ layers. Detects layer violations, annotation leaks, missing mappers, and inverted dependencies. Does NOT rewrite code — produces a structured violation report with severity and correction for each finding.
model: claude-sonnet-4-6
---

Eres un arquitecto senior especializado en Arquitectura Hexagonal (Ports & Adapters) para proyectos Spring Boot 3.4+ con Java 21+. Tu única función es detectar violaciones arquitectónicas. No reescribes código de producción.

## La Regla de Dependencia

La regla que auditas en cada revisión:

    ```
    Adapter (IN/OUT) → Application → Domain
    Domain NO conoce a nadie.
    Application conoce al Domain.
    Adapters conocen a todos pero NO entre ellos.
    ```

## Categorías de auditoría

Para cada violación indica: archivo, línea aproximada, tipo de violación, severidad y corrección recomendada.

### 1. Fuga de anotaciones de framework en Domain
Ninguna clase en `domain/model/`, `domain/port/` o `domain/service/` puede importar:
- `org.springframework.*` (ningún `@Service`, `@Component`, `@Transactional`, `@Autowired`)
- `jakarta.persistence.*` (ningún `@Entity`, `@Id`, `@Column`, `@OneToMany`)
- `com.fasterxml.jackson.*` (ningún `@JsonProperty`, `@JsonIgnore`)
- `jakarta.validation.*` (ningún `@NotNull`, `@Size` — la validación de entrada va en los DTOs del adaptador REST)
- Librerías de terceros (Guava, Apache Commons, Lombok) en `domain/model/`

Excepción aceptada: `@Service` en clases de `domain/service/` si el equipo ha decidido registrarlas directamente con Spring. Señalarlo como advertencia, no como error bloqueante, e indicar la alternativa de registrar como `@Bean` en `infrastructure/config/`.

### 2. Dependencias invertidas
- Cualquier clase en `domain/` que importe algo de `infrastructure/` o de `application/` que no sea un puerto.
- Cualquier clase en `application/` (servicios de aplicación) que importe de `infrastructure/`.
- Cualquier adaptador REST (`infrastructure/rest/`) que importe de `infrastructure/persistence/` directamente.
- Cualquier puerto de salida (`domain/port/out/`) que extienda una interfaz de Spring Data (`JpaRepository`, `CrudRepository`).

### 3. Lógica de negocio fuera del dominio
- Condicionales de reglas de negocio en controladores REST (`@RestController`) que deberían ser invariantes de dominio.
- Cálculos de negocio (descuentos, totales, validaciones de estado) en adaptadores de persistencia o en mappers.
- Validaciones de estado de entidades (ej: "no se puede confirmar un pedido sin líneas") implementadas en el servicio de aplicación en lugar de en la entidad de dominio.

### 4. Mappers en la capa equivocada
- Clases de mapeo entre dominio e infraestructura ubicadas en `domain/` en lugar de en `infrastructure/`.
- Lógica de conversión de tipos inline en controladores o adaptadores en lugar de delegada a un mapper dedicado.
- Una sola clase (`@Entity`) usada como entidad de dominio Y como entidad JPA a la vez (el anti-patrón más frecuente generado por agentes).

### 5. Puertos mal definidos
- Un puerto de salida (`domain/port/out/`) que declara métodos con tipos de retorno de infraestructura (`Page<OrderJpaEntity>`, `Optional<OrderEntity>`).
- Un caso de uso (`domain/port/in/`) que recibe o devuelve DTOs de la capa REST (`CreateOrderRequest`, `OrderResponse`).
- Métodos en el puerto que exponen detalles de la implementación (nombres como `saveAndFlush`, `findAllByStatusJpql`).

### 6. Construcción de objetos de dominio fuera del dominio
- `new Order(...)` llamado desde un adaptador o controlador usando el constructor de creación (en lugar de `Order.create()`).
- Reconstrucción de agregados desde base de datos sin usar `Order.reconstitute()` — los adaptadores deberían delegar al factory method correcto.

## Formato de respuesta

1. **Resumen ejecutivo** (2-3 líneas): módulo analizado, número de violaciones por severidad.
2. **Tabla de violaciones**:

| Categoría | Archivo | Línea | Violación | Severidad | Corrección |
|---|---|---|---|---|---|
| Fuga de anotaciones | Order.java | 12 | @Entity en clase de dominio | 🔴 Bloqueante | Crear OrderJpaEntity en infrastructure/persistence/ |

3. **Veredicto**: ✅ Arquitectura limpia / ⚠️ Violaciones menores / 🔴 Violaciones bloqueantes — no hacer merge.

## Severidades

- 🔴 **Bloqueante**: viola directamente la Regla de Dependencia o hace el dominio no testeable sin Spring (dominio con `@Entity`, puerto que extiende `JpaRepository`, controlador con lógica de negocio).
- 🟡 **Importante**: debilita la arquitectura sin romperla formalmente (mapper en capa equivocada, `@Service` en dominio cuando hay alternativa mejor, puerto con nombres de implementación).
- 🟢 **Sugerencia**: mejora de semántica o claridad sin impacto en la regla de dependencia.
```

3. **Lógica de negocio en controladores**: mueve las validaciones y reglas a métodos de la entidad de dominio correspondiente.

- `domain/` no depende de `infrastructure/` ni de `org.springframework.*`.
- `infrastructure/rest/` no depende de `infrastructure/persistence/`.
- Los puertos de salida en `domain/port/out/` no extienden interfaces de Spring Data.

Ejecuta los tests ArchUnit y reporta resultado.

## Fase 4 — Tests de regresión

Ejecuta el conjunto completo de tests del módulo:
    ```bash
    mvn test -pl <módulo>
    ```

Si algún test falla por los cambios estructurales (imports rotos, constructor cambiado), corrígelos. No modifiques tests para que pasen — solo actualiza los imports y los factory methods afectados por la refactorización.

#### 💡 Tip de Senior: "The ArchUnit Guardrail"

ArchUnit permite codificar las reglas de la arquitectura hexagonal como tests automáticos. Si el agente (o un humano) viola una regla, la build falla.

```java
package com.example.orders.architecture;

import com.tngtech.archunit.core.domain.JavaClasses;
import com.tngtech.archunit.core.importer.ClassFileImporter;
import com.tngtech.archunit.lang.ArchRule;
import org.junit.jupiter.api.Test;

import static com.tngtech.archunit.lang.syntax.ArchRuleDefinition.noClasses;
import static com.tngtech.archunit.library.Architectures.layeredArchitecture;

class HexagonalArchitectureTest {

    JavaClasses classes = new ClassFileImporter()
        .importPackages("com.example.orders");

    @Test
    void domainShouldNotDependOnInfrastructure() {
        ArchRule rule = noClasses()
            .that().resideInAPackage("..domain..")
            .should().dependOnClassesThat()
            .resideInAnyPackage(
                "..infrastructure..",
                "org.springframework..",
                "jakarta.persistence.."
            );
