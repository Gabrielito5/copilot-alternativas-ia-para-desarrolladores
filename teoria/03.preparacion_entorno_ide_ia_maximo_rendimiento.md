# Tema 3. Preparacion del entorno IDE + IA para maximo rendimiento

Un entorno mal configurado hace que la IA falle por falta de contexto, latencia o permisos. Un entorno bien configurado reduce friccion, mejora la calidad de las respuestas y evita fugas de informacion sensible.

---

## 1. Instalacion y configuracion de plugins (GitHub Copilot + Claude) en IntelliJ

### GitHub Copilot

1. Abre `Preferences` -> `Plugins`.
2. Busca `GitHub Copilot` e instala el plugin oficial de GitHub.
3. Reinicia IntelliJ cuando lo pida.
4. Ve a `Preferences` -> `Tools` -> `GitHub Copilot`.

### Que significa cada apartado en `Tools` -> `GitHub Copilot`

> Basado en la documentacion oficial de GitHub Copilot para JetBrains. Segun version del plugin, plan o tenant, algunos labels pueden variar.

#### A) `General`

- **Enable/Disable GitHub Copilot**: interruptor maestro del plugin en el IDE.
  - Cuando esta desactivado, desaparecen sugerencias inline.
  - Puedes desactivar globalmente o por lenguaje desde el estado de Copilot.
- **Behavior y apariencia de sugerencias**: opciones de como se presentan/aceptan sugerencias inline.
  - Impacta ergonomia de escritura (menos o mas intrusivo).
- **Auto update del plugin**: permite actualizar automaticamente.
  - Recomendado en equipos que priorizan mejoras rapidas y fixes de compatibilidad.
- **Update channel (`Stable` vs `Nightly`)**:
  - `Stable`: menos cambios, mas predecible para equipos.
  - `Nightly`: novedades antes, pero con mayor riesgo de inestabilidad.

#### B) `Completions`

- **Inline completions**: activa/desactiva completado en linea.
  - Si lo apagas, Copilot queda disponible para chat/otros flujos, pero no sugiere mientras escribes.
- **Languages**: lista de lenguajes habilitados/deshabilitados para completions.
  - Buen patron: habilitar solo lenguajes del repo para reducir ruido.
  - Si un lenguaje no debe usar IA (por politica), desactivalo explicitamente.
- **Model for completions** (si aparece en tu version): selector del modelo usado para sugerencias.
  - Usa tier rapido para boilerplate y tier balanceado para codigo no trivial.
  - En cuentas corporativas este selector puede venir limitado por politica.

#### C) Configuracion avanzada de lenguajes via `github-copilot.xml` (opcional)

- El plugin permite gobernar el allowlist de lenguajes en `github-copilot.xml`.
- Sirve para reglas mas estrictas (wildcard `*`, excepciones y bloqueo por lenguaje).
- Ruta en macOS (segun producto/version JetBrains):
  - `~/Library/Application Support/JetBrains/<product><version>/options/github-copilot.xml`

### Recomendacion de valores para curso/equipo

- `General`:
  - Copilot habilitado.
  - Canal `Stable` por defecto.
- `Completions`:
  - Inline activado.
  - Solo lenguajes del stack activo (Java, Kotlin, YAML, Markdown, Bash, etc.).
  - Modelo segun tarea/coste cuando el selector este disponible.

### Fuentes oficiales consultadas

- GitHub Docs - Configuring GitHub Copilot in your environment (seccion JetBrains): https://docs.github.com/en/copilot/how-tos/configure-personal-settings/configure-in-ide
- GitHub Docs - Installing the GitHub Copilot extension in your environment (seccion JetBrains): https://docs.github.com/en/copilot/how-tos/set-up/install-copilot-extension

### Claude en IntelliJ

1. En `Plugins`, busca el plugin oficial/disponible para integracion con Claude (o proveedor intermedio de tu organizacion).
2. Instala, reinicia, y abre su panel de configuracion.
3. Configura el modo de autenticacion (API key o login OAuth/SSO segun politica de empresa).
4. Define modelo por defecto segun tipo de tarea (rapido, balanceado, razonamiento profundo).

### Validacion rapida

- Verifica que el estado de Copilot este activo en la barra inferior del IDE.
- En `Tools` -> `GitHub Copilot` confirma: `General` y `Completions` configurados segun politica del equipo.
- Abre un archivo Java y escribe un metodo incompleto para validar inline completions.
- Desactiva temporalmente un lenguaje en `Completions` -> `Languages` y confirma que deja de sugerir en ese lenguaje.
- Si no aparece nada, revisa licencia, login, version del IDE/plugin, proxy/firewall y politicas del tenant.

## 2. Inicio de sesion, autorizacion y cuentas (personal vs corporativa/SSO)

### Cuenta personal

- Rapida para empezar, util en aprendizaje o proyectos no sensibles.
- Riesgo: mezclar repos corporativos y personales sin control centralizado.

### Cuenta corporativa con SSO

- Recomendada para trabajo real de empresa.
- Permite politicas de seguridad, auditoria, control de acceso y offboarding.
- Facilita cumplimiento legal y trazabilidad.

### Practica recomendada

- Usa cuenta corporativa para codigo de empresa.
- No mezcles extensiones autenticadas con correo personal en el mismo contexto corporativo.
- Verifica en el panel del plugin que el tenant/organizacion activo es el correcto.

### Validacion rapida

- Confirma que el plugin muestra la organizacion correcta.
- Comprueba que no hay prompts bloqueados por politicas de SSO.
- Documenta en el onboarding del equipo que cuenta usar para cada tipo de repo.

## 3. Activacion y configuracion de chats de IA en IntelliJ

1. Habilita el chat del plugin en `Tool Windows`.
2. Ajusta permisos de lectura de proyecto si el plugin lo solicita.
3. Configura contexto por defecto:
   - archivo actual
   - seleccion de codigo
   - modulo/proyecto completo (solo cuando haga falta).
4. Activa historial local si la politica lo permite; desactivalo en equipos compartidos.

### Flujo recomendado

- Preguntas locales: usa archivo actual o seleccion.
- Tareas multiarchivo: añade explicitamente clases y contratos relevantes.
- Tareas de arquitectura: aporta ADR/OpenAPI/README antes de pedir codigo.

### Validacion rapida

- Pide: "explica este metodo y sugiere test unitarios" sobre una clase concreta.
- Si responde sin contexto real, revisa permisos de indice/proyecto del plugin.

## 4. Ajustes clave del IDE para trabajar con IA (indices, proyecto, rendimiento)

### Indices y proyecto

- Deja que IntelliJ termine indexacion antes de sesiones intensas con IA.
- Marca carpetas correctamente (`Sources`, `Tests`, `Excluded`) para evitar ruido.
- Excluye `target/`, `build/`, artefactos generados y logs pesados.

### Rendimiento

- Aumenta memoria de IntelliJ si el proyecto es grande (`Help` -> `Change Memory Settings`).
- Desactiva plugins no usados para reducir latencia.
- Usa JDK del proyecto consistente con compilacion real (ej. Java 21+).

### Calidad de contexto

- Manten `README`, `CONTRIBUTING`, contratos OpenAPI/AsyncAPI y ADRs actualizados.
- La IA rinde mejor con convenciones explicitas que con inferencia implicita.

### Validacion rapida

- Mide tiempo de respuesta del chat antes/despues de limpiar indices y plugins.
- Si el IDE se bloquea o el chat tarda mucho, revisa RAM, antivirus y escaneo del repo.

## 5. Configuracion de repos y permisos para que la IA vea bien el contexto

### Estructura minima que ayuda a la IA

- `README.md` con arquitectura y comandos.
- `docs/adr/` con decisiones tecnicas.
- `openapi/` o `contracts/` con especificaciones.
- `src/main` y `src/test` bien separados.

### Permisos

- Da acceso al repo minimo necesario para la tarea.
- Evita abrir monorepo completo si solo trabajas en un modulo.
- En entornos corporativos, usa listas de inclusion/exclusion por carpetas.

### Recomendacion operativa

- Antes de pedir cambios, indica rutas exactas: `src/main/...`, `pom.xml`, `docker-compose.yml`.
- Pide validacion explicita: compilar, correr tests, revisar warnings.

### Validacion rapida

- Pide al chat un mapa del modulo actual (capas, dependencias, puertos/adaptadores).
- Si inventa estructura, faltan permisos/contexto o hay ruido en el repo abierto.

## 6. Buenas practicas para compartir contexto sin filtrar secretos (sanitizacion)

### Nunca compartas

- Tokens, passwords, API keys, secretos en `.env`.
- Datos personales reales (PII): emails, telefonos, IDs de clientes, trazas productivas.
- Certificados privados o configuracion de seguridad de produccion.

### Sanitizacion practica

- Reemplaza secretos por placeholders: `${DB_PASSWORD}`.
- Sustituye datos reales por sinteticos: `user@example.com`, `00000000-0000-0000-0000-000000000000`.
- Recorta logs al minimo necesario para depuracion.
- Comparte estructura y contratos antes que dumps completos.

### Barreras de seguridad recomendadas

- `git-secrets`, `gitleaks` o equivalentes en pre-commit/CI.
- `.gitignore` estricto para archivos sensibles.
- Politica documentada de "que SI / que NO" se pega en chat.

### Validacion rapida

- Ejecuta escaneo de secretos antes de compartir fragmentos.
- Si aparece un secreto, rota credencial y elimina el rastro de inmediato.

## 7. Retencion de datos: que se retiene y que no segun modalidad/plan

Las politicas cambian por proveedor, fecha, plan y configuracion tenant. Debes validar siempre en documentacion oficial y contrato vigente de tu organizacion.

### Matriz por plan (guia operativa para validacion)

> Nota: esta matriz es una guia de trabajo para onboarding. La configuracion real depende del proveedor, del contrato y del tenant activo en tu organizacion.

| Proveedor | Plan / modalidad | Uso potencial para entrenamiento | Retencion de prompts/respuestas | Metadatos operativos | Controles admin / compliance |
| :--- | :--- | :--- | :--- | :--- | :--- |
| GitHub Copilot | Individual (Free/Pro) | Puede variar segun configuracion y politica vigente | Puede existir retencion temporal de servicio | Si, habitualmente | Bajo-Medio |
| GitHub Copilot | Business | Normalmente mayor control organizacional sobre uso de datos | Politicas de retencion mas gobernables por organizacion | Si | Alto (SSO, politicas, auditoria) |
| GitHub Copilot | Enterprise | Enfocado a gobernanza y cumplimiento empresarial | Retencion y tratamiento definidos por contrato/politica enterprise | Si | Muy alto (controles corporativos avanzados) |
| Claude | Individual (Pro / API personal) | Depende de producto y ajustes de cuenta | Puede existir retencion de servicio segun politica vigente | Si | Bajo-Medio |
| Claude | Team | Mayor capacidad de gestion de espacio de trabajo | Retencion y uso definidos por configuracion de workspace | Si | Medio-Alto |
| Claude | Enterprise | Enfocado a seguridad, gobernanza y acuerdos corporativos | Definido por contrato enterprise y configuracion tenant | Si | Muy alto (SSO, auditoria, controles centralizados) |

### Reglas practicas (sin asumir politicas por defecto)

- No asumas que "pago" significa automaticamente "no retencion".
- No asumas que "enterprise" significa automaticamente "cero logs".
- En cuentas con SSO, la politica efectiva la define el administrador del tenant.
- Siempre valida fecha de ultima actualizacion de la politica antes de decidir.

### Checklist minimo antes de usar IA en repos corporativos

1. Confirmar si el contenido de prompts/respuestas se usa para entrenamiento del modelo.
2. Confirmar ventana de retencion de logs y metadatos.
3. Confirmar residencia de datos y cumplimiento (GDPR, ISO, SOC2, etc.).
4. Confirmar controles de admin: auditoria, DLP, export, borrado.
5. Guardar evidencia (captura/politica/version/fecha) para compliance.

### Referencias utiles

- GitHub Copilot Privacy: https://docs.github.com/en/copilot/privacy
- Anthropic Privacy: https://www.anthropic.com/privacy
- Politicas enterprise internas de tu compania (fuente principal para decisiones operativas).
