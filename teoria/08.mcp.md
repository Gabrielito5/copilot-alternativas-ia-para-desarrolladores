# Tema 8. MCP (MODEL CONTEXT PROTOCOL): EXTENDER COPILOT CON HERRAMIENTAS Y CONTEXTO REAL

El conocimiento de la IA está limitado a lo que "ve" en el IDE. **MCP** es el estándar abierto que permite a Copilot conectarse a fuentes de datos externas y herramientas en tiempo real, convirtiéndose en un asistente que realmente entiende el ecosistema de tu empresa.

---

## 1. MCPs: Conectar Copilot con sistemas externos

### El problema: la IA aislada

Sin MCP, el agente solo "ve" tu código fuente. Cuando le pides que genere una entidad JPA, no sabe cómo es la tabla real en la base de datos. Cuando le pides que implemente un ticket, no sabe qué dice el ticket. Cuando le pides que siga los patrones del equipo, no sabe cómo lo han resuelto otros servicios de la organización.

El resultado: el desarrollador actúa como un intermediario manual, copiando y pegando contexto entre sistemas.

```
Sin MCP (el dev como pegamento):

Jira Ticket #234                    Copilot Chat
┌────────────────┐      copy         ┌──────────────────┐
│ Bug: NPE en    │  ──────────────→  │ Aquí está el      │
│ OrderService   │      paste        │ stack trace,      │
│ Stack trace... │                   │ ayúdame a...      │
└────────────────┘                   └──────────────────┘
        ↑                                     ↑
        │           copy                      │
BD dev  │  ──────────────────────────────────→│
(schema)│           paste                     │
        └────────────────────────────────────→│
```

### La solución: un protocolo cliente-servidor

MCP define un protocolo estándar que permite al IDE (host) conectar el modelo de IA con servidores externos. Cada servidor expone **recursos** (datos que la IA puede leer) y **herramientas** (acciones que la IA puede ejecutar).

```
┌─────────────────────────────────────────────────────────────┐
│                       IDE / Terminal                         │
│                    (MCP Host / Client)                       │
│                                                             │
│   Agente IA ←──→ MCP Client ←──→ MCP Servers               │
│                               ┌───────────────────────┐    │
│                               │ postgres-dev           │    │
│                               │ github-org             │    │
│                               │ jira-team              │    │
│                               │ confluence-docs        │    │
│                               │ sentry-errors          │    │
│                               └───────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

### Qué diferencia MCP de simplemente "pegar contexto en el chat"

| Dimensión | Copiar y pegar | Con MCP |
|---|---|---|
| **Actualización** | El contexto es una foto fija del momento en que lo copiaste | El agente consulta la fuente en tiempo real |
| **Escala** | Limitado por lo que cabe en el chat | El agente decide qué consultar según la tarea |
| **Esfuerzo** | El dev hace de intermediario en cada prompt | El agente lo hace solo cuando lo necesita |
| **Precisión** | Errores de copia, formato incorrecto | Datos exactos directamente de la fuente |

## 2. MCP en Copilot: Casos típicos en equipos backend

### Caso 1: Contexto de Datos (Base de Datos)

Conectar un servidor MCP a la base de datos de **desarrollo** transforma la forma de generar código de persistencia.

**Sin MCP — el flujo habitual:**
```
Dev abre pgAdmin → Busca la tabla "orders" → Copia el DDL → 
Lo pega en el chat → La IA genera la entidad, pero:
  - Usa `Long` donde la BD tiene `UUID`
  - No detecta que `customer_id` es una FK
  - No sabe que `status` tiene un CHECK constraint con valores enum
```

**Con MCP:**
```
"Lee el esquema de la tabla 'orders' en la base de datos de desarrollo 
y genera la entidad JPA correspondiente con los tipos exactos. 
Infiere las relaciones @ManyToOne a partir de las claves foráneas."
```

El agente consulta el esquema real y genera:

```java
@Entity
@Table(name = "orders")
public class OrderEntity {
    
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    @Column(columnDefinition = "uuid")
    private UUID id;                          // ← La BD tiene UUID, no Long
    
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "customer_id", nullable = false)
    private CustomerEntity customer;          // ← Detectó la FK
    
    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 20)
    private OrderStatus status;               // ← Detectó el CHECK constraint
    
    @Column(name = "total_amount", precision = 19, scale = 4)
    private BigDecimal totalAmount;           // ← Precisión exacta del DECIMAL(19,4)
    
    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;
}
```

### Caso 2: Contexto de Infraestructura

Conectar con servidores MCP de AWS, Kubernetes o Docker:

```
"Analiza la configuración actual del pod 'order-service' en el namespace 
'backend-dev' de Kubernetes y genera el application.yml de Spring Boot 
con los valores de las variables de entorno que ya están inyectadas."
```

La IA lee la especificación del pod y genera configuración coherente con la infraestructura real, en lugar de inventar nombres de variables.

### Caso 3: Contexto de Documentación

Conectar con Confluence o Notion permite que la IA respete las reglas de negocio que viven fuera del código.

```
"Lee la página de Confluence 'Reglas de Facturación de IVA' 
y genera el servicio TaxCalculationService siguiendo exactamente 
las reglas descritas en esa documentación."
```

Sin MCP, la IA generaría la lógica fiscal basándose en el conocimiento genérico de IVA. Con MCP, respeta las particularidades de tu negocio: exenciones, tramos, tipos reducidos específicos que solo están documentados en tu Confluence.

### Caso 4: Contexto de Código (Otros repositorios)

Conectar con el GitHub MCP Server permite que la IA entienda cómo trabaja el resto de la organización:

```
"Busca en el repositorio 'payments-service' cómo implementaron 
el Outbox Pattern para publicar eventos Kafka de forma garantizada. 
Aplica la misma estrategia en este servicio de pedidos."
```

En lugar de generar una implementación genérica, el agente reutiliza la solución que ya probó y validó otro equipo de tu empresa.

## 3. GitHub MCP Server y MCP Registry: Qué aportan al flujo

### GitHub MCP Server: las capacidades más útiles para un Senior

| Capacidad | Prompt de ejemplo |
|---|---|
| **Buscar en repos** | "Busca en todos los repos de la organización cómo se gestiona la autenticación JWT" |
| **Leer Issues** | "Lee el bug #234, encuentra el código afectado en este repo y propón un fix" |
| **Leer PRs** | "Analiza las últimas 5 PRs del servicio de inventario y extrae los patrones de refactor que aplicaron" |
| **Leer archivos** | "Lee el `docker-compose.yml` del servicio de pagos y genera uno equivalente para este servicio" |
| **Historial de commits** | "¿Cuándo se introdujo la clase CircuitBreakerConfig? ¿Qué problema resolvía?" |

### MCP Registry: el catálogo de servidores disponibles

| Servidor MCP | Lo que aporta al Senior Backend |
|---|---|
| **PostgreSQL / MySQL** | Esquemas reales, tipos exactos, relaciones entre tablas |
| **GitHub** | Código de otros repos, issues, PRs, historial |
| **Jira / Linear** | Requisitos, criterios de aceptación, contexto del ticket |
| **Confluence / Notion** | Reglas de negocio, ADRs, documentación de arquitectura |
| **Sentry** | Stack traces reales, frecuencia de errores, breadcrumbs |
| **Datadog / Grafana** | Métricas de rendimiento, trazas de producción |
| **Slack** | Decisiones del equipo, conversaciones técnicas sobre el sistema |
| **Docker** | Configuración real de contenedores, imágenes base en uso |
| **Kubernetes** | Estado real de pods, configuración de deployments, variables de entorno |

### Centralización del conocimiento: eliminar las alucinaciones inter-servicio

El problema más frustrante al usar IA en arquitecturas de microservicios es cuando la IA "inventa" cómo funciona otro servicio de tu empresa:

```
# ❌ Sin MCP: la IA alucina el contrato del servicio de pagos
"El endpoint de pagos probablemente acepta un objeto PaymentRequest 
con los campos amount, currency y cardToken..."

# La IA lo está inventando. El contrato real puede ser completamente diferente.

# ✅ Con GitHub MCP: la IA lee el contrato real
"He leído el archivo openapi/payments-api.yaml del repositorio payments-service.
El endpoint POST /v2/charges acepta un ChargeCommand con los campos:
- amountInCents (Long, requerido)
- currency (String ISO 4217, requerido)  
- paymentMethodId (UUID, requerido, no el cardToken)
- idempotencyKey (String, recomendado)
Te genero el FeignClient correspondiente con los tipos exactos."
```

## 4. Patrones de uso: "trae contexto → genera cambio → abre PR → valida"

### El ciclo cerrado completo

Este es el patrón que distingue el uso superficial de MCP del uso que realmente acelera el trabajo de un Senior:

**Paso 1 — Trae contexto:**
```
"Lee el ticket JIRA-456. 
Busca en el repositorio 'billing-service' cómo implementaron el cálculo 
de IVA reducido para productos de alimentación (en el commit de febrero).
Resúmeme el enfoque antes de generar nada."
```

**Paso 2 — Genera el cambio:**
```
"Aplica el mismo enfoque de cálculo de IVA al módulo de suscripciones 
de este repositorio. Los tipos de suscripción están en 
SubscriptionType.java. Las reglas específicas están en el ticket."
```

**Paso 3 — Redacta la PR:**
```
"Redacta la descripción de la Pull Request para este cambio.
Incluye:
- El ticket JIRA-456 que implementa (con su título exacto)
- Los archivos modificados y por qué
- El enfoque elegido y por qué es coherente con billing-service
- Los criterios de aceptación del ticket y cómo los cumple cada cambio"
```

**Paso 4 — Valida:**
```
"Consulta el estado de los checks de CI de la rama actual en GitHub.
Si hay algún test fallando, léeme el output del test y propón el fix."
```

### El patrón "Detective de Incidencias"

Cuando llega un bug de producción, MCP convierte al agente en un investigador que tiene acceso a todas las fuentes de evidencia:

```
1. "Lee el error de Sentry #ERR-7821. 
    Muéstrame el stack trace completo y los breadcrumbs."

2. "Busca en este repositorio la clase mencionada en el stack trace 
    (OrderProcessingService, línea 142). Muéstrame el código."

3. "Busca en el historial de commits de ese archivo si hubo algún 
    cambio en los últimos 30 días que pueda haber introducido este bug."

4. "Propón un fix y un test de regresión que habría detectado el problema."
```

## 5. Riesgos: Contexto excesivo, datos sensibles y falta de control

### Riesgo 1: Ruido en el contexto

Conectar demasiadas fuentes o demasiados datos degrada la calidad de las respuestas. El modelo tiene una ventana de contexto finita: si la llenas con datos irrelevantes, desplazas el código relevante.

```
# ❌ MAL — Demasiado contexto, respuesta degradada
"Conecta a toda la base de datos de producción y genera las entidades JPA"
→ La IA recibe 300 tablas de esquemas distintos
→ Se pierde, mezcla entidades de distintos dominios
→ Las respuestas se vuelven genéricas e imprecisas

# ✅ BIEN — Contexto acotado al dominio de la tarea
"Conecta a la BD de desarrollo, esquema 'orders', 
y lee solo las tablas: orders, order_items, order_status_history"
→ La IA recibe 3 tablas del mismo dominio
→ Genera entidades coherentes con las relaciones correctas
```

### Riesgo 2: Fuga de datos (Exfiltración)

Este es el riesgo más serio. Los datos que el agente consulta a través de MCP viajan al servidor del LLM para ser procesados. Si el servidor MCP tiene acceso a datos reales de clientes, esos datos salen de tu infraestructura.

```
⚠️ Escenario de riesgo:

MCP conectado a BD de producción
    ↓
Prompt: "Busca ejemplos de órdenes con errores de pago para 
        entender el problema"
    ↓
El agente ejecuta: SELECT * FROM orders WHERE status = 'PAYMENT_FAILED' LIMIT 10
    ↓
Los datos reales de clientes (nombres, emails, importes) 
viajan al servidor del LLM externo
    ↓
Violación de GDPR / políticas de seguridad de la empresa
```

**Regla absoluta:** Nunca conectar MCP a bases de datos de producción ni a sistemas que contengan datos reales de clientes.

### Riesgo 3: Herramientas "Write" — permisos de escritura

Los servidores MCP pueden exponer herramientas de escritura además de lectura. Dar permiso de escritura a la IA sobre sistemas críticos es un riesgo que requiere evaluación explícita:

| Acción via MCP | Nivel de Riesgo | Política Recomendada |
|---|---|---|
| Leer esquema de BD dev | Bajo | ✅ Permitir libremente |
| Ejecutar SELECT en BD dev | Bajo-Medio | ✅ Permitir con esquema acotado |
| Leer Issues de GitHub | Bajo | ✅ Permitir libremente |
| Leer archivos de otros repos | Bajo | ✅ Permitir libremente |
| Ejecutar INSERT/UPDATE en BD | Alto | ❌ No permitir |
| Crear/cerrar Issues | Medio | ⚠️ Solo con confirmación explícita del dev |
| Crear PRs | Medio | ⚠️ Solo con revisión previa del contenido |
| Desplegar en Staging | Muy Alto | ❌ Nunca automáticamente |
| Desplegar en Producción | Crítico | ❌ Nunca |

> ⚠️ El peligro de las herramientas "write" no es que la IA actúe con mala intención: es que actúe con buena intención y criterio equivocado. Un agente que "ayuda" cerrando el ticket de Jira antes de que el código esté revisado, o desplegando en Staging sin que el Senior lo haya pedido explícitamente, introduce desorden difícil de rastrear.

## 6. Buenas prácticas: Permisos mínimos, entornos y auditoría

### Principio de menor privilegio en la configuración

El archivo de configuración MCP (`.vscode/mcp.json` en VS Code, o `~/.claude/settings.json` en Claude Code) debe reflejar el principio de menor privilegio:

```json
{
  "mcpServers": {
    "postgres-dev": {
      "command": "mcp-server-postgres",
      "args": [
        "--read-only",
        "--schema=orders",
        "--schema=products"
      ],
      "env": {
        "DATABASE_URL": "postgresql://readonly_user:***@localhost:5432/dev_db"
      }
    },
    "github-org": {
      "command": "mcp-server-github",
      "args": ["--read-only"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_READ_ONLY_TOKEN}"
      }
    }
  }
}
```

**Aspectos clave de esta configuración:**
- `--read-only`: sin permisos de escritura
- Esquemas específicos, no toda la base de datos
- Usuario de BD con permisos mínimos (no el usuario de la aplicación)
- Token de GitHub de solo lectura (no el PAT con todos los permisos)

### Separación por entornos: la regla de los tres niveles

```
DESARROLLO                  STAGING                     PRODUCCIÓN
┌─────────────────┐        ┌─────────────────┐        ┌─────────────────┐
│ MCP: BD local   │        │ MCP: Solo BD,   │        │ MCP: NUNCA      │
│ MCP: GitHub     │        │ solo lectura,   │        │ conectar         │
│ MCP: Jira       │        │ sin datos reales│        │ directamente     │
│ MCP: Confluence │        │                 │        │                 │
│ Acceso libre    │        │ Con supervisión │        │                 │
└─────────────────┘        └─────────────────┘        └─────────────────┘
```

Para los tests de integración, usa Testcontainers en lugar de conectar MCP a una base de datos compartida:

```java
// ✅ Tests con Testcontainers: la IA puede generar y ejecutar esto sin riesgo
@Testcontainers
@SpringBootTest
class OrderRepositoryIntegrationTest {
    
    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16")
        .withDatabaseName("test_db")
        .withInitScript("db/schema.sql");
    
    // El entorno es efímero: se crea para el test y se destruye al terminar
}
```

### Auditoría de logs MCP

Los servidores MCP generan logs de todas las operaciones. Revisarlos periódicamente permite detectar patrones de uso inesperados:

**Qué buscar en los logs:**

```bash
# Consultas a tablas que no corresponden a la tarea en curso
[MCP postgres-dev] SELECT * FROM users WHERE id = ...
# ← Si estabas trabajando en pedidos, ¿por qué la IA está leyendo usuarios?

# Volumen inusual de consultas
[MCP postgres-dev] 47 queries in 30 seconds
# ← Un agente en bucle puede generar cientos de consultas

# Consultas con datos que parecen de producción
[MCP postgres-dev] SELECT * FROM orders WHERE customer_email = 'juan.garcia@empresa.com'
# ← ¿Cómo tiene ese email la IA? ¿De dónde salió ese dato?
```

### Combinar MCP con archivos de instrucciones

Los archivos de instrucciones del equipo (`.github/copilot-instructions.md`) y el contexto MCP se complementan. Las instrucciones definen el "cómo"; MCP aporta el "qué hay" en los sistemas reales:

```markdown
# .github/copilot-instructions.md

## Stack y Versiones
- Java 21, Spring Boot 3.4, PostgreSQL 16
- Arquitectura hexagonal estricta

## Contexto MCP Disponible
- `postgres-dev`: BD de desarrollo, esquemas: orders, products, customers
  → Úsalo para consultar esquemas antes de generar entidades JPA
- `github-org`: Repositorios de la organización (read-only)
  → Antes de implementar algo nuevo, busca si otro servicio ya lo ha resuelto
- `jira-team`: Proyecto BACKEND de Jira (read-only)
  → Lee siempre el ticket antes de empezar a implementar

## Instrucción de Prioridad
Cuando tengas acceso MCP a la BD, úsalo para validar los tipos antes
de generar código de persistencia. Nunca asumas tipos; consúltalos.
```

## 7. MCP como estándar emergente del sector

MCP no es una herramienta propietaria de un proveedor: es un **protocolo abierto** creado por Anthropic y adoptado rápidamente por el ecosistema. Esto tiene implicaciones prácticas importantes para un Senior:

- Los servidores MCP que configures hoy funcionarán con distintos clientes IA (Claude Code, GitHub Copilot, Cursor, etc.)
- El ecosistema de servidores pre-construidos crece continuamente en el MCP Registry
- Puedes construir servidores MCP internos para exponer datos propietarios de tu empresa de forma controlada

### Cuándo construir un servidor MCP interno

Si tu empresa tiene sistemas internos no cubiertos por el registro público (ERPs propietarios, plataformas internas, wikis internas), puedes construir un servidor MCP que los exponga al agente:

```
Casos en los que merece construir un servidor MCP interno:
✅ Tu empresa tiene un catálogo de APIs internas no documentadas en OpenAPI
✅ Tienes reglas de negocio en un sistema legacy que la IA debe respetar
✅ Tienes un sistema de gestión de configuración interno (no Kubernetes estándar)
✅ Tus datos de referencia (tarifas, categorías, países) están en un sistema propietario
```

## 🧪 Ejercicio: "El Detective de Incidencias"

> **Contexto:** Se ha producido un error en producción. Tienes acceso a Sentry con los logs del error y al repositorio Git del servicio afectado.

**Setup (10 min):**
1. Configura un servidor MCP de PostgreSQL conectado a una base de datos local de desarrollo.
2. Crea las tablas de ejemplo con el siguiente script:
```sql
CREATE TABLE customers (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email VARCHAR(255) NOT NULL UNIQUE,
    name VARCHAR(255) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT now()
);

CREATE TABLE orders (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    customer_id UUID NOT NULL REFERENCES customers(id),
    status VARCHAR(20) NOT NULL CHECK (status IN ('PENDING', 'CONFIRMED', 'SHIPPED', 'CANCELLED')),
    total_amount DECIMAL(19, 4) NOT NULL,
    discount_code VARCHAR(50),
    created_at TIMESTAMP NOT NULL DEFAULT now()
);

CREATE TABLE order_items (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    order_id UUID NOT NULL REFERENCES orders(id),
    product_id VARCHAR(50) NOT NULL,
    quantity INTEGER NOT NULL CHECK (quantity > 0),
    unit_price DECIMAL(19, 4) NOT NULL
);
```

**Reto — Parte 1: Generación a partir del esquema real (10 min):**
1. Activa el MCP de PostgreSQL en tu IDE.
2. Envía este prompt:
   ```
   Lee el esquema de las tablas orders, order_items y customers 
   en la base de datos de desarrollo. Genera las entidades JPA 
   correspondientes siguiendo arquitectura hexagonal:
   - Las entidades van en infrastructure/persistence/
   - Los tipos Java deben coincidir EXACTAMENTE con los tipos SQL
   - Infiere las relaciones @ManyToOne y @OneToMany de las FK
   - Respeta los constraints NOT NULL y CHECK
   ```
3. Verifica que los tipos Java coinciden con los tipos SQL (UUID, DECIMAL con precisión, el enum de status).

**Reto — Parte 2: Test de integración con el esquema real (10 min):**
1. Continúa la conversación:
   ```
   Usando el mismo esquema que acabas de leer, genera un test de 
   integración con Testcontainers para OrderRepository.
   El test debe inicializar la BD con el mismo DDL que leíste del 
   esquema real. Cubre: creación de orden, búsqueda por ID, 
   búsqueda de órdenes de un cliente.
   ```

**Reflexión (5 min):**
- ¿Qué tipos habría asumido la IA sin el contexto MCP? (probablemente `Long` para los IDs en lugar de `UUID`)
- ¿Habría detectado el constraint CHECK del status sin leer el esquema?
- ¿Cuánto tiempo te habría llevado hacer esto sin MCP?

## 💡 Tips del IDE

### El "Contextual Glue": logs pesados via MCP

Un caso de uso poco obvio pero muy valioso: los archivos de log de aplicaciones grandes no caben en el chat de IntelliJ (decenas de MB). Con un servidor MCP local que exponga el sistema de archivos de forma controlada, puedes pedirle a la IA que los analice:

```
"Lee el archivo /var/log/app/order-service-error.log de hoy 
y dime cuáles son los 5 errores más frecuentes, 
agrupados por tipo de excepción y hora de ocurrencia."
```

### Verifica qué está consultando el agente

Muchos clientes MCP muestran las herramientas que el agente está invocando en tiempo real. Activa esa visibilidad y obsérvala mientras el agente trabaja: es la forma más rápida de detectar si el agente está consultando fuentes que no debería.

```
[Llamada a herramienta] postgres-dev.query
  SQL: SELECT table_name, column_name, data_type 
       FROM information_schema.columns 
       WHERE table_schema = 'orders'
[Resultado] 23 columnas en 3 tablas ✓

[Llamada a herramienta] postgres-dev.query  
  SQL: SELECT * FROM customers LIMIT 5
                               ↑
  ⚠️ ¿Por qué está leyendo datos de clientes para una tarea de esquema?
     Para y pregúntale al agente qué está haciendo.
```

## 8. Ejemplo Completo: Tu Primer Servidor MCP desde Cero

Este ejemplo documenta el ciclo completo: problema → construcción → registro → validación. Funcionando en menos de 20 minutos con Node.js y una API pública gratuita de clima (sin clave de API).

---

### Fase 0: El problema — Claude sin MCP no conoce el tiempo actual

Abre Claude Code y escribe:

```
¿Qué tiempo hace ahora mismo en Ciudad de México?
```

La respuesta será algo así:

```
No tengo acceso a datos en tiempo real, por lo que no puedo darte
el tiempo actual en Ciudad de México. Mi conocimiento tiene una fecha
de corte y no incluye condiciones meteorológicas actuales.
```

Correcto: el LLM no puede hacer llamadas a internet por sí solo. Ahora le vamos a dar esa capacidad mediante MCP.

---

### Fase 1: Crear el servidor MCP en Node.js

#### Estructura del proyecto

```bash
mkdir weather-mcp-server
cd weather-mcp-server
npm init -y
npm install @modelcontextprotocol/sdk zod
```

Añade `"type": "module"` en `package.json` para usar ES Modules:

```json
{
  "name": "weather-mcp-server",
  "version": "1.0.0",
  "type": "module",
  "scripts": {
    "start": "node index.js"
  },
  "dependencies": {
    "@modelcontextprotocol/sdk": "^1.0.0",
    "zod": "^3.22.0"
  }
}
```

#### El servidor (`index.js`)

Crea el fichero `index.js`:

```javascript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({
  name: "weather-server",
  version: "1.0.0",
  description: "Consulta el tiempo actual de cualquier ciudad usando Open-Meteo",
});

// Herramienta: obtener el tiempo actual de una ciudad
server.tool(
  "get_current_weather",
  {
    city: z.string().describe("Nombre de la ciudad (en español o inglés)"),
  },
  async ({ city }) => {
    try {
      // Paso 1: Geocodificar la ciudad → obtener latitud/longitud
      const geocodeUrl =
        `https://geocoding-api.open-meteo.com/v1/search` +
        `?name=${encodeURIComponent(city)}&count=1&language=es&format=json`;
      const geocodeRes = await fetch(geocodeUrl);
      const geocodeData = await geocodeRes.json();

      if (!geocodeData.results?.length) {
        return {
          content: [{
            type: "text",
            text: `No se encontró la ciudad: "${city}". Prueba con el nombre en inglés.`,
          }],
        };
      }

      const { latitude, longitude, name, country, timezone } = geocodeData.results[0];

      // Paso 2: Consultar el tiempo actual con las coordenadas
      const weatherUrl =
        `https://api.open-meteo.com/v1/forecast` +
        `?latitude=${latitude}&longitude=${longitude}` +
        `&current=temperature_2m,apparent_temperature,relative_humidity_2m,` +
        `wind_speed_10m,precipitation,weather_code` +
        `&timezone=${encodeURIComponent(timezone)}`;

      const weatherRes = await fetch(weatherUrl);
      const weatherData = await weatherRes.json();
      const c = weatherData.current;

      const result = [
        `Tiempo actual en ${name}, ${country}`,
        `─────────────────────────────────────`,
        `Temperatura:    ${c.temperature_2m}°C (sensación ${c.apparent_temperature}°C)`,
        `Humedad:        ${c.relative_humidity_2m}%`,
        `Viento:         ${c.wind_speed_10m} km/h`,
        `Precipitación:  ${c.precipitation} mm`,
        `Estado:         ${describeWeatherCode(c.weather_code)}`,
        `Hora local:     ${c.time}`,
      ].join("\n");

      return { content: [{ type: "text", text: result }] };

    } catch (error) {
      return {
        content: [{ type: "text", text: `Error consultando el tiempo: ${error.message}` }],
      };
    }
  }
);

// Mapa simplificado de códigos WMO (https://open-meteo.com/en/docs)
function describeWeatherCode(code) {
  const descriptions = {
    0: "Despejado",
    1: "Principalmente despejado", 2: "Parcialmente nublado", 3: "Nublado",
    45: "Niebla", 48: "Niebla con escarcha",
    51: "Llovizna ligera", 53: "Llovizna moderada", 55: "Llovizna densa",
    61: "Lluvia ligera", 63: "Lluvia moderada", 65: "Lluvia fuerte",
    71: "Nevada ligera", 73: "Nevada moderada", 75: "Nevada intensa",
    80: "Chubascos ligeros", 81: "Chubascos moderados", 82: "Chubascos violentos",
    95: "Tormenta eléctrica", 99: "Tormenta con granizo",
  };
  return descriptions[code] ?? `Código WMO ${code}`;
}

// Arrancar el servidor con transporte stdio (estándar para MCP local)
const transport = new StdioServerTransport();
await server.connect(transport);
```

> 💡 **¿Por qué Open-Meteo?** Es una API pública de clima completamente gratuita y sin clave de API. Devuelve datos en tiempo real con cobertura global. Perfecta para demostraciones sin fricción de registro.

### Fase 2: Registrar el servidor en Claude Code

#### ¿Cómo funciona el transporte stdio?

Antes de registrar, conviene entender qué pasará cuando Claude use el servidor:

```
┌──────────────────────────────────────────────────────────┐
│                       Claude Code                        │
│                                                          │
│  1. Detecta que necesita la herramienta "weather"       │
│  2. Lanza automáticamente: node /ruta/a/index.js        │
│     (proceso hijo, comunicación por stdin/stdout)        │
│  3. Envía la petición en formato JSON-RPC               │
│  4. Recibe el resultado e incorpora al contexto         │
│  5. Cierra el proceso cuando ya no lo necesita          │
└──────────────────────────────────────────────────────────┘
```

**No tienes que arrancar el servidor manualmente.** Claude lo lanza, lo usa y lo cierra solo. Tu único trabajo es registrar dónde está el fichero.

#### Opción A: Registro de proyecto (recomendado)

Crea o edita `.claude/settings.json` en la raíz de tu proyecto:

```json
{
  "mcpServers": {
    "weather": {
      "command": "node",
      "args": ["/ruta/absoluta/a/weather-mcp-server/index.js"]
    }
  }
}
```

> ⚠️ Usa la ruta absoluta. Claude Code lanza el proceso desde directorios variables, por lo que las rutas relativas pueden no resolverse.

#### Opción B: Registro global (disponible en todos tus proyectos)

Edita `~/.claude/settings.json`:

```json
{
  "mcpServers": {
    "weather": {
      "command": "node",
      "args": ["/ruta/absoluta/a/weather-mcp-server/index.js"]
    }
  }
}
```

#### Opción C: Registro por línea de comandos

```bash
claude mcp add weather node /ruta/absoluta/a/weather-mcp-server/index.js
```

Este comando modifica automáticamente `~/.claude/settings.json`.

#### Verificar que el servidor está registrado

```bash
claude mcp list
```

Salida esperada:

```
weather: node /ruta/absoluta/a/weather-mcp-server/index.js
```

### Diagnóstico: si el servidor no responde

Prueba el servidor de forma aislada antes de involucrar a Claude. El transporte stdio es JSON-RPC puro, por lo que puedes enviar una petición manualmente:

```bash
# Listar las herramientas que expone el servidor
echo '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}' | node index.js
```

Deberías ver la herramienta `get_current_weather` en el JSON de respuesta. Si hay un error de módulo:

```bash
# Verificar que las dependencias están instaladas
ls node_modules/@modelcontextprotocol

# Reinstalar si hace falta
npm install
```
